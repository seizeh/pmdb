# 0027 — 알림 파이프라인 정비기 (포그라운드 OS 알림·딥링크·iOS SceneDelegate 함정)

> 0023(지도 PlatformView)과 같은 성격의 **회고 문서** — 증상, 빗나간 가설, 결정적
> 증거, 최종 구조를 시간순으로 남긴다. 다음에 알림이 "안 온다/안 열린다"는 보고가

## 0. 결정값 (최종 확정)

| 항목 | Android | iOS |
| --- | --- | --- |
| 백그라운드/종료 표시 | 서버 FCM notification 페이로드 → OS | 동일 (APNs `aps.sound: default`) |
| 백그라운드 알림 채널 | **`default_high`** (매니페스트 `default_notification_channel_id`) | — |
| 포그라운드 표시 | realtime 알림 → `flutter_local_notifications` (`LocalNoticeService`) | **네이티브 `willPresent`** 가 OS 배너 직접 표시 |
| 포그라운드 건별 억제 | 보고 있는 채팅방·동기화 타입·비 resumed 상태 제외 | **불가**(OS 배너 한계) — 활성 방 메시지도 배너 |
| 알림 탭 → 라우팅 | FCM `onMessageOpenedApp` / `getInitialMessage` | **네이티브 `didReceive` → `pawmate/push_tap` 채널** (FCM 스트림 미사용) |
| 딥링크 단일 진입점 | `openFromPush(type, resourceType, resourceId)` — 푸시 탭·벨 패널·알림함 공용 | 동일 |
| flutter_local_notifications | 사용 (desugaring 필수) | **초기화 금지** (델리게이트 충돌) |

## 1. 시작 — "토스트 말고 OS 알림으로"

기존 포그라운드 알림은 realtime 구독 → 하단 토스트(AppToast)였다. 요구는 "앱 실행
중에도 백그라운드 푸시와 동일한 형태의 알림". 방향:

- 표시 경로는 realtime 유지(푸시는 토큰 없는 기기·시뮬레이터에서 안 오지만
  realtime 은 앱이 켜져 있으면 항상 도착) — 건별 억제 규칙도 Dart 에서 그대로.
- 표시 수단만 토스트 → `flutter_local_notifications` 시스템 알림으로 교체.
- 이중 표시 가드: 백그라운드 전환 직후 realtime 이 잠시 살아 있으면 FCM 과
  겹치므로 `lifecycleState == resumed` 일 때만 로컬 표시.
- 같은 채팅방 연속 메시지는 알림 id 를 방 id 해시로 고정해 갱신(도배 방지).

빌드 이슈 1: `flutter_local_notifications` 는 **core library desugaring** 요구 —
`android/app/build.gradle.kts` 에 `isCoreLibraryDesugaringEnabled = true` +
`coreLibraryDesugaring("com.android.tools:desugar_jdk_libs:2.1.4")`.

## 2. 1차 보고 — "백그라운드 탭이 커뮤니티로, 인앱 알림도 안 옴"

### 2.1 서버 파이프라인 전수 검증 (전부 정상이었다)

의심 순서대로 확인한 것들 — 모두 무혐의:

- `send-push` FCM 페이로드: `data` 에 `type/resource_type/resource_id` 완비.
- `notifications` 테이블이 `supabase_realtime` publication 에 포함.
- postgres_changes 복제 슬롯 active (`pg_replication_slots`).
- `realtime.subscription` 에 앱의 구독 등록(필터·claims 정상).
- RLS: JWT 시뮬레이션(`set local role authenticated` + `request.jwt.claims`)으로
  notifications/v_chat_rooms 조회 재현 — 정상. ⚠️ 이때 `app.uid()` 는 `tv`
  (token_version) 클레임까지 검사하므로 시뮬레이션에 tv 를 넣어야 한다.

### 2.2 에뮬레이터 원격 재현 — 결정적 증거

adb 로 직접 재현(§8 기법): 앱을 HOME 으로 내리고 테스트 알림 insert → FCM 도착
확인(dumpsys) → 알림 탭 → **커뮤니티 탭 위에 빈 알림 패널**. 그런데:

- 몇 초 뒤 벨을 다시 누르면 같은 목록이 **정상 표시**됨.
- API 로그: 탭 직후엔 조회 기록이 없거나 실패, 재시도 땐 200.

**확정** — 라우팅은 실행됐지만, **백그라운드 복귀 직후 끊긴 소켓 탓에 첫
HTTP 요청들이 일시 실패** → `fetchRoom` 실패 → 알림함 폴백(그 폴백의 목록
조회도 같은 이유로 실패해 "받은 알림이 없어요"). 기본 탭이 커뮤니티라
"커뮤니티로 연결된다"로 보였던 것.

**수정**: `openFromPush` 의 대상 조회(채팅방/게시글/후기)를 3회 × 0.4s
재시도(`fetchRetry`). 이후 에뮬레이터에서 백그라운드 탭 → 채팅방 직행,
뒤로가기 → 채팅 목록까지 스크린샷으로 검증.

### 2.3 알림함 라우팅 통일

벨 패널(notification_panel)과 알림함 화면(notifications_screen)이 각자
`AppPageRoute` 로 밀어 넣고 있었다(가로 전환, 탭 전환 없음 → 닫으면 커뮤니티로
복귀). 두 곳 모두 자체 라우팅을 제거하고 **`openFromPush` 를 그대로 호출** —
채팅 탭 전환 + rise 전환 + 닫으면 채팅 목록.

## 3. iOS 여정 — SceneDelegate 함정 2호

Android 가 완전 검증된 뒤에도 iPhone 만 "인앱 알림 안 옴 + 배너 탭 안 됨"이
지속. 단계별로:

### 3.1 헛발질들 (기록해 둘 가치가 있는)

1. **"로컬 알림 플러그인이 iOS 델리게이트를 가로챈다"** 가설 → 플러그인 소스
   확인 결과 **반증**: v18 은 `registerWithRegistrar` 에서 델리게이트를 잡지
   않는다(`initialize()` 호출 시에만). iOS 에서 initialize 를 막은 조치 자체는
   유지(안전).
2. **구 빌드 문제** — 실제로 여러 차례의 "여전히 안 됨"은 iPhone 에 수정본이
   설치되지 않아서였다. 네이티브 수정은 핫 리로드/재시작으로 절대 반영되지
   않는다. 또 **iOS 디버그 빌드는 디버거 없이 단독 실행 불가** — "Dart VM
   Service not discovered" 로 attach 가 실패하면 테스트 자체가 무효.
   기기 테스트는 `flutter run --release` 가 정답.
3. 에뮬레이터 저장소 6G 파티션 포화로 설치 실패(`INSTALL_FAILED_INSUFFICIENT_
   STORAGE`) → AVD `disk.dataPartition.size` 16G + wipe 로 해결(부수 트랙).

### 3.2 진짜 원인 — 알림센터 콜백 전체 유실

서버 API 로그로 교차 확인한 결정적 사실: iPhone 에서 **벨 패널 탭은 방 조회
시퀀스가 찍히는데, OS 배너 탭은 어떤 요청도 발생하지 않는다** = 탭 이벤트가
Flutter 에 도달하지 않음.

원인: **SceneDelegate 환경에서 firebase_messaging 의 swizzling 이
UNUserNotificationCenter 델리게이트를 설정하지 못한다.** 델리게이트가 없으면
iOS 는 ① 포그라운드 도착 알림을 그냥 버리고(배너 없음) ② 탭 이벤트도 아무에게도
전달하지 않는다. APNs 토큰을 명시 전달해야 했던 함정(0026 이전, ios-push-setup)과
같은 계열.

1차 시도 — `UNUserNotificationCenter.current().delegate = self` 지정만: **부족**.
델리게이트는 잡히지만(willPresent 호출됨) FlutterAppDelegate 가 받은 콜백을
FCM 플러그인이 처리하지 못해 "표시 안 함"으로 종료.

### 3.3 최종 — 네이티브 직접 처리 (AppDelegate.swift)

플러그인 내부 포워딩에 의존하지 않는다:

```swift
// didFinishLaunching:
UNUserNotificationCenter.current().delegate = self   // 지우면 조용히 죽음

// 포그라운드 표시 — OS 배너 직접
override func userNotificationCenter(_:, willPresent:, withCompletionHandler:) {
  completionHandler([.banner, .list, .sound, .badge])  // iOS 14 미만은 .alert 분기
}

// 알림 탭 — FCM data 를 Dart 라우팅으로
override func userNotificationCenter(_:, didReceive response:, withCompletionHandler:) {
  // userInfo 의 type/resource_type/resource_id → pawmate/push_tap 채널 invokeMethod("tap")
  // 엔진 준비 전(콜드 스타트)이면 pendingTap 버퍼 → Dart 가 getPendingTap 으로 회수
}
```

Dart(`push_service.dart`): iOS 는 `onMessageOpenedApp`/`getInitialMessage` 를
**쓰지 않고** `pawmate/push_tap` 채널만 사용(Android 는 기존 FCM 스트림 유지).

검증: `xcrun devicectl` 로 서명 빌드 직접 설치 + `--console` 로 네이티브 로그
확인(`willPresent — 포그라운드 배너 표시`, `알림 탭 — {...}`) → 서버 API 로그에
방 단건 조회→메시지 로드→읽음 처리 시퀀스 — 전 구간 일치.

빌드 이슈 2: `.banner/.list` 는 iOS 14+ — `#available(iOS 14.0, *)` 분기 필요
(deployment 설정에 따라 컴파일 에러).

트레이드오프: iOS 포그라운드는 OS 배너라 **건별 억제 불가** — 보고 있는
채팅방의 메시지도 배너가 뜬다(Android 는 억제 유지). 개선하려면 willPresent 에서
현재 방 id 를 조회하는 네이티브↔Dart 브리지가 필요(추후 과제).

## 4. 진동 — "화면 꺼지면 진동이 안 와"

- **Android(실질 버그)**: 백그라운드 FCM 알림이 **FCM 폴백 채널**
  (`fcm_fallback_notification_channel`, importance=3, 진동 미지정)로 게시되고
  있었다. logcat 경고("Missing Default Notification Channel metadata")가 힌트.
  매니페스트에 `com.google.firebase.messaging.default_notification_channel_id =
  default_high` 지정 → dumpsys 로 default_high(importance=4) 게시 확인.
  ⚠️ Android 채널은 생성 후 불변 — 채널 속성을 바꾸려면 새 id 가 필요하다.
- **iOS(진범: Mac iPhone 알림 미러링)**: 서버는 `aps.sound: default` 를 보내고
  있었고, willPresent 에 aps 원문 로그를 심어 **기기 도착 페이로드에 sound 가
  실려 있음을 실증**(서버·FCM 무혐의 확정). 잠금 실험(화면 안 켜짐·무음·알림만
  쌓임)으로 "조용히 전달" 정책임을 좁혔고 — 최종 원인은 **macOS 의 iPhone 알림
  미러링**: Mac 이 깨어 있으면 잠긴 iPhone 의 알림을 Mac 이 가져가 울리고
  (실제로 Mac 에서 소리가 나고 있었다) iPhone 은 조용히 둔다. 이 전달은
  **앱별 토글**(Mac 시스템 설정 → 알림 → "iPhone 에서 알림 허용")이라 카카오톡
  등은 목록에서 빠져 있고 새로 설치된 pawmate 만 기본 ON 이라 앱별 증상처럼
  보였다. 개발 중엔 Mac 이 항상 옆에 깨어 있어 재현율 100%. Mac 쪽 토글을
  끄자 즉시 정상. 화면 켜진 상태(포그라운드)에서 울렸던 건 네이티브
  willPresent 가 iOS 전달 정책과 무관하게 표시했기 때문 — 증상 비대칭의 원인.
- 참고: **에뮬레이터는 물리 진동이 없다** — 진동 검증은 실기기에서만 의미 있음.

## 5. 최종 아키텍처 (흐름)

```
서버: notifications INSERT
 ├─ 트리거 → send-push(FCM) : notification(title/body) + data(type/resource) + aps.sound
 │    ├─ Android 백그라운드/종료 → OS 표시 (default_high 채널: 소리·진동·헤드업)
 │    └─ iOS 백그라운드/종료 → OS 표시 (sound: default)
 └─ realtime(postgres_changes) → 앱(포그라운드)
      ├─ Android: LocalNoticeService → 시스템 알림 (억제 규칙: 활성 방·동기화 타입·비 resumed)
      └─ iOS: (realtime 표시는 no-op) FCM 도착 → 네이티브 willPresent → OS 배너

알림 탭 (어디서든) → openFromPush(type, resourceType, resourceId)
 ├─ Android: onMessageOpenedApp / getInitialMessage
 ├─ iOS: 네이티브 didReceive → pawmate/push_tap 채널 (콜드 스타트: pendingTap 버퍼)
 └─ 벨 패널·알림함 화면도 동일 함수 호출
      → 관련 탭 전환(popToRoot + tabRequest) → 상세를 rise 로 push
      → 대상 조회는 fetchRetry(3회) — 복귀 직후 일시 실패 흡수
      → 실패 시 알림함 폴백
```

## 7. 교훈

1. **"안 된다" 보고를 받으면 코드보다 먼저 '어느 기기, 어느 빌드'를 확정하라.**
   네이티브 수정은 재빌드·재설치 없이는 존재하지 않는 것과 같다. 이번 여정의
   체감 난이도 절반은 스테일 빌드였다.
2. **서버 API 로그는 클라이언트 디버거다.** 기기 로그를 못 붙여도, 탭 후 어떤
   REST 요청이 왔는지/안 왔는지로 "이벤트가 Dart 에 도달했는가"를 판별할 수 있다.
3. **SceneDelegate + firebase_messaging 은 swizzling 을 신뢰하지 말 것.**
   APNs 토큰(1호)에 이어 알림센터 델리게이트(2호)까지 — 앞으로 iOS 알림 관련
   침묵 버그는 무조건 AppDelegate 명시 처리부터 의심.
4. **관리자 권한 SQL 검증은 RLS 를 우회한다** — 앱 조건 재현은 반드시
   `authenticated` + `request.jwt.claims`(tv 포함) 시뮬레이션으로 (0026 계열 교훈).
5. Android 채널은 생성 즉시 불변, FCM 은 기본 채널 메타데이터가 없으면 진동 없는
   폴백 채널을 쓴다. 에뮬레이터로는 진동을 검증할 수 없다.

## 8. 다음에 알림이 이상하면 — 진단 순서

1. 어느 기기·어느 빌드인가? 네이티브 변경이 포함됐다면 재설치했는가?
   (iOS 테스트는 `flutter run --release`)
2. 서버: `notifications` 행 생성됨? `push_status = sent`? (`push_error` 확인)
3. 표시 안 됨(Android): dumpsys 로 채널 확인 — `default_high` 인가 폴백인가.
   표시 안 됨(iOS 포그라운드): 콘솔에 `push-native: willPresent` 찍히는가.
4. 탭 라우팅: 탭 직후 API 로그에 대상 단건 조회(`v_chat_rooms?id=eq...` 등)가
   있는가 — 없으면 탭 이벤트 유실(네이티브), 있는데 폴백이면 조회 실패(재시도/데이터).
5. realtime 수신: `realtime.subscription` 에 구독이 있는가, claims 의 sub 가
   맞는가.
6. 진동/무음(iPhone): 페이로드 의심 전에 **알림을 가로채는 다른 Apple 기기**부터
   — **Mac iPhone 알림 미러링**(개발 환경에선 거의 항상 켜져 있음!), Apple Watch,
   예약된 요약, 집중 모드 순. 판별 실험: 잠금 상태로 발송 → 화면이 안 켜지고
   조용히 쌓이면 "조용히 전달" 계열 확정. 페이로드 실증은 willPresent 의 aps
   원문 로그(`push-native: willPresent — ... aps=...`)로. Android 는 채널부터.
