# 0019 — 펫 개체 일치 검증 + 신뢰도 누적 (AI 인증 기준 사진 대조)

> PawMate(`pmdart`) + Supabase 백엔드(`pmdb`) 기준.
> **선행 의존: 0018(게시글 사진 실존 검증)** — `photo_verifications`, `verify-post-photo`,
> `create_post_verified`, `tg_posts_check_write` 의 토큰 검사가 이미 구현되어 있다고 전제한다.
> **선행 의존: 0017(지역 인증)** — `users.region_code`, `verify-location`.
> AI 공급자: **Google Gemini 2.5 Pro** (`gemini-2.5-pro`, generateContent + 구조화 출력, 멀티이미지).

---

## 0. 목표와 결정 요약

- 0018은 "실제 살아있는 개/고양이 + 그 동네에서 촬영"까지만 본다. 0019는 **"그게 이 사용자가 등록한 그 반려동물이냐"** 를 추가로 본다.
- **개체 식별(re-ID)이 아니라 일치도 추정**이다. 같은 품종·털색의 다른 개는 통과할 수 있다 — 이 한계를 전제로 설계한다. 목표는 사기 문턱을 높이는 것.
- 대조 기준은 **대표사진(`pets.image_url`)이 아니라** 펫마다 별도로 두는 **"AI 인증용 기준 사진"**. 기준 사진은 0018과 동일한 카메라+실존+위치 파이프라인을 통과해야 등록된다(갤러리에서 올린 대표사진은 신뢰하지 않음).
- 기준 사진이 없는 펫으로는 **검증 카테고리 게시글을 쓸 수 없다**(대표사진이 있어도). 먼저 기준 사진을 등록하도록 유도한다.
- 게시글 사진은 기준 사진과 Gemini 로 **2장 대조**해 `match_score`(0~1)를 얻는다. 실존+지역만 통과하면 **게시는 허용(소프트)**, 개체 일치는 별도로 기록한다(불일치 = 차단 아님, 검수 플래그).
- **신뢰도는 펫 단위**로 누적한다. 단 **검증 필수 카테고리(walk_together/walk_proxy/care/give_away)의 매칭만** `pets.pet_match_count` 에 가산한다. free/adoption 은 가산하지 않는다(애초에 미검증).
- 신뢰도는 **펫 프로필 + 사용자 프로필**(보유 펫 합산)에 배지로 노출한다.

> **0018 대비 의미 변경:** `posts.is_pet_verified` 는 0018에서 "사진 실존 통과"였으나, 0019부터 **"등록 펫과 개체 일치"** 를 뜻한다(`ai_matched`).

## 1. 전체 플로우 (end-to-end)

### 1-1. 기준 사진 등록 (펫 1마리당 1회, 소유자)

```
[pmdart] 반려동물 수정 화면 → "AI 인증용 사진" → 카메라 촬영
   │  petId + 좌표 + 사진(base64) 을 verify-post-photo(purpose='reference') 로 전송
   ▼
[Edge: verify-post-photo]  (커스텀 JWT → uid)
   ① 펫 소유자(owner) 확인  ② 활동지역 인증  ③ 모의위치 거절
   ④ 촬영지 역지오코딩 == users.region_code  ⑤ Gemini 실존 판별(1장)
   ⑥ 통과: media 에 <uid>/pets/ref/<ts>.jpg 업로드
        + record_photo_verification(purpose='reference', pet_id)
        + set_pet_ai_reference(pet, verification) → pets.ai_ref_* 채움
   ▼ { pass:true, imageUrl }
```

### 1-2. 게시글 작성 (개체 대조)

```
[pmdart] 게시글 작성 → 연결할 반려동물 선택 → "사진 촬영"
   │  (선택 펫에 기준 사진 없으면 차단 + 즉석 등록 유도)
   │  petId + 좌표 + 사진(base64) 을 verify-post-photo(purpose='post') 로 전송
   ▼
[Edge: verify-post-photo]
   ① 보호자 확인  ② 활동지역  ③ 모의위치  ④ 촬영지 == region_code
   ⑤ 펫 기준 사진(ai_ref_image_path) 다운로드 → Gemini 2장 대조
        → 실존(촬영본) + match_score
        · 기준 사진 없음 → fail(no_reference)  → [pmdart] 등록 유도
   ⑥ 통과(실존+지역): media 에 <uid>/posts/<ts>.jpg 업로드
        + record_photo_verification(purpose='post', pet_id, match_score, ai_matched)
   ▼ { pass:true, token, imageUrl, species, matched, matchScore }
[pmdart] create_post_verified(p_pet_ids, p_photo_token, …)
   ▼
[posts INSERT 트리거]  토큰 purpose='post' + pet_id 강제, is_pet_verified := ai_matched
[create_post_verified]  토큰 pet_id ∈ p_pet_ids 검증 + ai_matched 면 pet_match_count +1
   ▼  게시글 등록 완료
```

핵심: 기준 사진 자체가 검증된 실물이고, 게시 사진을 그 기준과 대조한다. **불일치라도 게시는 막지 않되**(오탈락 부담↓) 신뢰도는 일치할 때만 쌓인다.

## 4. AI 개체 대조 — Gemini 2.5 Pro (2장)

### 4-1. 호출

- `purpose='reference'`: 1장(촬영본) 실존 판별만(0018 `classifyPet` 동일).
- `purpose='post'`: **2장** — 기준 사진 + 촬영본을 함께 보낸다(`inline_data` 2개). 기준 사진은 service_role 로 `admin.storage.download(ai_ref_image_path)` 후 base64.

### 4-2. 응답 스키마(post)

```jsonc
{
  "species": "dog | cat | other | none",
  "dog_real": 0.0, "cat_real": 0.0, "dog_fake": 0.0, "cat_fake": 0.0,  // 촬영본 실존
  "same_pet": true,        // 같은 개체로 보이는가
  "match_score": 0.0,      // 개체 일치도 0~1 (품종·털색·무늬·반점·체형 종합)
  "match_reason": "짧은 근거(한국어, 80자 이내)"
}
```

### 4-3. 판정 (서버에서 계산)

```
real  = max(dog_real, cat_real);  fake = max(dog_fake, cat_fake)
PASS(게시 허용)  ⇔  real >= AI_PASS_THRESHOLD(0.70) AND real > fake     // 실존
matched(개체 일치) ⇔  match_score >= MATCH_THRESHOLD(0.60)               // 소프트
```

> `match_score` 가 낮아도 PASS(게시 허용)는 유지된다. matched=false 면 `is_pet_verified=false` 로 기록되고 신뢰도 가산은 없다(검수 대상). 두 임계값 모두 운영 중 튜닝 대상.

## 5. Edge Function `verify-post-photo` 확장 (pmdb)

0018 함수에 `petId`(검증 흐름 필수) + `purpose`('post' 기본 | 'reference') 를 추가했다. JWT 수동 검증·`reverseGeocode`·`--no-verify-jwt` 배포는 0018 그대로.

요청:

```jsonc
POST /functions/v1/verify-post-photo
Authorization: Bearer <login JWT>
{ "imageBase64": "...", "mimeType": "image/jpeg",
  "lat": 37.2, "lng": 127.07, "accuracy": 12.4, "isMocked": false,
  "petId": "<uuid>", "purpose": "post" }   // purpose: "reference" | "post"
```

공통 선행: 펫 보호자 확인(reference 는 owner 전용 → `forbidden_not_owner`) → 활동지역 인증(`not_verified`) → 모의위치(`mock_location`) → 역지오코딩 일치(`geocode_failed`/`region_mismatch`).

- **purpose='reference'**: 실존 판별(1장) → 통과 시 `<uid>/pets/ref/<ts>.jpg` 업로드 + `record_photo_verification(purpose='reference', pet_id)` + `set_pet_ai_reference` → `{ pass:true, imageUrl, species }`.
- **purpose='post'**: `ai_ref_image_path` 없으면 `{ pass:false, reason:"no_reference" }`. 있으면 기준 사진 다운로드 → `classifyAndMatch`(2장) → 실존 실패면 `not_real_pet`, 통과면 `<uid>/posts/<ts>.jpg` 업로드 + `record_photo_verification(purpose='post', pet_id, match_score, ai_matched)` → `{ pass:true, token, imageUrl, species, matched, matchScore }`.

응답(실패) 사유: `not_verified` / `mock_location` / `geocode_failed` / `region_mismatch` / `not_real_pet` / `no_reference` / `ai_unavailable`, 권한 `forbidden`·`forbidden_not_owner`·`pet_not_found`.

시크릿: 0018 과 동일(`GEMINI_API_KEY`, `NAVER_MAP_KEY(_ID)`, `JWT_SECRET`, `SUPABASE_*`). 신규 없음.

## 6. DB 반영 RPC + 트리거 (pmdb)

### 6-1. `record_photo_verification` 재정의

전달된 verification 이 `result='pass' AND purpose='reference' AND pet_id=p_pet` 인지 확인 후 `pets.ai_ref_*`/`ai_ref_verified_at` 를 채운다(`migration: 20260626091300`). SECURITY DEFINER.

### 6-3. `tg_posts_check_write` 확장

0018 본문 보존 + 검증 카테고리 토큰 검사에 `purpose='post' AND pet_id is not null` 추가, `new.is_pet_verified := v_pv.ai_matched`(매칭 성공 시에만 true)(`migration: 20260626091400`).

### 6-4. `create_post_verified` 확장

검증 카테고리에서 `p_photo_token` 의 `pet_id` 가 `p_pet_ids` 에 포함되는지 확인(아니면 raise → 다른 펫 우회 차단), 게시 후 `ai_matched` 면 `update pets set pet_match_count = pet_match_count + 1`(`migration: 20260626091500`).

## 8. 보안 / 위변조 방어 정리

| 위협 | 방어 |
| --- | --- |
| 대표사진(갤러리)을 기준으로 악용 | 대조 기준은 카메라+실존+위치 통과한 `ai_ref_*` 만. 대표사진은 신뢰 안 함 |
| 남의 펫/다른 펫으로 글 작성 | 토큰 `pet_id` 가 게시 연결 펫(`p_pet_ids`)에 포함돼야 함(create_post_verified) |
| 기준 사진 위조 | reference 등록도 실존+지역 검증 통과 필요, owner 만, service_role 만 `ai_ref_*` 기록 |
| 신뢰도(`pet_match_count`) 위조 | pets 검증 컬럼 클라 GRANT 미부여 → definer/RPC 만 가산 |
| 검증 컬럼 클라 위조 | `record_photo_verification`/`set_pet_ai_reference` authenticated 실행 불가(service_role 전용) |
| 같은 품종·색 다른 개 | (한계) 완전 차단 불가 → 소프트 플래그 + 반복 매칭 신뢰 누적으로 억제 |

> 0018 4중 방어(카메라 강제·위치 일치·실존 AI·1회용 토큰) 위에 **개체 대조**를 1겹 더한 것. 개체 동일성의 절대 증명은 불가하며, 비용 대비 충분한 억제를 목표로 한다.

## 9. 엣지 케이스

- **기준 사진 없음**: `no_reference` → 등록 유도. 대표사진이 있어도 동일.
- **공동보호자(co_guardian)가 글 작성**: 게시·매칭은 가능하나 **기준 사진 등록은 owner 만**.
- **다중 펫 연결(산책/돌봄)**: 사진은 1마리에 대한 것 → 촬영 대상 펫을 지정해 그 펫에 바인딩. 토큰 pet_id ∈ 선택 펫이면 통과.
- **개체 불일치(낮은 match_score)**: 게시는 되되 `is_pet_verified=false`, 신뢰도 미가산, 검수 대상 안내.
- **노화·미용으로 외형 변화**: 오탈락 가능 → 기준 사진 재등록(`다시 촬영`)으로 갱신.
- **Gemini 지연/오류**: `ai_unavailable` → 통과 아님, 재시도 안내.
- **촬영 펫을 게시 후 선택 해제**: 클라가 사진/토큰을 무효화, 서버도 `pet_id ∈ p_pet_ids` 로 거절.
