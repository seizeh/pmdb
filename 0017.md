# 0017 — 지역 인증 설계 (활동 지역 + GPS 동네인증)

> PawMate(`pmdart`) + Supabase 백엔드(`pmdb`) 기준.
> 게시글 사진 실존 검증(반려동물 + 촬영 위치 = 활동 지역)의 **선행 기능**으로, 사용자의 "활동 지역"을 GPS 현장 인증으로 확립한다.
> 역지오코딩 공급자: **Naver Cloud Platform Maps – Reverse Geocoding**.

---

## 0. 목표와 결정 요약

- **활동 지역 입력은 타이핑이 아니라 "현재 위치 기반 행정동 1개 설정"** 으로 한다. (당근마켓식)
- **인증은 GPS 현장 인증 → Edge Function 서버 검증** 으로만 가능하다. 클라이언트는 인증 플래그를 직접 쓸 수 없다.
- 활동 지역 단위는 **행정동(읍·면·동)**. 시/구는 너무 넓어 검증 의미가 약하고, 전체 주소는 사생활 노출.
- 저장 키는 **행정동코드(`region_code`)**. 게시글 작성 시 `posts.region_code` 와 **코드 일치 비교**로 지역 검증이 끝난다.
- 지금은 **단일 동네**. 다동네(여러 동 인증)는 `user_verified_areas` 테이블이 필요하므로 후속 확장으로 미룬다.

## 1. 이미 존재하는 스키마 (재사용)

`users` (0002.md):

| 컬럼 | 타입 | 용도 |
| --- | --- | --- |
| `latitude` | `decimal(10,7)` | 인증된 실제 좌표 |
| `longitude` | `decimal(10,7)` | 인증된 실제 좌표 |
| `address` | `varchar(100)` | 표시 라벨 (예: "경기 화성시 동탄2동") |
| `is_location_verified` | `boolean` | 인증 여부 (클라 수정 불가) |
| `last_verified_at` | `timestamptz` | 마지막 인증 시각 → 만료 판정 |
| `location_verify_fail_count` | `smallint` | 연속 실패 횟수 (throttle) |
| `location_verify_blocked_until` | `timestamptz` | 차단 만료 시각 |

`location_verifications` (지역 인증 이력 로그):

```
user_id, verified_lat, verified_lng, verified_radius_meters(not null),
result in ('success','failed','blocked'), fail_reason, created_at
```

> `posts` 에는 이미 `region_code varchar(20)`, `actual_lat/lng`, `display_lat/lng`, `display_address`, `is_location_hidden` 가 존재한다. 즉 게시글-지역 비교를 위한 컬럼은 추가가 필요 없다.

## 2. 스키마 보강 (이번에 추가)

행정동코드를 빠르게 비교하기 위해 `users` 에 `region_code` 한 컬럼만 추가한다.

```sql
-- migration: 2026XXXX_users_region_code.sql
alter table public.users
  add column if not exists region_code varchar(20);

comment on column public.users.region_code is
  '인증된 활동 지역의 행정동코드(Naver admcode, 10자리). is_location_verified=true 일 때 채워짐';

-- 게시글 작성 시 users.region_code = posts.region_code 비교를 위한 인덱스
create index if not exists users_region_code_idx
  on public.users (region_code);
```

**[중요] 인증 컬럼 쓰기 권한 차단 (보안 핵심).** 현재 `users` 는 `grant insert, update, delete on all tables ... to authenticated` 로 **모든 컬럼 UPDATE 가 열려 있다**(0005.md). `users` 는 SELECT 만 컬럼단위로 막혀 있고 UPDATE 는 막혀 있지 않아, 지금 그대로면 클라이언트가 `is_location_verified` 를 직접 위조할 수 있다(§9). 이번 마이그레이션에서 UPDATE 를 회수하고 안전 컬럼만 재부여한다.

```sql
-- migration: 2026XXXX_users_update_column_grants.sql
revoke update on public.users from authenticated;
grant update (
  nickname,
  profile_image_url, profile_image_thumbnail_url,
  profile_image_mime_type, profile_image_file_size,
  push_enabled
) on public.users to authenticated;

-- (선택) 만료 표시용으로 last_verified_at 읽기 허용
grant select (last_verified_at) on public.users to authenticated;
```

> 클라이언트가 실제로 수정하는 컬럼은 `profile_repository.updateProfile` 의 `nickname`·`profile_image_url` 뿐이다(전수 확인됨). 배포 전 `from('users').update(...)` 사용처를 한 번 더 감사할 것. `is_location_verified / region_code / latitude / longitude / last_verified_at / location_verify_*` 는 GRANT 미부여 → SECURITY DEFINER RPC(소유자 권한으로 컬럼 GRANT 우회)로만 변경된다.

> `location_verifications.verified_radius_meters` 는 NOT NULL 이므로, 포인트 인증에서는 **단말 GPS 정확도(accuracy, m)** 를 그대로 저장한다. (정확도 미상이면 0)

## 4. 활동 지역 입력 UX 스펙 (pmdart)

원칙: **사용자는 주소를 입력하지 않는다.** 현재 위치로만 동네를 잡는다.

내정보 탭(`my_info_tab.dart`)에 "활동 지역" 항목 추가:

- 미인증 상태: `활동 지역 · 미인증` + [동네 인증하기] 버튼.
- 인증 상태: `동탄2동 · 인증됨 ✓` + (만료 임박 시) [재인증] 버튼.

[동네 인증하기] → 신규 `LocationVerifyScreen`:

1. 안내 텍스트("현재 계신 동네를 인증합니다. 위치 권한이 필요해요").
2. [이 위치로 인증] 버튼 → GPS 좌표 획득(§11) → Edge Function 호출 → 결과 표시.
3. 성공 시 동네명 노출 후 내정보로 복귀, 실패/차단 시 사유 안내.

> 지도 연동(`flutter_naver_map`)은 이미 완료된 상태이므로 본 작업 범위에서 제외한다. 좌표 획득은 지도 SDK 대신 `geolocator` 로 직접 한다(§11 사유).

> 표시 단위는 **행정동만**(area3). 상위(시/구)는 부가 정보로만 보조 표기.

## 5. 인증 플로우 (end-to-end)

```
[pmdart] 내정보 → "동네 인증하기"
   │  geolocator: 고정확도 현재 좌표(lat,lng,accuracy,isMocked) 획득
   ▼
[Edge Function: verify-location]  (커스텀 JWT 수동 검증 → sub = uid)
   ① 차단 확인: users.location_verify_blocked_until > now() → 즉시 거절(blocked)
   ② 모의위치 거절: isMocked=true → 실패 처리(fail_reason='mock_location')
   ③ Naver Reverse Geocoding(coords=lng,lat, orders=admcode,…)
        → 실패/no results → 실패 처리(fail_reason='geocode_failed')
   ④ admcode 결과에서 region_code(code.id) + area1~3 라벨 추출
   ⑤ RPC record_location_verification(success, …) 호출(service_role)
        → users 인증 컬럼 업데이트 + 로그 insert + fail_count=0
   ▼
[pmdart] "○○동 · 인증됨 ✓" 표시
```

실패/차단 경로도 동일 RPC로 로그를 남기고 `fail_count` 를 증가시킨다.

## 6. Edge Function `verify-location` 스펙 (pmdb)

위치: `supabase/functions/verify-location/index.ts` (Deno). 기존 함수들과 동일하게 `_shared/cors.ts` 사용, **`--no-verify-jwt` 로 배포하고 함수 내부에서 커스텀 JWT 를 `JWT_SECRET` 으로 직접 검증**한다(login/phone 함수와 동일 패턴). 앱은 `main.dart` 의 `Supabase.initialize(accessToken: …)` 콜백으로 모든 요청에 로그인 JWT 를 자동 첨부하므로, `functions.invoke('verify-location', …)` 호출 시 별도 헤더 설정이 필요 없다.

요청:

```jsonc
POST /functions/v1/verify-location
Authorization: Bearer <login JWT>        // sub = uid
{
  "lat": 37.2008,
  "lng": 127.0726,
  "accuracy": 12.4,        // m, 선택
  "isMocked": false        // 단말 모의위치 플래그
}
```

응답(성공):

```jsonc
{
  "verified": true,
  "regionCode": "4159011000",
  "regionName": "동탄2동",
  "address": "경기 화성시 동탄2동"
}
```

응답(실패/차단):

```jsonc
{ "verified": false, "reason": "blocked", "blockedUntil": "2026-..." }
{ "verified": false, "reason": "geocode_failed" }
{ "verified": false, "reason": "mock_location" }
```

핵심 로직(스켈레톤):

```ts
import { corsHeaders } from "../_shared/cors.ts";

const SUPABASE_URL = Deno.env.get("SUPABASE_URL")!;
const SERVICE_ROLE = Deno.env.get("SUPABASE_SERVICE_ROLE_KEY")!;
const NAVER_KEY_ID = Deno.env.get("NAVER_MAP_KEY_ID")!;
const NAVER_KEY     = Deno.env.get("NAVER_MAP_KEY")!;

const FAIL_LIMIT = 5;          // 연속 실패 임계
const BLOCK_MINUTES = 60;      // 차단 시간

Deno.serve(async (req) => {
  if (req.method === "OPTIONS") return new Response("ok", { headers: corsHeaders });

  // 1) JWT 검증 → uid (sub) 추출  (login 함수와 동일 HS256 시크릿)
  const uid = await getUidFromJwt(req);          // 실패 시 401
  const { lat, lng, accuracy = 0, isMocked = false } = await req.json();

  const admin = serviceClient(SUPABASE_URL, SERVICE_ROLE);

  // 2) 차단 여부
  const { data: u } = await admin.from("users")
    .select("location_verify_blocked_until").eq("id", uid).single();
  if (u?.location_verify_blocked_until &&
      new Date(u.location_verify_blocked_until) > new Date()) {
    return json({ verified: false, reason: "blocked",
                  blockedUntil: u.location_verify_blocked_until });
  }

  // 3) 모의위치
  if (isMocked) {
    await admin.rpc("record_location_verification", {
      p_user: uid, p_lat: lat, p_lng: lng, p_accuracy: Math.round(accuracy),
      p_result: "failed", p_fail_reason: "mock_location",
      p_region_code: null, p_address: null,
      p_fail_limit: FAIL_LIMIT, p_block_minutes: BLOCK_MINUTES,
    });
    return json({ verified: false, reason: "mock_location" });
  }

  // 4) Naver 역지오코딩
  const geo = await reverseGeocode(lng, lat);    // §7
  if (!geo) {
    await admin.rpc("record_location_verification", {
      p_user: uid, p_lat: lat, p_lng: lng, p_accuracy: Math.round(accuracy),
      p_result: "failed", p_fail_reason: "geocode_failed",
      p_region_code: null, p_address: null,
      p_fail_limit: FAIL_LIMIT, p_block_minutes: BLOCK_MINUTES,
    });
    return json({ verified: false, reason: "geocode_failed" });
  }

  // 5) 성공 반영
  await admin.rpc("record_location_verification", {
    p_user: uid, p_lat: lat, p_lng: lng, p_accuracy: Math.round(accuracy),
    p_result: "success", p_fail_reason: null,
    p_region_code: geo.regionCode, p_address: geo.address,
    p_fail_limit: FAIL_LIMIT, p_block_minutes: BLOCK_MINUTES,
  });
  return json({ verified: true, regionCode: geo.regionCode,
                regionName: geo.regionName, address: geo.address });
});
```

필요한 시크릿(Edge Function Secrets):

- `NAVER_MAP_KEY_ID`, `NAVER_MAP_KEY` — Naver 역지오코딩 인증
- `JWT_SECRET` — 로그인 JWT 검증(이미 사용 중)
- `SUPABASE_SERVICE_ROLE_KEY`, `SUPABASE_URL` — 권한 작업

## 7. Naver Reverse Geocoding 연동 상세

공식 스펙(2026 기준):

- **Method / URL**: `GET https://naveropenapi.apigw.ntruss.com/map-reversegeocode/v2/gc`
- **헤더**: `x-ncp-apigw-api-key-id: {KEY_ID}`, `x-ncp-apigw-api-key: {KEY}`
- **쿼리**: `coords={lng},{lat}` (경도,위도 순서 주의), `output=json`, `orders=admcode,legalcode,addr`
- **좌표계**: 기본 `EPSG:4326`(WGS84) — 단말 GPS 그대로 사용 가능

응답에서 행정동을 뽑는다. `results[]` 중 `name == "admcode"` 항목 사용:

- `code.id` → **행정동코드(region_code)**
- `region.area1.name` → 시/도, `area2.name` → 시/군/구, `area3.name` → **읍/면/동(표시 핵심)**

구현 스케치:

```ts
async function reverseGeocode(lng: number, lat: number) {
  const url = "https://naveropenapi.apigw.ntruss.com/map-reversegeocode/v2/gc"
    + `?coords=${lng},${lat}&output=json&orders=admcode,legalcode,addr`;
  const res = await fetch(url, {
    headers: {
      "x-ncp-apigw-api-key-id": NAVER_KEY_ID,
      "x-ncp-apigw-api-key": NAVER_KEY,
    },
  });
  if (!res.ok) return null;
  const body = await res.json();
  if (body?.status?.code !== 0 || !Array.isArray(body.results)) return null;

  const adm = body.results.find((r: any) => r.name === "admcode");
  if (!adm) return null;

  const a1 = adm.region?.area1?.name ?? "";
  const a2 = adm.region?.area2?.name ?? "";
  const a3 = adm.region?.area3?.name ?? "";   // 행정동
  if (!a3 || !adm.code?.id) return null;       // 바다/주소 없음 등

  return {
    regionCode: adm.code.id as string,         // 예: "4159011000"
    regionName: a3,                            // 예: "동탄2동"
    address: [a1, a2, a3].filter(Boolean).join(" "),
  };
}
```

> 주의: NCP 콘솔에서 Maps 서비스(Reverse Geocoding) 이용 신청 후 발급된 키를 사용한다. 일부 신규 계정은 게이트웨이/키 헤더가 다를 수 있으니 콘솔의 "이용 신청 → 인증 정보"에서 호스트·헤더명을 최종 확인한다.

## 8. DB 반영 RPC (security definer)

Edge Function이 service_role 로 호출. 성공/실패/차단을 한 트랜잭션에서 처리한다.

```sql
-- migration: 2026XXXX_record_location_verification.sql
create or replace function app.record_location_verification(
  p_user         uuid,
  p_lat          numeric,
  p_lng          numeric,
  p_accuracy     int,
  p_result       text,           -- 'success' | 'failed' | 'blocked'
  p_region_code  varchar,
  p_address      varchar,
  p_fail_reason  varchar,
  p_fail_limit   int default 5,
  p_block_minutes int default 60
) returns void
language plpgsql
security definer
set search_path = ''
as $$
begin
  -- 이력 로그(반경 = GPS 정확도)
  insert into public.location_verifications
    (user_id, verified_lat, verified_lng, verified_radius_meters, result, fail_reason)
  values
    (p_user, p_lat, p_lng, greatest(coalesce(p_accuracy,0),0), p_result, p_fail_reason);

  if p_result = 'success' then
    update public.users
       set latitude = p_lat,
           longitude = p_lng,
           region_code = p_region_code,
           address = p_address,
           is_location_verified = true,
           last_verified_at = now(),
           location_verify_fail_count = 0,
           location_verify_blocked_until = null,
           updated_at = now()
     where id = p_user;
  else
    -- 실패 누적 → 임계 초과 시 차단
    update public.users
       set location_verify_fail_count = location_verify_fail_count + 1,
           location_verify_blocked_until = case
             when location_verify_fail_count + 1 >= p_fail_limit
               then now() + make_interval(mins => p_block_minutes)
             else location_verify_blocked_until end,
           updated_at = now()
     where id = p_user;
  end if;
end;
$$;

revoke all on function app.record_location_verification(
  uuid,numeric,numeric,int,text,varchar,varchar,varchar,int,int) from public, anon, authenticated;
```

> `is_location_verified` / `location_verify_*` 는 이 함수(service_role 경로)에서만 변경된다. 클라이언트의 `users` UPDATE RLS 에는 이 컬럼들이 포함되지 않으므로 위조 불가.

## 9. RLS / 보안 원칙 (현재 코드의 빈틈 + 수정)

- **현재 상태(주의).** `users_update` RLS 는 행만 검사(`id = app.uid()`)하고, 권한은 `grant insert, update, delete on all tables ... to authenticated` 로 **모든 컬럼 UPDATE 가 열려 있다**(0005.md). 즉 지금 그대로면 클라이언트가 `update users set is_location_verified = true where id = <me>` 를 실행할 수 있다. **SELECT 만 컬럼단위로 막혀 있고 UPDATE 는 막혀 있지 않다.**
- **수정.** §2 의 `revoke update on public.users` + 안전 컬럼만 `grant update` 로 인증 컬럼 쓰기를 차단한다. 이후 `is_location_verified / region_code / latitude / longitude / last_verified_at / location_verify_*` 는 **SECURITY DEFINER RPC(소유자 권한으로 컬럼 GRANT 우회)** 로만 변경 가능.
- 인증 컬럼 갱신 경로는 오직 `verify-location` Edge Function → `record_location_verification`.
- `users` SELECT 는 이미 컬럼단위로 제한(안전 컬럼만, 0005.md). `location_verifications` 는 본인+admin SELECT, INSERT 는 RPC(service_role/definer)로만.

## 10. 스푸핑 완화

GPS 는 모의위치로 속일 수 있다. 완전 차단은 불가하나 다음으로 충분히 억제한다.

- **모의위치 플래그**: Android `Position.isMocked` (geolocator) true 면 실패 처리.
- **Throttle**: 연속 실패 `FAIL_LIMIT`(기본 5) 초과 시 `BLOCK_MINUTES`(기본 60분) 차단.
- **재인증 만료**: `last_verified_at` 기준 N일(예: 60일) 경과 시 만료로 간주 → 게시 전 재인증 요구.
- **2차 방어선**: 게시글 작성 단계에서 "촬영 위치의 행정동 = `users.region_code`" 재확인(후속 0018 문서). 따라서 활동 지역 인증을 우회해도 게시 단계에서 다시 걸린다.

## 11. Flutter 변경 사항 (pmdart)

패키지 추가(`pubspec.yaml`). 지도 연동은 완료돼 있지만, **정확도(accuracy)와 모의위치 플래그(isMocked)** 를 얻기 위해 인증 좌표는 `geolocator` 로 직접 획득한다(지도 SDK와 무관). 위치 권한 인프라는 기존 지도 설정을 재사용할 수 있다.

```yaml
geolocator: ^13.0.0
permission_handler: ^11.3.0
```

플랫폼 권한:

- Android `AndroidManifest.xml`: `ACCESS_FINE_LOCATION`
- iOS `Info.plist`: `NSLocationWhenInUseUsageDescription`

신규/수정:

- `services/location_service.dart` (신규): 권한 요청 + 고정확도 좌표/accuracy/isMocked 획득.
- `services/location_repository.dart` (신규): `verify-location` Edge Function 호출 래퍼.
- `screen/location_verify_screen.dart` (신규): 인증 UI.
- `screen/tabs/my_info_tab.dart` (수정): "활동 지역" 항목 + 인증/만료 표시.
- `models/profile.dart` + `services/profile_repository.dart` (수정): `fetchProfile` 의 `public_profiles` 조회에 이미 노출돼 있는 `address`, `is_location_verified` 를 추가로 select(동네명은 `address` 에서 파생, 별도 필드 불필요). 만료 표시가 필요하면 `last_verified_at`(§2 의 SELECT 그랜트 추가분) 포함. `region_code` 는 서버 비교 전용이라 클라에 노출하지 않는다(최소 권한). `public_profiles` 뷰 정의는 그대로 사용 가능(이미 `address`·`is_location_verified` 포함).

호출 예:

```dart
final res = await Supabase.instance.client.functions.invoke(
  'verify-location',
  body: {'lat': pos.latitude, 'lng': pos.longitude,
         'accuracy': pos.accuracy, 'isMocked': pos.isMocked},
);
final ok = res.data['verified'] == true;
```

## 12. 게시글 단계와의 연계 (다음 문서 0018)

본 문서는 "활동 지역 확립"까지다. 게시글 사진 검증은 다음을 추가한다(요약):

1. 작성 시 게시글 좌표 → 역지오코딩 → `region_code` 산출.
2. `region_code == users.region_code` 가 아니면 차단(또는 `is_location_verified=false` 면 차단).
3. (별도) 카메라 강제 + 반려동물 AI 판별.
   → `posts` INSERT 트리거(`app.tg_posts_check_write`)에 지역 일치 검사를 추가한다.

## 13. 엣지 케이스

- **바다/주소 없는 좌표**: `admcode` 결과 없음 → `geocode_failed`. (Naver는 해상 좌표에 결과 미반환)
- **세종시**: 시/군/구 데이터가 없을 수 있음 → `area2` 공백 허용, `area3` 로 동 판정.
- **권한 거부**: 앱에서 위치 권한 거부 시 인증 진입 차단 + 설정 유도.
- **GPS 미획득(실내 등)**: 타임아웃 → 사용자에게 재시도 안내(서버 호출 안 함).
