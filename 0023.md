## 0. 목표와 결정 요약

flutter_naver_map 은 네이티브 뷰를 Flutter 트리에 합성하는 **PlatformView** 다. 이 위로 `Navigator.push` 전환이 돌면 깜빡임·검은 플래시·전환 잔렉, 심하면 맵 dispose/recreate 로 카메라·마커 상태 손실과 재초기화 비용이 터진다. 이 문서는 그 구조적 한계를 **회피가 아니라 아키텍처로** 푼다.

### 결정값(확정)

| 항목 | 결정 | 비고 |
| --- | --- | --- |
| 맵 인스턴스 | **앱 내 단 1개, 루트에 상주** | 재생성 금지 |
| 맵 위 상세 화면 | **라우트 push 금지 → 바텀시트/오버레이** | DraggableScrollableSheet 중심 |
| 맵 생존 | **keep-alive**(IndexedStack/Offstage) | 탭 전환·뒤로가기에도 유지 |
| 불가피한 풀스크린 전환 | **스냅샷 스왑**(정적 이미지로 애니메이션) | 네이티브 뷰를 전환에서 배제 |
| 합성 모드 | Android **Hybrid Composition/TLHC** 확인, iOS UiKitView | 플랫폼별 §11 |

### 핵심 원칙 4가지

1. **One Map** — 맵 위젯은 앱에서 하나만 생성해 살려둔다.
2. **No Push Over Map** — 맵을 덮는 라우트를 push 하지 않는다. 상세는 같은 라우트 안에서 시트/오버레이로 얹는다.
3. **Keep Alive** — 화면 이동에도 맵을 dispose 하지 않는다.
4. **Snapshot on Transition** — 정말 풀스크린 전환이 필요하면 라이브 맵을 정적 스냅샷으로 갈아끼우고 전환한다.

## 1. 문제 정의 — 왜 터지는가

> ⚠️ **정정(→ §13):** 아래 "전환 재합성/검은 플래시" 모델은 이 사태를 시작한 **백지
> 버그의 진짜 원인이 아니었다.** 실제 원인은 본문 콘텐츠의 **무한 너비 레이아웃 실패**다.
> 단, "맵을 트리에서 넣다 뺐다 하면 dispose/recreate 비용"(§7 안티패턴)은 **사실이었고**,
> 1차 구현이 그걸 어겨 실제로 재로딩 잼이 났다.

```
[Flutter 위젯 트리]            [네이티브 합성]
 ┌───────────────┐
 │  Flutter UI    │  ──────►  Flutter 텍스처 레이어
 │  ┌──────────┐  │
 │  │ NaverMap │  │  ──────►  ★ 네이티브 PlatformView 레이어 (별도 합성)
 │  └──────────┘  │
 └───────────────┘
```

- `Navigator.push` 전환 애니메이션 → **네이티브 레이어가 매 프레임 재합성** → 깜빡임/검은 플래시/잔렉.
- 라우트 스택 변화로 맵 위젯이 트리에서 빠지면 **dispose → 다음에 recreate** → 카메라 위치·마커·오버레이 전부 재생성(비싸고 깜빡임).
- Android Hybrid Composition 에서 특히 두드러짐. iOS UiKitView 는 상대적으로 안정적이나 동일 원인 존재.

**결론:** 전환 애니메이션과 라우트 스택에 네이티브 맵을 끌어들이는 순간 진다. 맵을 전환에서 **분리**해야 한다.

## 2. 해결 아키텍처 개요

```
[MapShellScreen]  (라우트는 여기 하나. 평생 1개)
  └─ Stack
       ├─ (0) NaverMap            ← 바닥. keep-alive. 절대 push 로 안 덮음
       ├─ (1) 지도 위 UI 오버레이   ← 검색바/카테고리 토글/내 위치 버튼 (Positioned)
       ├─ (2) 바텀시트 레이어        ← 마커 상세·후기 목록·후기 작성 (DraggableScrollableSheet)
       └─ (3) 풀스크린 오버레이(필요시) ← 스냅샷 스왑으로만 진입
```

- 시설 마커 탭 → **(2) 바텀시트**가 올라옴(라우트 전환 없음, 맵 살아있음).
- 후기 작성(0022) → 시트를 끝까지 끌어 풀높이로(여전히 같은 라우트).
- 전혀 다른 화면(예: 마이페이지)으로 갈 때만 (3) 스냅샷 스왑.

## 3. 단일 맵 + keep-alive 구현

맵을 **앱 루트 근처에 한 번만** 만들고, 탭/화면 전환에도 트리에서 빠지지 않게 유지한다.

```dart
// 탭 구조라면 IndexedStack 으로 모든 탭을 항상 트리에 유지(맵 탭이 dispose 되지 않음)
class RootTabs extends StatefulWidget {
  @override State<RootTabs> createState() => _RootTabsState();
}
class _RootTabsState extends State<RootTabs> {
  int _index = 0;
  @override
  Widget build(BuildContext context) {
    return Scaffold(
      body: IndexedStack(                // ★ Offstage 로 모두 살려둠 → 맵 재생성 없음
        index: _index,
        children: const [
          MapShellScreen(),              // 지도(맵 1개 상주)
          CommunityScreen(),
          MyPageScreen(),
        ],
      ),
      bottomNavigationBar: NavigationBar(
        selectedIndex: _index,
        onDestinationSelected: (i) => setState(() => _index = i),
        destinations: const [ /* 지도 / 커뮤니티 / 마이 */ ],
      ),
    );
  }
}
```

```dart
// 맵 셸: 맵은 여기서 단 한 번 생성. AutomaticKeepAlive 로 보강
class MapShellScreen extends StatefulWidget {
  const MapShellScreen({super.key});
  @override State<MapShellScreen> createState() => _MapShellScreenState();
}
class _MapShellScreenState extends State<MapShellScreen>
    with AutomaticKeepAliveClientMixin {
  @override bool get wantKeepAlive => true;          // ★ 화면이 비활성돼도 유지

  NaverMapController? _controller;

  @override
  Widget build(BuildContext context) {
    super.build(context);                            // keep-alive 필수 호출
    return Stack(children: [
      // (0) 바닥 맵
      NaverMap(
        options: const NaverMapViewOptions(/* 초기 카메라 등 */),
        onMapReady: (c) => _controller = c,          // 컨트롤러 1회 확보
      ),
      // (1) 지도 위 UI (검색바/토글/내 위치) — Positioned 오버레이
      const _MapTopControls(),
      // (2) 바텀시트 레이어
      const _DetailSheetLayer(),
    ]);
  }
}
```

- **IndexedStack** 은 비활성 탭도 트리에 유지 → 맵 dispose 안 됨.
- `AutomaticKeepAliveClientMixin` 으로 PageView/탭 어디서든 보강.
- 컨트롤러는 `onMapReady` 에서 한 번만 잡아 앱 수명 동안 재사용.

## 4. 상세는 push 대신 바텀시트

> ⚠️ **정정(→ §14, §15):** "상세는 시트로"라는 **방향은 맞았다**(최종도 시트다). 그러나
> 여기서 권장한 **`DraggableScrollableSheet` 는 실제로 크래시했다**(`viewport.dart`
> `child.geometry!` null). 최종 해법은 **커스텀 `AnimationController` 시트(`MapBottomSheet`)**
> 다. `showModalBottomSheet`/`DraggableScrollableSheet` 를 라이브 지도 위에 직접 쓰지 말 것.

마커 탭·후기 목록·후기 작성 전부 **같은 라우트 안에서** 시트로 처리한다. `showModalBottomSheet` 는 사실 라우트를 push 하지만 **맵을 완전히 덮지 않는 투명/부분 라우트**라 네이티브 재합성을 거의 자극하지 않는다. 가장 깔끔한 건 **상태로 직접 제어하는 `DraggableScrollableSheet`**.

```dart
// (2) _DetailSheetLayer: 선택된 시설이 있으면 시트 표시
class _DetailSheetLayer extends StatelessWidget {
  const _DetailSheetLayer();
  @override
  Widget build(BuildContext context) {
    final sel = context.watch<MapSelection>().facility;   // 선택 상태(Provider/Riverpod 등)
    if (sel == null) return const SizedBox.shrink();
    return DraggableScrollableSheet(
      initialChildSize: 0.35,    // 마커 탭 시 미리보기 높이
      minChildSize: 0.0,
      maxChildSize: 0.95,        // 끝까지 끌면 풀스크린처럼(라우트 전환은 없음)
      snap: true,
      snapSizes: const [0.35, 0.95],
      builder: (ctx, scroll) => FacilityDetailSheet(  // 0021 정보 + 0022 후기
        facility: sel, scrollController: scroll,
      ),
    );
  }
}
```

- 마커 탭 → `MapSelection.select(facility)` 만 호출 → 시트가 올라온다(맵은 그대로).
- 후기 작성(0021)도 이 시트 안에서 또는 시트 위 다이얼로그/`showModalBottomSheet` 로. **풀스크린 라우트로 빼지 않는다.**
- 닫기는 시트를 내리거나 `MapSelection.clear()`.

> **showModalBottomSheet vs DraggableScrollableSheet:** 둘 다 가능. 전자는 코드가 짧지만 라우트 스택을 건드린다(대개 맵을 안 덮어 무난함). 후자는 라우트를 전혀 안 건드려 가장 안전하고, 시트 높이를 상태로 정밀 제어할 수 있다. **맵 위 상세는 후자를 권장.**

## 5. 불가피한 풀스크린 전환 — 스냅샷 스왑

> ⚠️ **정정(→ §16):** 이 문서의 시그니처 기법이지만, 시설/게시글 상세 시트에는 **결국
> 불필요해서 제거했다.** `takeSnapshot()` 자체는 1.4.4 에 존재하고(아래 시그니처는 실제로
> `Future<File>`) 한때 디딤돌로 썼지만, **라이브 지도 위에 Flutter 시트가 정상 합성**되는 게
> 실기기로 확인되어(§16) 얼릴 이유가 없어졌다. "정말 다른 라우트로 풀스크린 전환"이 필요한
> 경우(예: 마이페이지)에 한해 참고용으로 남겨둔다.

전혀 다른 화면으로 풀스크린 전환이 정말 필요할 때만 쓴다. 라이브 맵을 **정적 이미지로 갈아끼운 뒤** 전환해 네이티브 뷰를 애니메이션에서 뺀다.

```dart
Future<void> pushOverMap(BuildContext context, Widget page) async {
  // 1) 현재 맵을 비트맵으로 캡처 (flutter_naver_map 컨트롤러)
  //    ※ 버전에 따라 시그니처 상이 — takeSnapshot 류 API 확인 필요
  final snapshotFile = await _controller!.takeSnapshot();   // 파일/바이트 반환
  // 2) 라이브 맵 위에 스냅샷 이미지를 덮어 '얼린' 상태로 만든다
  _mapSelection.freezeWithSnapshot(snapshotFile);           // (3) 오버레이에 이미지 표시
  await WidgetsBinding.instance.endOfFrame;                 // 한 프레임 대기(이미지 반영)
  // 3) 이제 전환해도 애니메이션에 도는 건 '이미지'뿐 → 깜빡임 없음
  await Navigator.of(context).push(MaterialPageRoute(builder: (_) => page));
  // 4) 돌아오면 스냅샷 제거 → 라이브 맵 복원
  _mapSelection.unfreeze();
}
```

- 전환 전: 라이브 맵 → **스냅샷 이미지로 위장**.
- 전환 중: 무거운 네이티브 뷰가 아니라 **가벼운 이미지가 애니메이션** → 검은 플래시/잔렉 제거.
- 복귀 후: 이미지 제거하면 맵은 살아있던 그대로(카메라·마커 유지).

> 핵심은 **"전환에 참여하는 픽셀을 네이티브가 아니라 Flutter 이미지로 만든다"**는 것. 스냅샷 API 가 없거나 무거우면, 애초에 4번(시트)으로 설계해 전환 자체를 없애는 게 낫다.

## 6. push 가 꼭 필요하면 — 맵을 안 덮는 라우트 / Overlay

- **부분 화면 패널**: 맵 일부가 보이는 비전면 라우트(`opaque: false` + 부분 크기)로 띄우면 재합성 자극이 적다.
- **`Overlay.of(context).insert`**: 라우트 스택을 건드리지 않고 위젯을 띄운다(툴팁·플로팅 카드·임시 배너에 적합).
- 어느 쪽이든 **맵을 100% 불투명하게 덮는 풀스크린 push 만 피하면** 대부분의 증상이 사라진다.

## 7. 안티패턴 (하지 말 것)

- ❌ 마커 탭마다 `Navigator.push(MapDetailPage())` 로 **맵 위에 풀스크린 라우트** 쌓기 → 본 이슈의 직접 원인.
- ❌ 탭/화면마다 **NaverMap 위젯을 새로 생성** → 매번 재초기화·상태 손실·깜빡임.
- ❌ 맵을 `PageView`/조건부 build 로 **트리에서 자주 넣다 뺐다** → dispose/recreate 반복.
- ❌ 전환 애니메이션 중 맵 카메라/마커를 동시에 갱신 → 합성 부하 가중.
- ❌ Android 합성 모드를 방치(구형 Virtual Display) → 성능·z-order 문제.

## 8. 0021/0022 흐름과의 매핑

| --- | --- | --- |
| 시설 마커 탭 | 상세 페이지 push | `DraggableScrollableSheet` 상세(맵 유지) |

→ 0020/0021 이 이미 바텀시트 흐름이라 **이 아키텍처와 자연스럽게 맞물린다.** 풀스크린 push 만 시트로 치환하면 됨.

## 9. 플랫폼별 주의

- **Android**: Hybrid Composition / TLHC(Texture Layer Hybrid Composition) 경로인지 확인. 구형 Virtual Display 는 z-order·성능 문제. flutter_naver_map·Flutter 버전 최신 유지.
- **iOS**: UiKitView 라 상대적으로 안전하나 동일 원칙 적용. 전환 시 스냅샷 스왑은 동일하게 유효.

## 13. 진짜 원인 — 본문 "무한 너비" 레이아웃 실패

증상: 지도 마커를 탭해 상세를 띄우면 **본문이 백지**(AppBar/프레임만 보이고 내용 0). PART I §1 은
이를 "전환 애니메이션 중 네이티브 재합성/검은 플래시"로 봤지만 **틀렸다.** 결정적 반례 두 개:

1. **이슈 #10** — 지도를 조건부로 아예 빌드하지 않아 트리에 PlatformView 가 하나도 없는데도
   본문이 여전히 백지(= 합성 문제라면 말이 안 됨).
2. 라이브 맵 위에 올린 **`Container` 는 유한 너비로 멀쩡히** 그려짐(= 합성이 아니라 레이아웃 문제).

**메커니즘:** 이 화면 컨텍스트(`IndexedStack` > keep-alive `MapTab` > 조건부 `Scaffold` > 시트)에서
본문이 받는 너비 제약이 **tight-infinity** 로 손상됐다. 그 결과:

- **머티리얼 버튼**(`FilledButton`/`OutlinedButton`)은 `maximumSize: Size.infinite` 로 가로를 꽉
  채우려다 `BoxConstraints forces an infinite width` 로 레이아웃 abort → 본문 전체가 무너져 백지/겹침.
- **`Spacer`/`Expanded`** 도 무한 폭에서 flex 분배 불가 → 동일하게 실패.
- 반면 **`Text`/`Container`/`Center`** 등 **intrinsic 크기** 위젯은 무한에서도 생존(그래서 백지 중에도
  `Center(Text)` 만은 보였던 것).

**근본 해법(레이아웃):** 위젯 트리 레벨 래퍼(`SizedBox`/`ConstrainedBox`/`Positioned`/`LayoutBuilder`/
`UnconstrainedBox`)는 `enforce` 라 tight-∞ 를 **못 줄인다.** 유일하게 통한 건

```dart
Align(                         // ← 제약을 loosen (0..∞) 으로 풀고
  alignment: Alignment.topLeft,
  child: SizedBox(             // ← 그 안에서 화면폭으로 tight 하게 다시 고정
    width: MediaQuery.of(context).size.width,
    child: child,             // 이제 본문은 유한 폭을 받음 → Expanded/머티리얼 정상
  ),
)
```

이 `Align > SizedBox(화면폭)` 조합을 **시트 프레임(`MapBottomSheet`)이 항상 제공**한다(§15). 그래서
시트에 올리는 본문은 폭 걱정 없이 일반 위젯을 써도 된다.

## 14. 시도와 실패의 연대기 (고생 기록 — 지우지 말 것)

순서대로 실제로 겪은 길이다. 각 단계가 왜 막혔는지 남긴다.

1. **라이브 맵 위 라우트/`showModalBottomSheet` 로 상세** → 본문 **백지**. (원인 오판: PlatformView
   합성 충돌이라 의심 — PART I 가설.)
2. **`DraggableScrollableSheet`(PART I §4 권장)** → **크래시.** `viewport.dart` 의 `child.geometry!`
   가 null(시트 내부 스크롤 뷰포트 지오메트리 계산 실패). 폐기.
3. **탭 내 스왑** — `MapTab.build` 가 시트가 열리면 지도 Stack 대신 **상세 Scaffold 를 반환**(지도
   위젯을 트리에서 제거). 백지는 사라졌다(PlatformView 가 없으니 width 손상 컨텍스트도 사라짐 +
   `Align>SizedBox` 적용). **그러나** 이건 PART I **§7 안티패턴("맵을 트리에서 넣다 뺐다")을 정면
   위반** → 닫을 때마다 지도 **dispose→recreate**(타일·마커 재로딩 1~2초). ← 사용자가 "닫으면 지도가
   다시 렌더링돼 거슬린다"고 지적한 바로 그 잼.
4. **스냅샷 스왑 바텀시트**(PART I §5 차용) — 닫기 잼을 줄이려 `controller.takeSnapshot()`(`Future<File>`)
   으로 지도를 정적 이미지로 얼리고 그 위에 커스텀 `MapBottomSheet`. 동작은 했지만 **여전히 지도를
   트리에서 빼는 구조**라 닫을 때 재생성은 남았고, 복잡도만 늘었다.
5. **게시글 목록도 시트로 통일**(#29) — 그전까지 동네 클러스터는 별도 전용화면(`_RegionPostsScreen`)
   풀스크린 스왑이었는데, 시설 상세와 같은 `showSheetOverMap` 으로 통일. 카드 대신 컴팩트 행 + 56px
   썸네일.
6. **최종 — 라이브 지도 위 직접 오버레이**(#29 perf 커밋) — "PlatformView 위엔 Flutter 못 얹는다"는
   대전제를 실기기로 **반증**(§16). 스냅샷·맵 제거를 **전부 들어냄.** `build()` 는 시트가 열려도 **같은
   `NaverMap` 을 유지**하고 그 위에 시트만 얹는다. → **닫아도 재생성/재로딩 0.** 백지(§13)는 시트의
   `Align>SizedBox` 가, 닫기 잼(§7)은 맵 유지가 각각 해결.

## 15. 최종 아키텍처 (확정)

핵심: **맵은 한 번 만들어 계속 트리에 두고, 시트는 그 위 오버레이로만 얹는다(라우트 push·스냅샷·맵
제거 전부 없음).**

### 15.1 `MapTab.build` — 맵은 시트가 열려도 유지

```dart
@override
Widget build(BuildContext context) {
  super.build(context);                       // AutomaticKeepAlive 필수 호출
  final sheetOpen = _sheetChild != null;
  return PopScope(
    canPop: !sheetOpen,                        // 시트 열려 있으면 뒤로가기로 닫기
    onPopInvokedWithResult: (didPop, _) {
      if (!didPop && sheetOpen) _closeSheetOverMap();
    },
    child: Scaffold(
      backgroundColor: Colors.white,
      body: Stack(children: [
        // (0) 바닥 맵 — 시트가 열려도 절대 트리에서 빠지지 않음(= 재생성 없음)
        ClipRect(child: NaverMap(
          options: NaverMapViewOptions(
            initialCameraPosition: _lastCamera ?? _initialPosition,
            customStyleId: _customStyleId,
            locationButtonEnable: false,
            consumeSymbolTapEvents: false,
          ),
          onMapReady: (c) { _controller = c; _initLoad(); },
          onCameraIdle: _onCameraIdle,         // _lastCamera 갱신
          onMapTapped: (_, _) => FocusManager.instance.primaryFocus?.unfocus(),
        )),
        // (1) 시트 오버레이 — 라이브 지도 위에 바로(스크림이 지도를 어둡게)
        if (sheetOpen)
          Positioned.fill(child: MapBottomSheet(
            onClose: _closeSheetOverMap,
            heightFactor: _sheetHeight,
            child: _sheetChild!,
          )),
        // (2) 지도 UI(검색/칩/로딩/내위치) — 시트가 열리면 숨김
        if (!sheetOpen) ...[ /* _SearchField, 카테고리 칩, _MyLocationButton */ ],
      ]),
    ),
  );
}
```

### 15.2 시트 열고/닫기 — 스냅샷 없음

```dart
/// 라이브 지도 위에 MapBottomSheet 로 content 를 올린다(재사용).
/// 지도는 트리에 그대로 유지 → 닫아도 재생성/재로딩 없음.
void showSheetOverMap(Widget content, {double heightFactor = 0.6}) {
  FocusManager.instance.primaryFocus?.unfocus();
  setState(() { _sheetChild = content; _sheetHeight = heightFactor; });
}

void _closeSheetOverMap() => setState(() => _sheetChild = null);

// 호출부
void _showFacilitySheet(Facility f) => showSheetOverMap(
      FacilityDetailContent(facility: f, color: _colorFor(f.category),
          label: kFacilityLabels[f.category] ?? f.category));

void _showRegionPosts(PostCluster cl) =>
    showSheetOverMap(RegionPostsContent(cluster: cl), heightFactor: 0.7);
```

### 15.3 `MapBottomSheet` — 커스텀 슬라이드 시트(DraggableScrollableSheet 대체)

`lib/widgets/map_bottom_sheet.dart`. `AnimationController`(0=숨김, 1=올라옴)로 직접 제어.

- **스크림**: `Positioned.fill` > `GestureDetector(behavior: opaque, onTap: _dismiss)` >
  `ColoredBox(black 0.4*t)`. 바깥 탭 → 닫기. (이 스크림이 라이브 지도를 어둡게 덮어 "진짜 바텀시트"
  처럼 보이게 하고, 동시에 지도로 가는 탭/제스처를 가로채 간섭을 막는다.)
- **패널**: `Positioned(left:0, width:w, height:h, bottom: viewInsets.bottom - (1-t)*h)` 로 `bottom`
  값을 애니메이션해 서랍처럼 슬라이드(`Transform` 레이어는 PlatformView 위 합성 이슈 회피차 안 씀).
- **본문 폭 고정**: 패널 Column 안에서 `Expanded > Align(topLeft) > SizedBox(width: w, child)` —
  이게 §13 의 무한너비 해법을 **모든 시트 본문에 자동 적용**한다.
- **드래그 닫기**: 손잡이 `onVerticalDragUpdate/End` 로 아래로 쓸면 닫힘.

## 16. 핵심 반전 — Flutter는 라이브 PlatformView 위에 정상 합성된다

PART I 의 대전제("PlatformView 한계 때문에 맵을 전환/오버레이에서 빼야 한다")는 **실기기에서 반증**됐다.

- **z-order 검증:** 스냅샷-스왑 단계(§14-4)에서 **불투명 스냅샷 `Image`(=Flutter)가 라이브 지도를
  완전히 덮었다.** 이는 Flutter 레이어가 PlatformView 레이어 **위**에 그려진다는 직접 증거다
  (legacy Virtual Display 였다면 네이티브가 위로 뚫고 나와 못 덮음). 로그상 백엔드는
  **`PlatformView is using SurfaceProducer backend`** + **Impeller(OpenGLES)** = TLHC 경로.
- 그래서 스냅샷을 **걷어내고 라이브 지도 위에 시트를 바로 얹어도** 시설/게시글 시트가 **정상 렌더**됐다.
- **재생성 0 검증:** 시트를 닫을 때 로그에 지도 재초기화 마커
  (`Installing custom TextureView` / `GPU Identifier` / `ParseStyle …`)가 **0건**. (지도가 실제로
  생성될 때는 `ParseStyle` 만 ~100줄 찍힌다 — 명확히 구분됨.) 즉 닫아도 **dispose/recreate 안 함.**
- 부수효과: `takeSnapshot()` 캡처 단계가 없어져 **여는 속도도 빨라짐.**

> 교훈: "PlatformView 위 합성이 깨진다"는 건 **구형 Virtual Display 시절의 통념**이다. 현행 Flutter
> (Impeller + SurfaceProducer/TLHC)에선 **Flutter 오버레이가 라이브 지도 위에 정상 합성**된다.
> 가정하지 말고 **실기기 로그/스냅샷으로 z-order 를 직접 확인**할 것.

## 17. 지도 위 시트 본문 — 위젯 작성 규칙 (갱신)

- 폭은 `MapBottomSheet` 가 `Align>SizedBox(화면폭)` 로 유한 고정해 준다 → 본문에서 **`Expanded`/
  `Spacer`/머티리얼 위젯 모두 안전**(긴 주소 줄바꿈, 우측 정렬 날짜 등). 과거 "머티리얼·Spacer·Expanded
  금지"는 §14-3(탭 내 스왑) 시절의 **보수적 잔재이며 현재는 불필요**.
- (참고) `FacilityDetailContent` 의 버튼이 아직 `GestureDetector+Container` 탭인 것은 그 잔재일 뿐,
  머티리얼 버튼으로 바꿔도 동작한다.
- 새 시트가 필요하면 **`showModalBottomSheet`/`DraggableScrollableSheet` 를 직접 쓰지 말고**
  `MapTab.showSheetOverMap(content, {heightFactor})` 를 쓸 것. (지도와 무관한 일반 화면/라우트에선
  머티리얼 시트 자유롭게 OK.)

## 20. 한 줄 요약 (갱신본 — §11 대체용)

> 백지의 진짜 원인은 합성이 아니라 **본문 무한 너비**였고, 시트가 `Align>SizedBox` 로 폭을 유한
> 고정하면 끝난다. 그리고 현행 Flutter(Impeller/SurfaceProducer)에선 **Flutter 시트가 라이브 지도
> 위에 정상 합성**되므로, 지도 위 상세는 **맵을 빼지도 얼리지도 말고**(빼면 닫을 때 재생성 잼) 라이브
> 지도 위에 `showSheetOverMap` 으로 바로 얹는다. One Map·Keep Alive·인플레이스 시트(§3)는 유지,
> 스냅샷 스왑·DraggableScrollableSheet·맵 제거(§4·§5)는 버린다.
