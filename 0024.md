# 약관·동의(Consent) 설계 — 온보딩 게이트 · 버전관리 · 기록

> 상태: 설계(미구현). 작성 2026-07-02. 관련: 이용약관/개인정보/위치정보 컴플라이언스.
> **주의**: 본 문서와 첨부 초안은 법률자문이 아니다. 이 앱에 필요한 행정 절차는 **위치정보(위치기반서비스)사업 신고 1건**뿐이며(통신판매업 신고 불요 — 아래 참조), 제출 전 전문가 확인 권장. 초안은 표준 양식 기반 뼈대이며, 사업자 정보(§7) 확정 후 검토 필요.
> **통신판매 비해당 전제**: 본 서비스는 시설 정보 제공과 이용자↔시설 **연결**만 하고 결제·주문이 앱 안에서 일어나지 않으므로 「전자상거래법」상 통신판매(중개)에 해당하지 않는다고 본다 → **통신판매업 신고·청약철회 등 조항 불요**. 단, 향후 앱 내 결제·주문·정산 흐름이 생기면 통신판매(중개) 해당 여부 재검토 필요.

| --- | --- | --- |

## 1. 문서 목록 & 필수/선택 분류

동의 UI에 뜨는 문서와, "게시만 하는" 문서를 구분한다. **처리방침은 동의 대상이 아니라 상시 게시 대상**이다.

| doc_type | 문서 | 온보딩 구분 | 근거 |
| --- | --- | --- | --- |
| `tos` | 서비스 이용약관 | **필수 동의** | — |
| `privacy` | 개인정보 처리방침 | **게시(notice)** · 동의기록 안 함 | 개인정보보호법 §30 |
| `collect_req` | 개인정보 수집·이용(필수항목) | **필수 동의** | 개인정보보호법 §15 |
| `collect_opt` | 개인정보 수집·이용(선택항목) | **선택 동의** | 개인정보보호법 §15 |
| `overseas` | 개인정보 국외이전 | **필수 동의** | 개인정보보호법 §28-8 |
| `lbs` | 위치기반서비스 이용약관 | **필수 동의** | 위치정보법 §18 |
| `age14` | 만 14세 이상 확인 | **필수 체크**(동의문 아님, 확인) | 개인정보보호법 §22-2 |
| `marketing` | 마케팅 정보 수신 | **선택 동의** | 정보통신망법 §50 |
| `service_notice` | 거래 비당사자(정보제공·연결) 고지 | **게시(notice)** | 서비스 책임 명확화(통신판매 비해당) |
| `third_party` | 개인정보 제3자 제공(시설 연결 시) | **이용 시점 필수 동의**(온보딩 아님) | 개인정보보호법 §17 |

> **원칙(위법 방지)**: 선택 동의를 필수처럼 강제 금지. 필수 동의는 "서비스 제공에 반드시 필요한 것"만. 선택 미동의로 가입 거부 금지.
> **제3자 제공**은 온보딩에서 미리 받지 말고 **실제 시설 예약·연락 연결이 일어나는 시점**에 해당 항목·수령자를 특정해 받는다(포괄동의 지양).
>
> **국외이전 — 동의 vs 게시(갈음) 선택**: 개인정보보호법 §28-8은 국외이전에 동의를 원칙으로 하되, **계약 이행에 필요한 처리위탁·보관형 이전이고 그 사실을 처리방침에 공개(또는 통지)한 경우 동의 없이 가능**하다(§28-8①3). 즉 Supabase 같은 순수 인프라 처리위탁은 (a) `overseas`를 **필수 동의**로 받거나, (b) **처리방침 게시로 갈음**하는 두 방법이 있다. 갈음(b)이 온보딩 마찰이 적고 실무상 흔하다. 리전이 국내면 애초에 국외이전 아님. → 리전·법무 확인 후 §1의 `overseas` 행을 `required` 또는 `notice` 중 하나로 확정.

## 2. DB 설계

### 2.1 약관 원문·버전 (`app.terms`)

```sql
-- 문서 원문과 버전. 앱은 이 테이블에서 현재 버전을 읽어 표시.
create table app.terms (
  id           uuid primary key default gen_random_uuid(),
  doc_type     text not null,          -- 'tos','privacy','collect_req',...(§1)
  version      integer not null,       -- 문서별 1,2,3...
  title        text not null,
  body         text not null,          -- 전문(markdown)
  summary      text,                   -- 온보딩 카드용 요약(선택)
  consent_kind text not null           -- 'required' | 'optional' | 'notice'
                 check (consent_kind in ('required','optional','notice')),
  effective_at timestamptz not null,   -- 시행일(미래 예약 가능)
  created_at   timestamptz not null default now(),
  unique (doc_type, version)
);
create index terms_doctype_idx on app.terms(doc_type, effective_at desc);

alter table app.terms enable row level security;
-- 읽기: 로그인 전에도 약관 조회가 필요하므로 anon/authenticated SELECT 허용(원문은 비민감).
grant select on app.terms to anon, authenticated;
create policy terms_read on app.terms for select
  using (effective_at <= now());       -- 시행된 버전만 공개
-- 쓰기: service_role(관리자 엣지)만. 정책 미부여 = 차단.
```

"현재 유효 버전" = `doc_type`별 `effective_at <= now()` 중 `version` 최대. 뷰로 고정:

```sql
create view app.terms_current as
select distinct on (doc_type) *
from app.terms
where effective_at <= now()
order by doc_type, version desc;
grant select on app.terms_current to anon, authenticated;
```

### 2.2 동의 기록 (`app.user_agreements`) — append-only 감사 로그

```sql
-- 누가 · 어떤 문서의 어떤 버전에 · 언제 동의/거부했는지. 증빙 의무 대응.
-- 선택항목 '거부'도 기록(agreed=false). 마케팅 철회도 새 행으로 append.
create table app.user_agreements (
  id         uuid primary key default gen_random_uuid(),
  user_id    uuid not null references public.users(id) on delete cascade,
  doc_type   text not null,
  version    integer not null,
  agreed     boolean not null,        -- 동의 true / 거부·철회 false
  agreed_at  timestamptz not null default now(),
  ip         inet,                    -- 증빙(선택)
  user_agent text
);
create index ua_user_idx on app.user_agreements(user_id, doc_type, agreed_at desc);

alter table app.user_agreements enable row level security;
-- 정책 0개 + anon/authenticated GRANT 없음 → service_role(엣지)만 기록/조회.
```

> append-only 로그를 쓰는 이유: 동의 이력(언제 어떤 버전에 동의)이 그대로 증빙이 된다. "현재 상태"는 `user_id+doc_type`의 **최신 행**으로 판정.

## 3. 엣지/RPC 흐름

### 3.1 회원가입 시 동의 기록

기존 `register`(또는 `login` 최초) 엣지 함수에 동의 payload를 함께 받아 **서버측에서** 기록. `user_id`는 서버가 방금 만든 계정 id를 사용(클라 신뢰 금지).

```
POST /register { email, password, agreements: [{doc_type, version, agreed}], age14: true }
1. age14 != true            → 400 (14세 미만 차단)
2. 필수(required) doc_type 전부 agreed=true 인지 서버가 재검증(누락/false → 400)
   - 서버는 app.terms_current 로 "현재 필수 목록"을 만들어 대조(클라가 보낸 목록 신뢰 금지)
3. 계정 생성 → 각 agreements 행 INSERT(app.user_agreements), version 은 서버가 terms_current 로 확정
4. 응답: 토큰(+ pending 없음)
```

### 3.2 재동의 필요 판정 (`app.pending_terms`)

로그인/앱 실행 시 호출. **현재 필수 버전에 최신 동의가 없는** 문서를 반환.

```sql
create or replace function app.pending_terms()
returns table(doc_type text, version int, title text, consent_kind text)
language sql stable security definer set search_path to '' as $$
  with cur as (select * from app.terms_current where consent_kind = 'required'),
  latest as (
    select distinct on (doc_type) doc_type, version, agreed
    from app.user_agreements
    where user_id = app.uid()
    order by doc_type, agreed_at desc
  )
  select c.doc_type, c.version, c.title, c.consent_kind
  from cur c
  left join latest l on l.doc_type = c.doc_type
  where l.doc_type is null            -- 한 번도 동의 안 함
     or l.agreed = false              -- 과거 거부
     or l.version < c.version         -- 개정됨 → 재동의 필요
$$;
```

- 결과가 비어있지 않으면 앱은 **재동의 모달**을 띄우고, 완료 전 주요 기능 차단.
- 선택(optional)/게시(notice)는 여기서 제외 → 가입/이용을 막지 않음.

### 3.3 동의 갱신·철회 (`app.record_agreement`)

```
POST record_agreement { doc_type, version, agreed }
- user_id = app.uid() (서버 확정)
- app.user_agreements 에 새 행 append (마케팅 on/off, 재동의 모두 이 경로)
- 마케팅 철회 = { 'marketing', 현재버전, false }
```

### 4.1 회원가입 화면 동의 블록

```
[ ] 전체 동의합니다  (선택 포함 일괄 — 단, 체크해도 각 항목 상태는 개별 저장)
────────────────────────────
[ ] (필수) 만 14세 이상입니다
[ ] (필수) 서비스 이용약관 동의                [보기 >]
[ ] (필수) 개인정보 수집·이용 동의             [보기 >]
[ ] (필수) 개인정보 국외이전 동의             [보기 >]
[ ] (필수) 위치기반서비스 이용약관 동의        [보기 >]
[ ] (선택) 개인정보 수집·이용(선택항목) 동의   [보기 >]
[ ] (선택) 마케팅 정보 수신 동의              [보기 >]
────────────────────────────
개인정보 처리방침 [전문 보기]   (동의 대상 아님 · 상시 게시)
거래 비당사자 고지 [보기]        (게시)

[동의하고 가입]  ← 필수 항목 전부 체크 전까지 비활성
```

- **[전체 동의]**는 편의 기능. 체크 시 필수+선택 모두 체크되지만, 서버엔 **항목별 agreed 값**으로 전송(선택 거부도 false로 기록).
- 각 `[보기 >]`는 `app.terms_current`(또는 `terms` by version)에서 `body`를 로드해 마크다운/웹뷰로 표시. 표시한 **version을 그대로 기록**.
- 필수 미체크 시 가입 버튼 비활성 — 선택 미체크로는 절대 막지 않음.

### 4.2 재동의(개정) 팝업

- 앱 실행/로그인 직후 `pending_terms()` 호출 → 결과 있으면 모달.
- 변경된 필수 약관만 표시, "무엇이 바뀌었는지" 요약(선택) 노출 후 재동의.
- 재동의 완료까지 핵심 기능(글쓰기·예약연결 등) 차단, 단순 탐색은 허용 여부 정책 선택.

### 4.3 설정 화면

- 마케팅 수신 on/off 토글 → `record_agreement('marketing', v, on)`.
- 약관/처리방침 전문 상시 열람 메뉴.
- 회원탈퇴 → 계정 삭제(§5).

## 5. 개정·재동의·파기 운영

- **개정 절차**: 새 `terms` 행 INSERT(version+1, 미래 `effective_at`) → **사전 고지**. 이용자에게 불리하거나 중대한 변경은 **시행 30일 전** 고지·개별통지, 그 외 최소 7일 전 고지(문서별 명시).
- **재동의 트리거**: 필수 문서 version 상승 시 `pending_terms()`가 자동 감지 → 다음 접속 시 모달.
- **처리방침 개정**은 재동의가 아니라 **게시·고지**로 충분(동의 대상 아님). 단 수집 항목·목적이 실질 변경되면 해당 `collect_*` 동의는 재동의 대상.
- **탈퇴/파기**: `on delete cascade`로 동의기록·토큰 제거. 단 법령상 보존 의무가 있는 정보(있다면)는 분리 보관 후 기간 만료 시 파기 — 처리방침의 보유기간과 일치시킬 것.
