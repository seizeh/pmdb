# 0018 — 게시글 사진 실존 검증 (카메라 강제 + AI 반려동물 판별 + 촬영 위치 일치)

> PawMate(`pmdart`) + Supabase 백엔드(`pmdb`) 기준.
> **선행 의존: 0017(지역 인증)** — `users.region_code`, `verify-location` Edge Function, `location_service.dart` 가 이미 구현되어 있다고 전제한다.
> AI 공급자: **Google Gemini 2.5 Pro** (`gemini-2.5-pro`, generateContent + 구조화 출력).
> 역지오코딩 공급자: **Naver Reverse Geocoding** (0017 §7 재사용).

---

## 0. 목표와 결정 요약

- 사진 첨부는 **갤러리 금지**. "사진 등록" 버튼은 **기기 카메라를 즉시 실행**해 방금 찍은 1장만 받는다.
- 촬영 직후 **GPS 좌표**를 함께 잡고, **촬영 위치의 행정동 == 내 활동지역(`users.region_code`)** 일 때만 통과시킨다.
- 찍은 사진은 **서버(Edge Function)** 가 **Gemini 2.5 Pro** 로 판별한다. **실제 살아있는 개/고양이**가 찍혀야 통과, 화면 재촬영·인쇄물·일러스트·인형·AI 생성 이미지는 거절.
- 검증은 **게시글 작성 전(前)단계**다. 통과 시 서버가 **검증 토큰**을 발급하고, 게시글 INSERT 트리거가 그 토큰을 검사한다 → **클라이언트가 검증을 우회할 수 없다.**
- AI 결과는 클래스별 신뢰도 컬럼 **`ai_dog_real / ai_cat_real / ai_dog_fake / ai_cat_fake`** (0~1)로 로그 테이블에 저장한다.
- **AI 키(Gemini)·이미지 원본 바이트는 서버만 본다.** 통과한 바이트를 서버가 직접 Storage 에 올려 "AI가 본 사진 == 게시된 사진" 을 보장한다.

## 1. 전체 플로우 (end-to-end)

```
[pmdart] 게시글 작성 → "사진 등록" 탭
   │  ① image_picker(source: camera) → 카메라 즉시 실행, 갤러리 진입 불가
   │  ② 촬영 완료 즉시 geolocator 로 좌표/accuracy/isMocked 획득 (0017 location_service 재사용)
   │  ③ 사진 bytes(base64) + 좌표를 verify-post-photo 함수로 전송
   ▼
[Edge Function: verify-post-photo]  (커스텀 JWT 수동 검증 → sub = uid)
   ① 활동지역 인증 상태 확인: is_location_verified=true & 만료 안 됨 → 아니면 fail(not_verified)
   ② 모의위치 거절: isMocked=true → fail(mock_location)
   ③ Naver 역지오코딩(coords) → 촬영지 region_code
        region_code != users.region_code → fail(region_mismatch)
   ④ Gemini 2.5 Pro 판별 → {species, dog_real, cat_real, dog_fake, cat_fake, reason}
        real 개/고양이 아님 → fail(not_real_pet)
   ⑤ 통과: service_role 로 media 버킷에 원본 업로드 + photo_verifications 로그(토큰 발급)
   ▼ { pass:true, token, imageUrl, species }
[pmdart] createPost(imageUrl, verificationToken, …)
   ▼
[posts INSERT 트리거 tg_posts_check_write]
   image_url 이 있으면 → 유효한 미사용 photo_verifications 토큰 필수
   (본인 + 토큰 image_url 일치 + region_matched + ai_pass + 미만료) → 통과 시 토큰 소진
   ▼  게시글 등록 완료
```

핵심: **카메라 강제**는 1차 방어(UX), **서버 토큰 검사**가 2차 방어(위변조 차단)다. 클라이언트가 갤러리 사진을 억지로 끼워 넣어도, 서버 검증을 통과한 토큰이 없으면 INSERT 가 막힌다.

## 2. 스키마 보강 (이번에 추가)

### 2-1. 검증 로그 + 토큰 테이블 `photo_verifications`

0017 의 `location_verifications` 와 같은 결을 따른다. AI 신뢰도 4개 컬럼을 여기 둔다.

```sql
-- migration: 2026XXXX_photo_verifications.sql
create table public.photo_verifications (
  id              uuid primary key default gen_random_uuid(),
  user_id         uuid not null references public.users(id),

  -- 촬영 위치 검증
  shot_lat        decimal(10,7),
  shot_lng        decimal(10,7),
  shot_accuracy_m smallint,
  region_code     varchar(20),                 -- 촬영지 행정동코드(Naver admcode)
  region_matched  boolean not null default false,

  -- AI 판별 결과(클래스별 신뢰도 0~1)
  ai_species      varchar(10),                 -- 'dog' | 'cat' | 'other' | 'none'
  ai_dog_real     numeric(4,3) not null default 0,
  ai_cat_real     numeric(4,3) not null default 0,
  ai_dog_fake     numeric(4,3) not null default 0,
  ai_cat_fake     numeric(4,3) not null default 0,
  ai_pass         boolean not null default false,
  ai_reason       varchar(200),

  -- 검증 토큰(게시글 INSERT 가 소진)
  image_url       text,                        -- 통과 시 서버가 업로드한 공개 URL
  image_path      text,                        -- media 버킷 내 경로(<uid>/posts/…)
  result          varchar(10) not null
                    check (result in ('pass','fail')),
  fail_reason     varchar(40),
  consumed_at     timestamptz,                 -- 게시글에 사용되면 채워짐(재사용 차단)
  expires_at      timestamptz not null,        -- 발급 후 N분 유효
  created_at      timestamptz not null default now()
);

create index photo_verifications_user_idx
  on public.photo_verifications (user_id, created_at desc);

-- 미사용·미만료 토큰 빠른 조회
create index photo_verifications_token_open_idx
  on public.photo_verifications (id)
  where consumed_at is null and result = 'pass';

comment on table public.photo_verifications is
  '게시글 사진 실존 검증(촬영 위치 일치 + AI 반려동물 판별) 로그 및 1회용 토큰';
```

> `numeric(4,3)` = 0.000~1.000 범위(점수). 게시글당 정확히 1개의 통과 토큰이 소진된다.

### 2-2. `posts` 에 검증 결과 요약 컬럼

게시글에는 압축된 결과만 비정규화로 남긴다(피드/관리자용).


### 2-3. 쓰기 권한 (0017 의 컬럼 GRANT 원칙 유지)

`photo_verifications` 와 `posts` 의 검증 컬럼은 **클라이언트가 직접 쓰지 못한다.** 0017 §2 에서 `users` UPDATE 를 안전 컬럼만 재부여했던 것과 동일하게, 여기서도 트리거/RPC(SECURITY DEFINER, service_role) 경로로만 채운다.

```sql
-- photo_verifications: 클라이언트는 service_role/definer 경로로만 접근.
-- ※ 0005.md 의 'grant ... on all tables' 는 그 시점 존재 테이블에만 적용되고,
--   default privileges 설정이 없으므로(전수 확인) 이 신규 테이블은 authenticated 가
--   기본적으로 아무 권한도 없다. 따라서 INSERT/UPDATE/DELETE 차단은 별도 revoke 불필요.
alter table public.photo_verifications enable row level security;

-- (선택) 클라이언트가 검증 이력을 직접 조회할 일이 있으면, RLS 정책 + 테이블 SELECT GRANT
--   를 둘 다 줘야 한다(GRANT 없으면 RLS 정책만으로는 조회 불가). 기본은 미부여(서버만 접근).
-- grant select (id, result, fail_reason, ai_species, region_matched, created_at)
--   on public.photo_verifications to authenticated;
-- create policy pv_select_own on public.photo_verifications
--   for select using (user_id = app.uid() or app.is_admin());

-- posts: table-wide INSERT 가 authenticated 에 부여돼 있다(0005.md L32, 전수 확인).
--   → 회수 후 안전 컬럼만 재부여해 검증 결과 컬럼을 클라가 못 쓰게 한다.
revoke insert on public.posts from authenticated;
grant insert (
  user_id, category, title, content, scheduled_at,
  image_url, image_mime_type, image_file_size, image_thumbnail_url,
  image_width, image_height
) on public.posts to authenticated;
-- photo_verification_id / is_pet_verified / ai_pet_species 는 미부여
-- → 트리거(definer)가 토큰 검증 후 서버측에서 채운다.
```

> **배포 전 확인:** `posts` table-wide INSERT GRANT(0005.md L32 `grant insert, update, delete on all tables ... to authenticated`)는 확인됨. 위 컬럼 화이트리스트가 클라이언트 `createPost` 가 실제로 보내는 컬럼(user_id, category, title, content, scheduled_at, image_*)을 모두 포함하는지 대조한다. `actual_lat/lng`, `display_*`, `region_code` 등 위치 컬럼은 별도 정책(0017/후속)에서 다룬다. **참고:** §8 의 `create_post_verified`(SECURITY DEFINER, 테이블 소유자=postgres) 경로는 RLS·컬럼 GRANT 를 우회하므로, 이 화이트리스트는 "클라가 `posts` 에 직접 INSERT 하는 경로"를 막는 방어선이다(두 경로 중 RPC 로 일원화 권장).

## 3. 카메라 강제 (pmdart) — 갤러리 금지

현재 `storage_service.pickImage()` 는 `ImageSource.gallery` 다. 게시글 사진은 **카메라 전용 경로**로 분리한다.

```dart
// services/storage_service.dart — 게시글 검증 전용 촬영 메서드 추가
/// 게시글용: 카메라만. 갤러리 진입 불가, 방금 찍은 1장만 반환.
Future<XFile?> capturePostPhoto() => _picker.pickImage(
      source: ImageSource.camera,          // ← 갤러리 아님. OS 카메라 즉시 실행
      preferredCameraDevice: CameraDevice.rear,
      maxWidth: 1600,
      imageQuality: 85,
      requestFullMetadata: false,          // EXIF 위치는 신뢰하지 않음(위치는 geolocator로 별도 취득)
    );
```

UX 규칙:

- 게시글 작성 화면의 사진 영역은 **카메라 아이콘 1개만** 노출. "앨범에서 선택" 옵션 제거.
- `capturePostPhoto()` 가 `null`(사용자 취소)이면 등록 진행 불가.
- 촬영 성공 → **곧바로** `LocationService.current()` (0017) 로 좌표 취득 → 검증 함수 호출. 촬영과 좌표 취득 사이 시간차를 최소화(다른 화면 이동 차단).

> 한계: `ImageSource.camera` 는 OS 카메라를 직접 띄우므로 "갤러리에서 가져오기"는 구조적으로 막힌다. 다만 루팅/모의 카메라 앱은 우회 가능 → **그래서 서버 토큰(§5,§7)이 최종 방어선**이다. 클라이언트만으로는 완전 차단 불가임을 전제로 설계한다.

플랫폼 권한 추가:

- Android `AndroidManifest.xml`: `<uses-permission android:name="android.permission.CAMERA"/>`
- iOS `Info.plist`: `NSCameraUsageDescription` (예: "반려동물 사진을 촬영해 게시글 실존 인증에 사용합니다")

## 4. AI 판별 — Gemini 2.5 Pro

### 4-1. 모델 / 호출

- 모델: **`gemini-2.5-pro`** (멀티모달, 이미지+텍스트 입력).
- 엔드포인트: `POST https://generativelanguage.googleapis.com/v1beta/models/gemini-2.5-pro:generateContent`
- 인증: 헤더 `x-goog-api-key: {GEMINI_API_KEY}` (또는 `?key=` 쿼리).
- 이미지: `inline_data` 에 base64(촬영본, 1600px/85% ≈ 수백 KB → 인라인 한도 20MB 내 충분).
- **구조화 출력**: `generationConfig.responseMimeType = "application/json"` + `responseSchema` 로 JSON 강제. 자유 텍스트 파싱 대신 스키마 고정.

### 4-2. 프롬프트 의도

> "이 사진에 **실제로 살아있는 개 또는 고양이가 물리적으로 존재**하는지 판별하라. 화면 재촬영(스크린/모니터), 인쇄물, 일러스트/만화, 봉제인형/장난감, AI 생성 이미지는 **fake** 로 분류한다."

응답 스키마:

```jsonc
{
  "species": "dog | cat | other | none",
  "dog_real": 0.0,   // 살아있는 실제 개가 찍혔을 신뢰도
  "cat_real": 0.0,   // 살아있는 실제 고양이
  "dog_fake": 0.0,   // 개지만 화면/인쇄/일러스트/인형/AI 등
  "cat_fake": 0.0,   // 고양이지만 가짜
  "reason": "짧은 근거(한국어, 80자 이내)"
}
```

### 4-3. 통과 판정 (서버에서 계산)

```
PASS  ⇔  max(dog_real, cat_real) >= AI_PASS_THRESHOLD          (기본 0.70)
        AND max(dog_real, cat_real) > max(dog_fake, cat_fake)  (real 이 fake 보다 우세)
species = dog_real >= cat_real ? 'dog' : 'cat'
```

> `AI_PASS_THRESHOLD` 는 운영 중 튜닝 대상(0.6~0.8). 초기 0.70 권장. 임계 미만/동률/other/none → fail(not_real_pet).

## 5. Edge Function `verify-post-photo` 스펙 (pmdb)

위치: `supabase/functions/verify-post-photo/index.ts` (Deno). `_shared/cors.ts` 사용, **`--no-verify-jwt` 배포 + 함수 내부에서 로그인 JWT 를 `JWT_SECRET` 으로 수동 검증**(0017/login 과 동일 패턴). `main.dart` 의 `accessToken` 콜백이 토큰을 자동 첨부.

요청:

```jsonc
POST /functions/v1/verify-post-photo
Authorization: Bearer <login JWT>          // sub = uid
{
  "imageBase64": "<jpeg base64>",
  "mimeType": "image/jpeg",
  "lat": 37.2008,
  "lng": 127.0726,
  "accuracy": 12.4,
  "isMocked": false
}
```

응답(통과):

```jsonc
{
  "pass": true,
  "token": "1f0c…",                  // photo_verifications.id
  "imageUrl": "https://…/media/<uid>/posts/….jpg",
  "species": "dog",
  "expiresAt": "2026-…"
}
```

응답(실패):

```jsonc
{ "pass": false, "reason": "not_verified" }     // 활동지역 미인증/만료
{ "pass": false, "reason": "mock_location" }
{ "pass": false, "reason": "region_mismatch", "expected": "동탄2동", "got": "반월동" }
{ "pass": false, "reason": "not_real_pet", "ai": { "species":"other", "reason":"…" } }
```

핵심 로직(스켈레톤):

```ts
import { corsHeaders, json } from "../_shared/cors.ts";

const SUPABASE_URL = Deno.env.get("SUPABASE_URL")!;
const SERVICE_ROLE = Deno.env.get("SUPABASE_SERVICE_ROLE_KEY")!;
const NAVER_KEY_ID = Deno.env.get("NAVER_MAP_KEY_ID")!;
const NAVER_KEY    = Deno.env.get("NAVER_MAP_KEY")!;
const GEMINI_KEY   = Deno.env.get("GEMINI_API_KEY")!;

const AI_PASS_THRESHOLD = 0.70;
const TOKEN_TTL_MIN     = 15;     // 토큰 유효시간
const REVERIFY_DAYS     = 60;     // 활동지역 인증 만료(0017 정책과 일치시킬 것)

Deno.serve(async (req) => {
  if (req.method === "OPTIONS") return new Response("ok", { headers: corsHeaders });

  const uid = await getUidFromJwt(req);                 // 실패 시 401 (0017과 동일 구현)
  const { imageBase64, mimeType = "image/jpeg",
          lat, lng, accuracy = 0, isMocked = false } = await req.json();
  const admin = serviceClient(SUPABASE_URL, SERVICE_ROLE);

  // 1) 활동지역 인증 상태
  const { data: u } = await admin.from("users")
    .select("region_code, is_location_verified, last_verified_at")
    .eq("id", uid).single();
  const expired = !u?.last_verified_at ||
    (Date.now() - new Date(u.last_verified_at).getTime()) > REVERIFY_DAYS*864e5;
  if (!u?.is_location_verified || expired || !u?.region_code) {
    return json({ pass: false, reason: "not_verified" });
  }

  // 2) 모의위치
  if (isMocked) return await failAndLog(admin, uid, lat, lng, accuracy, "mock_location");

  // 3) 촬영지 역지오코딩 → 행정동 일치
  const geo = await reverseGeocode(lng, lat);           // 0017 §7 재사용
  if (!geo) return await failAndLog(admin, uid, lat, lng, accuracy, "geocode_failed");
  if (geo.regionCode !== u.region_code) {
    await failAndLog(admin, uid, lat, lng, accuracy, "region_mismatch", geo.regionCode);
    return json({ pass: false, reason: "region_mismatch",
                  expected: /* users 동네명 */ undefined, got: geo.regionName });
  }

  // 4) Gemini 판별
  const ai = await classifyPet(imageBase64, mimeType);  // §6
  const real = Math.max(ai.dog_real, ai.cat_real);
  const fake = Math.max(ai.dog_fake, ai.cat_fake);
  const aiPass = real >= AI_PASS_THRESHOLD && real > fake;
  if (!aiPass) {
    await admin.rpc("record_photo_verification", {
      p_user: uid, p_lat: lat, p_lng: lng, p_accuracy: Math.round(accuracy),
      p_region_code: geo.regionCode, p_region_matched: true,
      p_species: ai.species, p_dog_real: ai.dog_real, p_cat_real: ai.cat_real,
      p_dog_fake: ai.dog_fake, p_cat_fake: ai.cat_fake, p_ai_pass: false,
      p_ai_reason: ai.reason, p_result: "fail", p_fail_reason: "not_real_pet",
      p_image_url: null, p_image_path: null, p_ttl_min: TOKEN_TTL_MIN,
    });
    return json({ pass: false, reason: "not_real_pet", ai });
  }

  // 5) 통과: 서버가 직접 업로드(AI가 본 바이트 == 게시 바이트 보장)
  const path = `${uid}/posts/${Date.now()}.jpg`;
  const bytes = Uint8Array.from(atob(imageBase64), c => c.charCodeAt(0));
  await admin.storage.from("media").uploadBinary(path, bytes,
        { contentType: mimeType, upsert: false });
  const imageUrl = admin.storage.from("media").getPublicUrl(path).data.publicUrl;
  const species  = ai.dog_real >= ai.cat_real ? "dog" : "cat";

  const { data: tok } = await admin.rpc("record_photo_verification", {
    p_user: uid, p_lat: lat, p_lng: lng, p_accuracy: Math.round(accuracy),
    p_region_code: geo.regionCode, p_region_matched: true,
    p_species: species, p_dog_real: ai.dog_real, p_cat_real: ai.cat_real,
    p_dog_fake: ai.dog_fake, p_cat_fake: ai.cat_fake, p_ai_pass: true,
    p_ai_reason: ai.reason, p_result: "pass", p_fail_reason: null,
    p_image_url: imageUrl, p_image_path: path, p_ttl_min: TOKEN_TTL_MIN,
  });   // returns photo_verifications.id

  return json({ pass: true, token: tok, imageUrl, species,
                expiresAt: /* now + TTL */ undefined });
});
```

필요한 시크릿(Edge Function Secrets):

- `GEMINI_API_KEY` — Gemini 2.5 Pro 호출 (**신규**)
- `NAVER_MAP_KEY_ID`, `NAVER_MAP_KEY` — 역지오코딩 (0017 과 공유)
- `JWT_SECRET` — 로그인 JWT 검증 (공유)
- `SUPABASE_SERVICE_ROLE_KEY`, `SUPABASE_URL` — 권한 작업/업로드 (공유)

## 6. Gemini 호출 상세 (`classifyPet`)

```ts
const SCHEMA = {
  type: "object",
  properties: {
    species:   { type: "string", enum: ["dog","cat","other","none"] },
    dog_real:  { type: "number" }, cat_real: { type: "number" },
    dog_fake:  { type: "number" }, cat_fake: { type: "number" },
    reason:    { type: "string" },
  },
  required: ["species","dog_real","cat_real","dog_fake","cat_fake","reason"],
};

const PROMPT = `이 사진에 실제로 살아있는 개 또는 고양이가 물리적으로 존재하는지 판별해라.
- 화면 재촬영(모니터/휴대폰 화면), 인쇄물/사진의 사진, 일러스트·만화, 봉제인형·장난감,
  AI 생성 이미지는 fake 로 본다.
- 각 클래스 신뢰도(dog_real, cat_real, dog_fake, cat_fake)를 0~1로 매겨라.
- reason 은 한국어 80자 이내.`;

async function classifyPet(imageBase64: string, mimeType: string) {
  const url =
    "https://generativelanguage.googleapis.com/v1beta/models/gemini-2.5-pro:generateContent";
  const res = await fetch(url, {
    method: "POST",
    headers: { "content-type": "application/json", "x-goog-api-key": GEMINI_KEY },
    body: JSON.stringify({
      contents: [{
        role: "user",
        parts: [
          { text: PROMPT },
          { inline_data: { mime_type: mimeType, data: imageBase64 } },
        ],
      }],
      generationConfig: {
        responseMimeType: "application/json",
        responseSchema: SCHEMA,
        temperature: 0,
      },
    }),
  });
  if (!res.ok) throw new Error(`gemini ${res.status}`);
  const body = await res.json();
  const txt = body.candidates?.[0]?.content?.parts?.[0]?.text ?? "{}";
  const v = JSON.parse(txt);
  return {
    species: String(v.species ?? "none"),
    dog_real: clamp01(v.dog_real), cat_real: clamp01(v.cat_real),
    dog_fake: clamp01(v.dog_fake), cat_fake: clamp01(v.cat_fake),
    reason: String(v.reason ?? "").slice(0, 200),
  };
}
const clamp01 = (n: unknown) => Math.min(1, Math.max(0, Number(n) || 0));
```

> Gemini 호출 실패/타임아웃은 `not_real_pet` 가 아니라 **별도 reason(`ai_unavailable`)** 으로 처리해 사용자에게 "잠시 후 재시도" 안내(통과로 오인 금지). 비용/지연을 위해 이미지를 1024px 정도로 더 줄여 보내도 무방.

### 7-2. 게시글 INSERT 트리거에 토큰 검사 추가

0008 의 `app.tg_posts_check_write()` 에 **사진이 있으면 통과 토큰 필수** 규칙을 추가한다. 트리거가 토큰을 검증·소진하고 검증 컬럼을 채운다(클라이언트는 못 쓰는 컬럼들).

```sql
-- migration: 2026XXXX_posts_require_photo_token.sql
-- createPost 가 검증 토큰을 넘기는 경로: posts.photo_verification_id 로 받되,
-- 이 컬럼은 클라 INSERT GRANT 에서 제외(§2-3). 따라서 클라는 별도 통로로 토큰을 전달한다.
-- 방법: 임시 세션 변수 set_config('app.photo_token', <token>, true) 를 같은 트랜잭션에서 설정.
-- (Supabase RPC 'create_post_verified' 로 감싸 토큰+게시 데이터를 함께 받는 것을 권장 — §8)

create or replace function app.tg_posts_check_write()
returns trigger language plpgsql security definer set search_path = ''
as $$
declare
  v_user_type text;
  v_token uuid := nullif(current_setting('app.photo_token', true), '')::uuid;
  v_pv    public.photo_verifications%rowtype;
begin
  -- … 기존 user_type / 카테고리 / 펫 / 일정 검증 (0008 내용 유지) …

  -- [신규] 사진이 첨부되면 서버 검증 토큰을 강제
  if new.image_url is not null then
    if v_token is null then
      raise exception 'posts: 사진 게시글은 사진 실존 검증이 필요합니다';
    end if;
    select * into v_pv from public.photo_verifications
      where id = v_token
        and user_id = new.user_id
        and result = 'pass'
        and ai_pass = true
        and region_matched = true
        and consumed_at is null
        and expires_at > now()
        and image_url = new.image_url;        -- AI가 본 사진 == 게시 사진
    if not found then
      raise exception 'posts: 유효하지 않거나 만료된 사진 검증입니다';
    end if;

    -- 토큰 소진 + 검증 결과 비정규화(클라가 못 쓰는 컬럼을 definer 가 채움)
    update public.photo_verifications set consumed_at = now() where id = v_pv.id;
    new.photo_verification_id := v_pv.id;
    new.ai_pet_species        := v_pv.ai_species;
    new.is_pet_verified       := true;
  end if;

  return new;
end;
$$;
```

> **사진 필수 정책(선택).** "모든 게시글에 사진+검증 강제"라면 `if new.image_url is null then raise exception ...` 한 줄로 강화. 현재 앱은 사진이 선택이므로, 우선 "사진이 있으면 반드시 검증"으로 둔다. 카테고리별(예: 산책/돌봄만 필수)로 좁힐 수도 있다.

## 8. 게시글 작성 경로 — `create_post_verified` RPC (권장)

토큰을 트랜잭션 내 세션 변수로 안전하게 넘기기 위해, 클라이언트의 `posts` 직접 INSERT 대신 **RPC 한 방**으로 감싼다. 토큰 전달·INSERT·소진이 한 트랜잭션에서 원자적으로 일어난다.

```sql
-- migration: 2026XXXX_create_post_verified.sql
create or replace function public.create_post_verified(
  p_category varchar, p_title varchar, p_content text,
  p_scheduled_at timestamptz, p_pet_ids uuid[],
  p_image_url text, p_image_mime varchar, p_image_size int,
  p_photo_token uuid default null
) returns uuid
language plpgsql security definer set search_path = ''
as $$
declare v_uid uuid := app.uid(); v_post uuid;
begin
  if v_uid is null then raise exception 'posts: 로그인이 필요합니다'; end if;
  -- 트리거가 읽을 토큰을 트랜잭션 로컬로 설정
  perform set_config('app.photo_token', coalesce(p_photo_token::text,''), true);

  insert into public.posts (user_id, category, title, content, scheduled_at,
                            image_url, image_mime_type, image_file_size)
  values (v_uid, p_category, p_title, p_content, p_scheduled_at,
          p_image_url, p_image_mime, p_image_size)
  returning id into v_post;

  if p_pet_ids is not null and array_length(p_pet_ids,1) >= 1 then
    insert into public.post_pets (post_id, pet_id)
      select v_post, unnest(p_pet_ids);
  end if;
  return v_post;
end;
$$;
grant execute on function public.create_post_verified(
  varchar,varchar,text,timestamptz,uuid[],text,varchar,int,uuid) to authenticated;
```

> 이렇게 하면 `posts` INSERT 자체를 클라이언트에 열어둘 필요가 줄어든다(§2-3 의 INSERT GRANT 를 더 좁히거나, RPC 전용으로 전환 가능). 기존 `createPost` 직접 INSERT 경로와 둘 중 하나로 일원화할 것.

## 10. 보안 / 위변조 방어 정리

| 위협 | 방어 |
| --- | --- |
| 갤러리 옛 사진 첨부 | `ImageSource.camera` 강제(1차) + 서버가 본 바이트만 업로드(2차) |
| 클라가 검증 우회하고 바로 INSERT | `posts` INSERT 시 트리거가 통과 토큰 필수·소진. 토큰 없으면 거절 |
| 토큰 재사용/공유 | `consumed_at` 1회 소진 + `expires_at`(기본 15분) + `user_id` 일치 |
| AI가 본 사진 ≠ 게시 사진 | 트리거 `image_url = new.image_url`. 통과 URL 은 서버 업로드본뿐 |
| 다른 동네에서 촬영 | 촬영지 역지오코딩 == `users.region_code` 강제 |
| 모의 GPS | `isMocked` 거절(0017 정책 일관) |
| 화면 재촬영/AI 이미지 | Gemini fake 분류 → `dog_fake/cat_fake` 우세 시 거절 |
| 검증 컬럼 클라 위조 | `photo_verifications` 클라 INSERT/UPDATE 차단, `posts` 검증 컬럼 GRANT 미부여 → definer 트리거만 기록 |
| Gemini 키 노출 | 키는 Edge Function 시크릿. 클라이언트는 AI를 직접 호출하지 않음 |

> 완전 차단은 불가(루팅+가상 카메라). 목표는 **비용 대비 충분한 억제**: 카메라 강제 + 위치 일치 + AI fake 판별 + 1회용 서버 토큰의 4중 결합.

## 11. 엣지 케이스

- **AI 점수 동률/애매(0.6~0.7)**: 임계 미만 → fail. 사용자에게 "반려동물이 또렷이 보이게 다시 촬영" 안내.
- **개·고양이 외 동물(토끼 등)**: 현재 스키마는 dog/cat 전용 → `other` → fail. 종 확장 시 컬럼/임계 추가(후속).
- **여러 마리/사람 포함**: 한 마리라도 real 개/고양이가 우세하면 통과(프롬프트에 명시).
- **Gemini 지연/오류**: `ai_unavailable` → 통과 아님, 재시도 안내.
- **토큰 만료 후 등록**: 트리거에서 `expires_at` 검사로 거절 → 재촬영.
- **활동지역 인증 만료(0017 N일)**: `not_verified` → 동네 재인증 먼저.
- **오프라인/권한 거부(카메라·위치)**: 진입 차단 + 설정 유도.
