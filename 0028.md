## 배경 — 왜 QR 전단만으로는 안 되는가

지도의 각 매장에 QR을 배포해 후기를 모으는 단순 모델은 두 개의 구조적 공백이 있다.

1. **업체의 동기 공백** — 유저 0명인 앱의 "노출"은 사장님에게 가치가 없고, 돈·쿠폰·예약으로
   보상할 수단도 없다. 사장님이 입으로 권해야 하는데 그 이유가 비어 있다.
2. **유저의 동기 공백** — 스캔 → 설치 → 전화번호 인증 → 위치 동의 → 지역 인증 → 후기 작성은
   무거운 퍼널이다. 자발적 스캔의 가입 전환은 한 자릿수 %, 매장당 월 몇 명 수준이 현실이다.

해법의 본질은 하나다: **업체가 보호자에게 뭔가를 보내야 하는 순간을 앱이 독점한다.**
사장님에게 매 손님마다 쓸 도구를 주면, 그 도구의 수령 링크가 곧 가입 유도가 된다.

| 업종 | 도구 | 훅의 성격 |
| --- | --- | --- |
| 미용 | 전후 사진 공유 | 완료 시점마다 1회 — 개발량 최소, 파일럿 적합 |
| 위탁(호텔·유치원) | 돌봄 알림장(사진·식사·배변·특이사항) | 맡기는 기간 내내 반복 — 훅 최강(키즈노트 모델) |
| 분양 | 새 보호자 스타터 패키지(접종 알림·체크리스트) | 시점이 무기(반려 생활 최고 전환 순간), 단 **앱 밖 채널로만** |

우선순위: **미용 → 위탁 → 분양.** 알림장이 도구-시장 적합도는 최강이지만 개발량이 가장 크므로,
미용 파일럿으로 "사진 전달" 원형을 검증한 뒤 알림장으로 확장한다(같은 원형의 반복 발행형).

## 설계 원칙

1. **계정은 하나, 권한은 업종 모듈** — "미용업체냐 분양업체냐"를 업체 단위로 나누면 겸업(미용+분양
   등)에서 반드시 모순이 생긴다. 겸업·중복은 이 업계의 기본값이므로, 판단 단위를 업체가 아니라
   **기능(업종)** 으로 둔다. 업종별 등록·허가 증빙이 있으면 해당 모듈만 켠다.
2. **설치 전 가치 먼저** — 모든 훅(사진·알림장·QR)은 수신자가 **앱 설치 전에** 내용을 볼 수 있어야
   성립한다. 설치해야 열리는 링크는 "후기 써주세요"와 같은 무게다. 공유 뷰어(§3)가 공통 기반.
3. **불변식은 DB 트리거로** — 영업자 게시 차단 같은 정책 불변식은 앱 필터·Edge Function 검증이
   아니라 DB 트리거로 강제한다(직접 INSERT 우회 사례의 교훈, 자기초대 차단과 동일 원칙).
4. **정책 충돌은 채널 분리로 회피** — 분양업체는 입점 파트너가 아니라 **가입 유통 채널**이다.
   앱 안에 분양 기능을 만들지 않으므로 사업계획서 §2.5·동물보호법상 판매 알선 리스크와 충돌하지
   않는다. §2.5의 금지 대상은 "영업자의 상업적 판매 **게시**"지 영업자와의 접촉 자체가 아니다.
5. **밀도 우선 파일럿** — 지역 매칭 앱은 밀도가 생명. 시 전체가 아니라 **1개 동네 5~10개 매장**으로
   시작하고, 스캔→열람→설치→가입→후기→7일 잔존을 측정한 뒤 확장을 판단한다(§8).
6. **개인정보 최소 보존** — 미가입 보호자의 연락처를 **평문으로 저장하지 않는다**. 리포트 수신자
   대조용 전화번호는 키드 해시(HMAC)로만 저장하고 claim 성사 시 파기(§4.2). 링크 전송(문자·카톡)의
   주체는 업체이고, 앱은 토큰 링크만 발급한다. 수집 항목 추가는 pmlegal 처리방침 동시 반영.
7. **캠페인은 정식 오픈 후** — **위치기반서비스사업 신고(방통위) 수리** 이전에 오프라인 캠페인을
   시작하지 않는다. (결제·중개가 없어 통신판매업 신고 대상은 아님 — 대기할 것은 위치기반 신고.)

## 1. 업종 모듈 권한 — `business_licenses`

### 1.1 0025와의 관계

0025의 `business_profiles`는 **사업자(계정) 단위** 인증이다: 사업자번호·국세청 상태·facilities
대조·승인. 이 문서는 그 아래에 **업종 단위** 증빙 층을 추가한다. 기능 게이트의 진실의 원천은
`business_licenses` 행 존재 여부이고, `business_category`는 파생 표시값으로 강등한다(§1.4).

### 1.2 스키마

```sql
create type app.biz_license_type as enum (
  'grooming',      -- 동물미용업 (등록제)
  'boarding',      -- 동물위탁관리업 (등록제)
  'sales',         -- 동물판매업 (허가제)
  'production',    -- 동물생산업 (허가제)
  'exhibition',    -- 동물전시업 (등록제) — 당장 모듈 없음, enum 만 선점
  'transport'      -- 동물운송업 (등록제) — 〃
);

create table app.business_licenses (
  id            uuid primary key default gen_random_uuid(),
  user_id       uuid not null references public.users(id) on delete cascade,
  license_type  app.biz_license_type not null,
  license_no    varchar(40) not null,        -- 등록·허가번호(지자체 발급)
  document_path text not null,               -- 등록·허가증 사본(비공개 버킷, 0025 서류와 동일 정책)
  status        varchar(12) not null default 'pending'
                check (status in ('pending','approved','rejected')),
  reviewed_by   uuid references public.users(id),
  reviewed_at   timestamptz,
  reject_reason text,
  created_at    timestamptz not null default now(),
  unique (user_id, license_type)             -- 계정당 업종별 1행
);
```

- 신청 전제: `business_profiles` 승인 완료(사업자 인증 없이 업종 증빙만 낼 수 없음).
- 승인 흐름은 0025 관리자 승인과 같은 패턴(admin RPC + 감사 로그). facilities 공공데이터의
  업종 컬럼과 대조해 **참고 신호**로 표시하되, 자동승인은 두지 않는다(문서 위조 리스크가
  사업자번호 대조보다 크고, 건수가 적어 수동 검토 비용이 낮다).
- 서류 보존·파기는 0025 §8 정책에 편입. 수집 항목 추가이므로 **pmlegal 동시 수정**.

### 1.3 모듈 게이트

```sql
-- 게이트 헬퍼 — 도구 RPC 첫 줄에서 호출.
-- SECURITY DEFINER 로 만들 경우 set search_path 고정 필수(0025 관례와 동일).
create function app.has_license(p_type app.biz_license_type) returns boolean ...
  -- exists(select 1 from app.business_licenses
  --        where user_id = app.uid() and license_type = p_type and status = 'approved')
```

- 앱은 `public_profiles`(또는 전용 RPC)로 승인 업종 목록을 받아 도구 진입점을 그리고,
  서버는 각 도구 RPC 에서 `has_license()` 로 재검증한다(서버 불신 원칙, 0025 §원칙1).
- `sales`/`production` 승인은 어떤 인앱 모듈도 열지 않는다 — 스타터 패키지 QR 발급 대상
  명단(운영)에만 쓰인다.

### 1.4 `business_category` — 표시용 파생 필드로 강등

- **진실은 라이선스 테이블(업종 집합)**, `business_category`(단일값)는 지도·프로필의 대표
  표기용 파생값이다.
- 대표 업종은 **승인된 라이선스 중에서 업체가 선택**한다(기본값 = 최초 승인 업종). 미승인
  업종은 선택지에 아예 나오지 않는다 — 0025 대조 로직은 라이선스 테이블만 검증하면 되고,
  표시값과 충돌할 일이 없다.
- 지도 노출은 **대표 배지 + 보조 태그**(승인 업종 전체) — 시설 다중 카테고리(형제 행)의
  업체 계정 쪽 대응이다.

## 2. 공통 차단선 — 영업자 분양·입양 게시 금지

- **트리거**: `posts` INSERT/UPDATE 에서 `category in ('give_away','adoption')`(분양·입양 계열)
  이고 작성자가 승인된 `business_profiles` 보유자면 예외 발생. 업종 조합·활성 모드 무관.
  라이선스가 아니라 **업체 인증 자체**가 기준(겸업 우회 봉쇄).
- 앱은 작성 화면에서 해당 카테고리를 미리 숨긴다(UX) — 단 최종 강제는 트리거(원칙 3).
- 기존 데이터 점검: 적용 전 영업자 계정의 해당 카테고리 게시글 건수 확인 후 처리 방침 결정.
- **구조적 한계**: 트리거 기준이 "승인된 `business_profiles`"라 **업체 인증을 안 한 영업자**는
  코드로 못 막는다. 이 잔여분은 §0 의 "신고·모니터링 대상" 운영 정책이 담당한다(코드 답 없음).

## 3. 공유 뷰어 — 설치 전 가치의 공통 기반

모든 도구의 전제 조건. 앱은 웹 표면이 없고 인증이 커스텀 JWT 라, **로그인 없이 열리는
공개 조각**을 Edge Function 하나로 만든다.

### 3.1 구조

```sql
create table app.share_links (
  token        varchar(24) primary key,      -- URL-safe 랜덤(서버 생성)
  kind         varchar(20) not null,         -- 'care_report' | 'facility_preview' | ...
  ref_id       uuid not null,                -- kind 별 대상(리포트 id, 시설 id …)
  created_by   uuid not null references public.users(id),
  expires_at   timestamptz not null,         -- 기본 30일(도구별 조정)
  view_count   int not null default 0,
  revoked_at   timestamptz
);
create index share_links_ref_idx on app.share_links (kind, ref_id); -- 리포트→링크 역조회
```

- **`share-view` Edge Function**: `GET /share-view?t=<token>` → 토큰 검증(만료·회수) →
  kind 별 데이터 조회(service_role, 필요 최소 필드만) → **HTML 직접 반환**.
  사진은 비공개 버킷의 **단기 signed URL** 로 삽입(페이지 자체는 오래 살아도 원본 URL 유출 방지).
- 페이지 구성: 매장명 + 콘텐츠(사진 등) + OG 태그(카톡 미리보기) + "앱에서 계속 받기" 버튼
  (스토어 딥링크). 후기 유도·가입 강제는 여기서 하지 않는다 — 가치 먼저.
- 열람은 `view_count` 증가 + 퍼널 이벤트 기록(§8). 개인 식별 없음(쿠키·전화번호 수집 안 함).
- QR 캠페인의 "스캔 시 매장 페이지 + 동네 미리보기"도 같은 함수의 `kind='facility_preview'`
  로 처리한다(시설·후기 요약은 공개 데이터라 토큰 없이도 가능하나, 매장별 귀속 계측을 위해
  매장마다 고유 토큰 QR 을 발급한다).

### 3.2 보안 노트

- 토큰은 추측 불가 길이(≥16바이트 랜덤)·만료 필수. 링크 소지 = 열람 권한(사진 내용의 민감도가
  낮은 도구부터 시작하는 이유이기도 함). 알림장처럼 기간이 긴 콘텐츠는 회수(`revoked_at`) 지원.
- **claim 겸용 링크**(`care_report`)는 열람과 claim 의 창이 다르다 — 열람은 링크 만료(기본
  30일)까지 반복 허용(가족 공유 감안), **토큰 단독 claim 은 발행 후 72시간**·1회 claim 후 잠금
  (§4.2). 전화번호 대조 자동 연결은 토큰과 무관하게 링크 수명(30일) 내내 동작.
- service_role 은 Edge Function 내부에서만 사용(클라이언트 금지 원칙 유지).

## 4. 도구 원형 — `care_reports` (미용 전후 사진 → 알림장 공용)

미용 "전후 사진"과 위탁 "알림장"은 같은 원형의 1회성/반복형이다. 테이블을 처음부터 공용으로 설계한다.

### 4.1 스키마

```sql
create table app.care_reports (
  id           uuid primary key default gen_random_uuid(),
  business_id  uuid not null references public.users(id),  -- 발행 업체(계정)
  kind         varchar(12) not null check (kind in ('grooming','boarding')),
  pet_label    varchar(50) not null,          -- 업체가 입력하는 아이 이름(미가입 보호자 전제)
  photos       jsonb not null default '[]',   -- [{path, caption?}] — 미용: 전/후 2장+
  body         jsonb not null default '{}',   -- kind 별 필드: boarding 은 식사·배변·특이사항 등
  note         text,
  recipient_phone_hmac bytea,                 -- 수신 보호자 전화번호 키드 해시(§4.2). claim 성사 시 null 로 파기
  recipient_key_version smallint not null default 1,  -- HMAC 키 버전 선점 — 키 유출 시 무마이그레이션 로테이션
  claimed_by   uuid references public.users(id),  -- 연결된 보호자 계정
  claimed_at   timestamptz,                   -- 첫 claim 시각 — 이후 잠금(재 claim 불가)
  created_at   timestamptz not null default now()
);
```

- **발행 플로우(미용)**: 업체 모드 → 전후 사진 촬영/선택 → 아이 이름 + **고객 전화번호 입력**
  (예약 때문에 어차피 보유) → 발행 → `share_links` 토큰 생성 → 공유 시트(카톡·문자는
  **업체 폰에서** 전송). RPC 첫 줄 `has_license('grooming')`. 전화번호는 서버에서 HMAC 처리
  후 즉시 버린다(평문 미저장·미로깅).
- **수령 플로우**: 보호자는 링크로 즉시 열람(설치 불요) → "앱에서 우리 아이 기록 모아보기"
  버튼 → 설치·가입.
- 사진 버킷은 비공개 + signed URL(§3). 업체 모드 사진과 개인 사진의 경로 분리는 0026 관례를 따른다.

### 4.2 claim — 전화번호 해시 대조가 1차, 토큰은 폴백

계정 인증이 이미 전화 OTP 기반이므로 **가입 계정의 인증된 번호**를 그대로 대조 근거로 쓴다.

1. **자동 연결(기본)**: 가입 완료(또는 로그인) 시 서버가 `HMAC(내 인증 번호)` = `recipient_phone_hmac`
   인 미연결 리포트를 찾아 자동 연결. 보호자 마찰 0 — **가입한 순간 리포트가 이미 도착해 있는**
   흐름. 링크가 제3자에게 새도 남의 번호로는 claim 불가.
2. **업체 승인 폴백(예외)**: 번호 불일치(가족 명의 가입 등)로 자동 연결이 안 된 경우에만,
   토큰 열람 화면에서 연결 요청 → 업체 발행 목록에 승인 버튼. 업체 부담은 예외 케이스로 한정.
3. **v1 토큰 claim 의 최소 장치**(사진 공유는 민감도가 낮아 토큰 단독 claim 도 허용하되):
   토큰 단독 claim 창은 발행 후 **72시간** + 첫 claim 후 잠금(`claimed_at`) + 업체 발행 목록에
   **수령자 표시**(오연결을 업체가 발견 가능). 열람 자체는 링크 만료(30일)까지 계속 가능.
4. **개인정보**: 해시는 서버 시크릿 키드 HMAC(전화번호 공간이 작아 무염 SHA 는 전수 대입에
   취약). `key_version` 컬럼을 선점해 키 유출 시 무마이그레이션 로테이션(평시엔 단일 키 유지).
   claim 성사 시 `recipient_phone_hmac` 즉시 null 파기. **미연결분 파기는 링크 만료(30일)와
   동기화** — 72h 에 파기하면 링크 받고 며칠 뒤 설치하는 보호자(현실적으로 다수)의 자동 연결이
   전부 죽고 폴백만 남는다. 72h 는 토큰 단독 claim 잠금용 수치일 뿐, 키드 해시라 보존 부담이
   낮으므로 자동 연결 창은 링크 수명만큼 연다. 처리방침에 수집·보존(30일)·파기 명시(pmlegal
   동시 수정, §6).

### 4.3 딥링크 — 도입하지 않는다 (계측 먼저)

전화번호 대조가 토큰 의존을 없애므로 디퍼드 딥링크의 문제 자체가 거의 소멸한다.

- 카톡·문자 링크는 유실되지 않으므로 "가입 후 재방문"이 실제로 동작 — 랜딩(공유 뷰어)에
  설치 감지 + "앱에서 열기" 버튼만 제대로 만든다.
- 유실 위험은 QR 현장 스캔 경로뿐 — 필요해지면 클립보드 방식(iOS 붙여넣기 프롬프트 감안)이
  싸고 충분. 서드파티 SDK(Airbridge/Branch)는 **발행→claim 전환율 계측으로 실제 이탈이
  확인된 뒤**에만 검토.
- 지금 할 일은 도입이 아니라 계측: 발행→열람→가입→claim 퍼널 이벤트를 먼저 심는다(§7).

### 4.4 알림장(위탁, P2) — '건'을 모델링하지 않는다

수동 종료는 업체가 잊고, 날짜 기반은 연장·조기퇴소에 깨진다. 시작·종료 결정 자체를 소멸시킨다.

- **스레드 = 반려동물 × 업체 관계 1개, 상시 유지.** 기록(`care_reports`, `kind='boarding'`)은
  스레드에 시간순으로 쌓이고 **날짜로 그룹핑**해 표시한다. "위탁 건"의 경계는 UI 의 날짜
  구분선일 뿐, 엔티티가 아니다.
- **무입력 N일 경과 시 자동 보관**(업체 목록에서 하단 접힘), 새 기록이 오면 자동 복귀.
  호텔형(단건)과 유치원형(상시 등원)이 같은 구조로 커버된다. 오탐의 피해가 "하단 접힘"뿐이고
  자동 복귀가 있어 저위험 — **임계값은 설정 테이블로**(무마이그레이션 튜닝, 기본 7일). 주중
  등원형이 주말+연휴로 오탐 보관되는 게 확인되면 boarding 기본만 10일로 올린다.
- 건 단위 요약 리포트가 실제로 필요해지는 시점에 `stay` 엔티티를 나중에 도입한다(스레드
  구조는 그대로 두고 요약 층만 추가 — 마이그레이션 부담 없음).
- 가입 보호자에게는 인앱 알림(0027 파이프라인)으로 도착 통지, 미가입 보호자는 매 발행마다
  업체가 링크 재전송.

## 5. 분양 스타터 패키지 (P3, 앱 밖 채널)

- 구성: 예방접종·건강검진 일정 알림, 초보 보호자 체크리스트, 동네 병원·미용실 지도 바로가기.
  개발 실체는 **콘텐츠 + 기존 알림 인프라**(0027)이며 신규 서버 구조가 거의 없다.
- 배포: `sales`/`production` 허가 업체에 스타터 QR 을 제공 — QR 은 `facility_preview` 변형
  랜딩(스타터 소개 + 설치)으로 연결. **앱 안에 분양업체용 기능·노출은 만들지 않는다**(원칙 4).
- 가입 시 온보딩 분기: "최근에 아이를 데려왔어요" → 접종 스케줄 등록 유도.

| 네이버 POI 약관 | `ensure_naver_facility` 의 지역검색 결과 영속 저장이 약관 허용 범위인지 확인(조사는 별도 트랙). **스키마는 결론 전에 미리 가른다**: `facilities.source` 로 후기 허용을 게이트 — 후기 작성은 first-party 시설(공공데이터 localdata·제휴 자체등록)만, `source='naver'` 캐시 POI 는 **표시 전용**. 결론이 "부착 금지"여도 마이그레이션 없이 정책만 확정하면 되고, "허용"이면 게이트 스위치만 연다. 제휴 업체는 어차피 자체 등록이라 후기 기능이 제휴 트랙과 자연스럽게 묶인다 | pmdb(후기 게이트) + 조사 태스크 |

| 캠페인 시점 | **위치기반서비스사업 신고(방통위) 수리**·정식 오픈 이후에만 오프라인 배포 시작(통신판매업 신고는 결제·중개 부재로 비대상) | 운영 |

## 7. 파일럿 퍼널 계측

"측정하고 확장 판단"이 성립하려면 계측이 선행 조건이다. 외부 SDK 없이 최소 자체 테이블로 시작한다.

```sql
create table app.funnel_events (
  id          bigint generated always as identity primary key,
  event       varchar(30) not null,   -- 'share_view' | 'store_click' | 'signup' | 'claim' | 'review' | ...
  token       varchar(24),            -- share_links 귀속(있으면) → 매장·도구별 전환 집계
  user_id     uuid,                   -- 가입 이후 이벤트만
  props       jsonb not null default '{}',
  created_at  timestamptz not null default now()
);
create index funnel_events_token_idx on app.funnel_events (token, event); -- 매장·도구별 전환 집계
-- 보존: 원시 이벤트 1년(경과분 배치 삭제) — 장기 추세는 집계 뷰/스냅샷으로.
```

- 기록 주체: 발행 RPC(`report_issued`), `share-view` 함수(열람·스토어 클릭), signup 엣지(가입),
  claim 처리(자동 연결·승인 폴백 구분 기록), 앱(후기). 디퍼드 딥링크가 없으므로 설치 귀속은
  전화번호 대조 성사(`claim`)가 사실상의 귀속 신호다.
- 핵심 지표: 매장별 발행 수 → 열람율 → 스토어 클릭율 → 가입율 → claim 율(자동/폴백 비율 포함)
  → 후기율 → D7 잔존. **발행→claim 이탈이 커야만** 딥링크 SDK 검토를 연다(§4.3).
- 폴백 승인은 이벤트로 구분 기록되므로 **업체별 폴백 비율 모니터링**이 공짜로 따라온다 —
  비정상적으로 높은 업체는 오·남용 조사 대상(§4.2 폴백 남용 방어의 운영 축).
- 확장 판단 기준(파일럿 종료 시): 업체 재사용률(주당 발행 수 유지 여부)이 1순위 —
  도구가 사장님에게 실제로 쓸 이유가 됐는지가 이 전략의 가설이기 때문.

## 8. 로드맵

| 단계 | 내용 | 산출물 |
| --- | --- | --- |
| **P0 기반** | `business_licenses` + 승인 플로우, 공유 뷰어(`share_links` + `share-view`), 영업자 게시 차단 트리거, 대가성 표시, `funnel_events` | 파일럿 가능 상태 |
| **P1 미용 파일럿** | `care_reports(kind='grooming')` 발행·claim, 1개 동네 5~10개 미용업체 제휴, QR(매장 미리보기) 병행 | 퍼널 수치, 업체 재사용률 |
| **P2 위탁 알림장** | `kind='boarding'` 반복 발행 + 위탁 건 스레드 + 인앱 알림 | 잔존·재방문 수치 |
| **P3 분양 스타터** | 접종 알림 콘텐츠 + 스타터 QR 랜딩 + 온보딩 분기 | 분양 채널 가입 수 |

P0 내부 순서는 공유 뷰어를 먼저(모든 도구의 전제) — 라이선스 승인 플로우는 파일럿 업체가
소수라 초기엔 관리자 수동으로 갈음할 수도 있으나, 게이트 함수(`has_license`)는 처음부터 둔다.
