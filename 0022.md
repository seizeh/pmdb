# 0022 — 시설 후기(사진 포함) (별점·사진 후기 → 시설 마커에 표시)

> PawMate(`pmdart`) + Supabase 백엔드(`pmdb`) 기준.
> **선행 의존: 0021(시설 지도 — facilities 테이블/네이버 실시간 애견카페), 0019/0020(media 버킷·사진 업로드 패턴), 0017(geofence·위치).**
> 인증: 커스텀 HS256 JWT, `app.uid()` = JWT sub. 쓰기는 SECURITY DEFINER RPC 경유.

---

## 0. 목표와 결정 요약

- 지도 시설(동물병원·애견미용·반려동물호텔·애견카페)에 **사진 포함 별점 후기**를 달 수 있게 한다.
- 마커 탭 바텀시트에 **평균 별점·후기 수**를 보여주고, 후기 목록·작성으로 진입한다.

### 결정값(확정)

| 항목 | 결정 | 비고 |
| --- | --- | --- |
| 카페 후기 anchor | **첫 후기 시 facilities 로 승격** | `source='naver'`, `ext_id`=이름+주소 해시. CSV 시설과 동일 집계 |
| 사진 정책 | **갤러리 허용** | 후기 특성상 미리 찍은 사진 OK. 0019 카메라 강제 **미적용** |
| 작성 자격 | **누구나(로그인)** | 1인·1시설 1후기. 신고·숨김 모더레이션으로 사후 관리 |
| 별점 | **5점 별점 + 글 + 사진** | 시설별 평균 별점·후기 수 캐시 |
| 사진 저장 | **공개 `media` 버킷 재사용** | 경로 `<uid>/facility_review/<review_id>/<i>.jpg` |
| 쓰기 경로 | **SECURITY DEFINER RPC** | 클라 직접 INSERT 금지(0019/0020 정합) |

### ★ 가장 큰 함정 — 애견카페는 DB row 가 없다

후기는 `facility_id`(uuid)에 매달려야 하는데, CSV 3종은 `facilities.id` 가 있지만 **애견카페는 0020 §4-1 네이버 실시간이라 DB 에 없다.** 게다가 네이버 지역검색은 **안정적인 place id 를 주지 않는다**(업체명·주소·전화·좌표뿐). 그래서 **첫 후기를 다는 순간 그 카페를 `facilities` 로 승격 저장**하고(`source='naver'`, `ext_id`=정규화한 이름+도로명주소 해시), 그 id 에 후기를 단다. 이후 같은 카페의 후기는 동일 row 를 재사용한다.

## 1. 전체 플로우

```
[지도 마커 탭]  (CSV 시설 = facilities.id 있음 / 애견카페 = id 없음)
   │
   ├─ ① 바텀시트: 평균 별점·후기 수 + 후기 목록(app.facility_reviews_of)
   │
   └─ ② [후기 작성]
         │  카페면: app.ensure_naver_facility(name,addr,phone,lng,lat) → facility_id 확보(승격)
         │  사진: 갤러리/카메라 선택 → media 버킷 업로드(<uid>/facility_review/<rid>/i.jpg)
         ▼
      [Edge Function: submit-facility-review]  (로그인 JWT 검증)
         │  ③ 입력 검증(별점 1~5, 본문 길이, 사진 수) → RPC 호출
         ▼
      [RPC app.add_facility_review]  (SECURITY DEFINER)
         │  ④ facility_reviews insert (1인·1시설 unique)
         │  ⑤ 트리거가 facilities.avg_rating / review_count 갱신
         ▼
      [지도/바텀시트 갱신]  평균 별점·후기 반영
```

## 2. 스키마

### 2-1. `facilities` 에 집계 컬럼 추가 (0021 확장)

```sql
-- migration: 2026XXXX_facilities_review_aggs.sql
alter table public.facilities
  add column if not exists avg_rating   numeric(2,1) not null default 0,  -- 0.0~5.0
  add column if not exists review_count integer      not null default 0;
```

> 승격된 카페(`source='naver'`)도 `facilities` 행이므로 동일 컬럼을 그대로 쓴다. 단 카페는 좌표가 KATECH→WGS84 변환값(0020 §4-1)이라 `src_x/src_y` 는 비우고 `geom` 만 채운다.

### 2-2. `facility_reviews`

```sql
-- migration: 2026XXXX_facility_reviews.sql
create table public.facility_reviews (
  id            uuid primary key default gen_random_uuid(),
  facility_id   uuid not null references public.facilities(id) on delete cascade,
  user_id       uuid not null,                       -- app.uid()
  rating        smallint not null check (rating between 1 and 5),
  body          text check (char_length(body) <= 1000),
  photo_paths   text[] not null default '{}',        -- 버킷 경로(서명/공개 URL 생성용)
  photo_urls    text[] not null default '{}',        -- 공개 URL(편의 캐시)
  visibility_status public.visibility_status not null default 'visible',
  created_at    timestamptz not null default now(),
  updated_at    timestamptz not null default now(),
  -- 1인 1시설 1후기(수정은 update). 재작성 막고 평점 어뷰징 완화
  constraint facility_reviews_user_uq unique (facility_id, user_id),
  -- 사진 최대 5장(앱·서버 공통 정책)
  constraint facility_reviews_photos_max check (array_length(photo_paths,1) is null
                                                or array_length(photo_paths,1) <= 5)
);
create index facility_reviews_fac_idx  on public.facility_reviews (facility_id)
  where visibility_status = 'visible';
create index facility_reviews_user_idx on public.facility_reviews (user_id);

comment on table public.facility_reviews is
  '시설 후기. 1인1시설1후기. 사진은 media 버킷 경로. visibility 로 숨김/신고 관리.';
```

`visibility_status` enum 은 0017/0018 의 것을 재사용한다(`visible` / `hidden_by_user` / `hidden_by_admin` / `deleted_by_user` / `deleted_by_admin`).

### 2-3. RLS

```sql
alter table public.facility_reviews enable row level security;
grant select on public.facility_reviews to authenticated;

-- 읽기: 공개(visible)거나 본인 글
create policy fr_select on public.facility_reviews
  for select using (visibility_status = 'visible' or user_id = app.uid());

-- INSERT/UPDATE/DELETE GRANT 없음 → 클라 직접 쓰기 불가. RPC(SECURITY DEFINER)만.
```

## 3. 카페 승격 RPC `app.ensure_naver_facility`

네이버 카페는 후기 직전에 `facilities` 로 승격해 `facility_id` 를 만든다(이미 있으면 그 id 반환).

```sql
-- migration: 2026XXXX_ensure_naver_facility_rpc.sql
create or replace function app.ensure_naver_facility(
  p_name    text,
  p_address text,
  p_phone   text,
  p_lng     double precision,
  p_lat     double precision
) returns uuid
language plpgsql security definer set search_path = ''
as $$
declare
  v_ext  text;
  v_id   uuid;
begin
  if app.uid() is null then raise exception 'auth required'; end if;
  -- 안정 키: 공백 제거 + 소문자 이름|주소 해시(네이버가 place id 를 안 주므로)
  v_ext := md5(lower(regexp_replace(coalesce(p_name,'')||'|'||coalesce(p_address,''),
                                    '\s', '', 'g')));
  insert into public.facilities
    (category, source, ext_id, name, address, phone, is_open, geom)
  values ('pet_cafe', 'naver', v_ext, p_name, p_address, p_phone, true,
          st_setsrid(st_makepoint(p_lng, p_lat), 4326)::geography)
  on conflict (source, ext_id) do update set name = excluded.name  -- no-op로 RETURNING 보장
  returning id into v_id;
  return v_id;
end;
$$;
revoke all on function app.ensure_naver_facility(text,text,text,double precision,double precision) from public;
grant execute on function app.ensure_naver_facility(text,text,text,double precision,double precision) to authenticated;
```

> **승격 신뢰 주의:** 이름 표기 차이("○○애견카페" vs "○○ 애견카페")로 해시가 갈리면 같은 카페가 2개 row 가 될 수 있다. 정규화(공백·괄호·지점명 제거)를 보수적으로 맞추고, 운영 중 중복이 보이면 병합 도구를 둔다(§9).

## 4. 후기 작성 RPC `app.add_facility_review`

```sql
-- migration: 2026XXXX_add_facility_review_rpc.sql
create or replace function app.add_facility_review(
  p_review_id  uuid,            -- 클라 생성(사진 경로와 일치시키기 위함)
  p_facility   uuid,
  p_rating     smallint,
  p_body       text,
  p_paths      text[] default '{}',
  p_urls       text[] default '{}'
) returns uuid
language plpgsql security definer set search_path = ''
as $$
declare v_uid uuid := app.uid();
begin
  if v_uid is null then raise exception 'auth required'; end if;
  if p_rating < 1 or p_rating > 5 then raise exception 'rating 1..5'; end if;

  insert into public.facility_reviews
    (id, facility_id, user_id, rating, body, photo_paths, photo_urls)
  values (p_review_id, p_facility, v_uid, p_rating, p_body,
          coalesce(p_paths,'{}'), coalesce(p_urls,'{}'))
  on conflict (facility_id, user_id) do update      -- 같은 시설 재작성 = 수정
    set rating = excluded.rating, body = excluded.body,
        photo_paths = excluded.photo_paths, photo_urls = excluded.photo_urls,
        visibility_status = 'visible', updated_at = now()
  returning id into p_review_id;
  return p_review_id;
end;
$$;
revoke all on function app.add_facility_review(uuid,uuid,smallint,text,text[],text[]) from public;
grant execute on function app.add_facility_review(uuid,uuid,smallint,text,text[],text[]) to authenticated;
```

### 4-1. 집계 트리거 (평균 별점·후기 수)

매 조회마다 집계하지 않고, 후기 변경 시 `facilities` 캐시를 갱신한다.

```sql
-- migration: 2026XXXX_facility_review_aggs_trigger.sql
create or replace function app.refresh_facility_aggs(p_facility uuid)
returns void language sql security definer set search_path = '' as $$
  update public.facilities f set
    review_count = sub.cnt,
    avg_rating   = coalesce(round(sub.avg_r, 1), 0)
  from (select count(*) cnt, avg(rating)::numeric avg_r
          from public.facility_reviews
         where facility_id = p_facility and visibility_status = 'visible') sub
  where f.id = p_facility;
$$;

create or replace function app.tg_facility_review_aggs()
returns trigger language plpgsql security definer set search_path = '' as $$
begin
  perform app.refresh_facility_aggs(coalesce(new.facility_id, old.facility_id));
  return null;
end; $$;

create trigger facility_reviews_aggs
  after insert or update or delete on public.facility_reviews
  for each row execute function app.tg_facility_review_aggs();
```

> 숨김/삭제(visibility 변경)도 `update` 라 트리거가 평균을 다시 계산한다 → 숨겨진 후기는 별점에서 자동 제외.

## 5. 후기 조회 RPC `app.facility_reviews_of`

```sql
-- migration: 2026XXXX_facility_reviews_of_rpc.sql
create or replace function app.facility_reviews_of(
  p_facility uuid, p_limit int default 20, p_offset int default 0
) returns table (
  id uuid, user_id uuid, rating smallint, body text,
  photo_urls text[], created_at timestamptz, is_mine boolean
) language sql stable security definer set search_path = '' as $$
  select r.id, r.user_id, r.rating, r.body, r.photo_urls, r.created_at,
         (r.user_id = app.uid()) as is_mine
    from public.facility_reviews r
   where r.facility_id = p_facility
     and r.visibility_status = 'visible'
   order by r.created_at desc
   limit least(p_limit, 50) offset p_offset;
$$;
grant execute on function app.facility_reviews_of(uuid,int,int) to authenticated;
```

작성자 닉네임·프로필은 클라에서 user_id 로 별도 조회하거나, 이 RPC 에 join 을 추가한다(프로필 테이블 스키마에 맞춰 확장).

## 6. Flutter — 작성·표시 (pmdart)

### 6-1. 후기 작성

```dart
// 0) 카페면 먼저 승격해서 facility_id 확보 (CSV 시설은 이미 f['id'] 있음)
String facilityId;
if (facility.source == 'naver') {
  final r = await supabase.rpc('ensure_naver_facility', params: {
    'p_name': facility.name, 'p_address': facility.address,
    'p_phone': facility.phone, 'p_lng': facility.lng, 'p_lat': facility.lat,
  });
  facilityId = r as String;
} else {
  facilityId = facility.id;
}

// 1) 사진: 갤러리 허용(후기는 미리 찍은 사진 OK) — 0019 와 정반대
final imgs = await ImagePicker().pickMultiImage(imageQuality: 80);   // 최대 5장 제한은 UI
final reviewId = const Uuid().v4();                                   // 경로·DB id 일치
final paths = <String>[]; final urls = <String>[];
for (var i = 0; i < imgs.length && i < 5; i++) {
  final path = '${uid}/facility_review/$reviewId/$i.jpg';
  await supabase.storage.from('media').upload(path, File(imgs[i].path));
  paths.add(path);
  urls.add(supabase.storage.from('media').getPublicUrl(path));
}

// 2) 후기 저장 (Edge Function 또는 직접 RPC). 별점 1~5 + 본문
await supabase.functions.invoke('submit-facility-review', body: {
  'review_id': reviewId, 'facility_id': facilityId,
  'rating': rating, 'body': bodyText, 'paths': paths, 'urls': urls,
});
```

- **갤러리 허용이 핵심.** 0019 펫 인증은 도용 방지로 카메라 강제였지만, 후기는 미리 찍은 사진을 올리는 게 정상이라 `pickMultiImage`(갤러리)를 쓴다. 카메라 촬영도 옵션으로 같이 제공.
- `reviewId` 를 클라에서 만들어 **사진 경로와 DB id 를 일치**시킨다(0019 패턴).

### 6-2. 표시

```dart
// 마커 탭 바텀시트: 평균 별점·후기 수 (facilities 캐시 컬럼)
Text('★ ${f.avgRating}  (${f.reviewCount})');

// 후기 목록
final reviews = await supabase.rpc('facility_reviews_of',
    params: {'p_facility': facilityId, 'p_limit': 20});
// 각 항목: 별점 + 본문 + 사진 썸네일(photo_urls) + 내 글이면 수정/삭제
```

- 지도 마커 라벨/아이콘에 평균 별점을 작게 얹으면 탐색에 좋다(과밀 시 줌인 후).
- **카페는 후기가 0개면 facilities 에 없으니** 바텀시트 평균은 "후기 없음"으로 표시하고, 첫 후기 작성 시 승격된다.

## 7. 보안 / 정책

| 항목 | 처리 |
| --- | --- |
| 후기 쓰기 | RPC(SECURITY DEFINER)만. 클라 직접 INSERT GRANT 없음 |
| 작성자 위조 | `user_id` 는 RPC 안에서 `app.uid()` 로 세팅(클라 입력 무시) |
| 평점 어뷰징 | 1인·1시설 unique(재작성=수정). 대량 가짜계정은 신고·모더레이션 대응 |
| 사진 소유 | 경로 `<uid>/...` 로 본인 폴더에만 업로드. 버킷 정책으로 타인 폴더 쓰기 차단 |
| 숨김/신고 | `visibility_status`(0017/0018 재사용). 숨긴 후기는 평균에서 자동 제외 |
| 카페 승격 남용 | `ensure_naver_facility` 는 로그인 필수 + 해시 키로 중복 방지 |
| 부적절 내용 | 신고 → `hidden_by_admin`. 필요 시 텍스트/이미지 자동검수(추후) |

## 8. 임계값·정책 정리

| 값 | 기본 | 의미 |
| --- | --- | --- |
| 별점 범위 | **1~5** | check 제약 |
| 본문 길이 | **≤1000자** | check 제약 |
| 사진 수 | **≤5장** | DB check + 앱 제한 |
| 1인 1시설 | **1후기**(수정 가능) | unique(facility_id,user_id) |
| 목록 페이지 | **20건/요청(최대 50)** | RPC limit |
| 평균 갱신 | **후기 변경 트리거** | 조회 시 집계 안 함 |

## 9. 엣지 케이스

- **카페 중복 승격**: 이름 표기 차이로 같은 카페가 2 row → 정규화 보강 + 운영 병합 도구(후기 facility_id 이전 후 빈 row 정리).
- **승격 후 카페 폐업/이전**: facilities 에 남아 후기는 유지. `is_open` 은 카페가 네이버 실시간이라 갱신 경로가 없음 → 후기 시설은 "사용자 등록 시설" 취급(주의 표시 가능).
- **사진 업로드 후 RPC 실패**: 고아 사진 발생 → 주기 정리 잡(참조 없는 `facility_review/*` 삭제) 또는 RPC 성공 후에만 확정.
- **후기 삭제**: `visibility_status='deleted_by_user'` 로 소프트 삭제 → 트리거가 평균 재계산. 사진은 정리 잡에서 제거.
- **폐업 CSV 시설 후기**: 시설이 `is_open=false` 여도 과거 후기는 보존(이력). 지도 노출만 필터.
- **빈 본문/사진만**: 별점만 있어도 허용할지 정책 결정(현재 body nullable, 사진 0장 허용).
