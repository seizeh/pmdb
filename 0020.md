# 0020 — 반려동물 신원 인증 (영상 등록 → 프레임 저장 → 게시글 사진 동일개체 매칭)

> PawMate(`pmdart`) + Supabase 백엔드(`pmdb`) 기준.
> **선행 의존: 0017(지역 인증), 0018(게시글 사진 검증 — 카메라 강제/지역 일치/서버 토큰).**
> AI 공급자: **Google Gemini 2.5 Pro** (`gemini-2.5-pro`, 멀티 이미지 비교 + 구조화 출력).

## 0. 목표와 결정 요약

- 반려동물을 **등록할 때 짧은 영상**을 찍되, 앱이 **무작위 동작 임무**(예: 머리 쓰다듬기, 앞발 잡기)를 지시하고 **그 임무를 수행하는 모습**을 찍게 한다. AI가 영상을 보고 **임무 수행 + 실제 살아있는 펫 + 동일 개체**가 맞으면 **승인**한다.
- 승인되면 그 **영상에서 프레임 N장**만 뽑아 **펫의 기준(reference) 이미지**로 저장한다. **원본 영상은 저장하지 않는다**(AI 판별용으로만 잠깐 전송, 우리 Storage·DB 어디에도 보관 안 함).
- 게시글을 쓸 때 **카메라로 찍은 사진**을, 작성자가 선택한 **그 펫의 기준 프레임들과 비교**해 **같은 개체인지** AI로 판정한다.
- 판정 기준은 **동일 개체 추정(재식별) · 중간 신뢰 통과**. 같은 펫인데 각도·조명이 달라 애매한 경우는 관대하게 통과(정상 사용자 오거부 최소화).
- **결정값(AskUserQuestion 확정):** 인증 단위 = **펫별(등록 시)** · 저장/추출 = **프레임만, 클라이언트 추출** · 매칭 = **동일 개체 추정·중간 신뢰**.
- **세부 정책(확정):** 프레임 추출 = **단일 영상 1개에서 전 구간 분산 샘플링**(미션별 2회 촬영 ✕) · 영상 길이 = **약 11초**(iPhone 17 HD60 ≈ 3.2MB/s → ~35MB, 인라인 100MB 한도 여유) · 임무 수 = **2개**(추후 보강) · 기준 프레임 버킷 = **공개 `media` 재사용**(도용 동기 낮다고 판단) · 펫 없는 글 사진 = **고의 면제**(자유게시판 카테고리가 있어 무관 사진 허용).
- **등록정보 교차검증(확정):** 등록 영상 AI 가 동물종(강아지/고양이)·품종·털색을 추출해 `pets` 등록값과 대조한다. **동물종·품종 불일치는 소프트 경고**(등록은 통과, 앱이 "확인해 주세요" 안내), **털색은 기록만**(불일치 무관). 품종·색 AI 판별 오류가 흔해 하드 차단하지 않는다.
- **동작 임무(challenge)는 능동형 라이브니스**다. 임무가 매번 무작위라 미리 찍어둔 영상·화면 재생·사진으론 통과 불가. "사람 손과 실제 펫의 상호작용"을 요구해 스푸핑을 크게 억제한다.
- 0018의 **카메라 강제·촬영 위치=활동지역 일치·1회용 서버 토큰**은 그대로 유지한다. 0018의 게시 단계 "실제 개/고양이냐(종·라이브니스)" 판정은 **"이 펫이 맞느냐(동일 개체)"** + 라이브니스(화면 재촬영/AI 생성 차단)로 **대체·확장**한다.

## 1. 전체 플로우

### (A) 펫 등록 — 신원 영상 인증

```
[pmdart] 펫 등록(pet_edit_screen) → pets INSERT 로 pet_id 확보
   │  ① 앱이 무작위 임무 1~2개 제시 (예: "머리 쓰다듬기" + "앞발 잡기")
   │  ② image_picker.pickVideo(source: camera, maxDuration 6~8s) → 임무 수행 영상 촬영
   │  ③ video_thumbnail 로 고정 타임스탬프 프레임 4장 추출 (클라)
   │  ④ 영상(base64, AI용) + 프레임 N장(base64, 저장용) + pet_id + challenge[] 전송
   ▼
[Edge Function: enroll-pet-identity]  (JWT 수동검증 → uid; pet_guardians 직접 확인)
   ① Gemini(영상 입력): (a) 실제 살아있는 개/고양이 (b) 영상 내내 같은 개체
        (c) 지시한 임무(challenge)를 실제로 수행했는가  (d) 추정 품종·털색(참고)
        → 아니면 fail(not_real_pet / not_consistent_pet / challenge_failed)
   ①' 등록정보 대조: AI 종/품종 vs pets 등록값 → 불일치는 warnings[](소프트, 차단 안 함)
   ② 통과: service_role 로 프레임 N장만 업로드 + pet_identity_frames N행 insert
        + pets.identity_verified=true(+ai_breed/ai_colors/info_match).  ★ 영상 미저장
   ▼ { enrolled:true, frames:[…urls], species:"dog", warnings:[…], challengePassed:true }
```

> **영상 미저장 원칙:** 영상은 임무 검증을 위해 Gemini 로 **인라인 전송(2026.1부터 100MB까지)** 될 뿐, Supabase Storage·DB 어디에도 쓰지 않는다. 저장되는 건 클라이언트가 뽑은 **프레임 N장**뿐이다.

### (B) 게시글 작성 — 동일 개체 매칭

```
[pmdart] 게시글 작성 → 펫 선택(_selectedPetIds 중 1마리 = 검증 대상)
   │  ① capturePostPhoto()(0018, 카메라 전용) → 사진 1장
   │  ② LocationService.current()(0017) → 좌표/accuracy/isMocked
   │  ③ 사진(base64) + 좌표 + targetPetId 를 verify-post-photo 로 전송
   ▼
[Edge Function: verify-post-photo]  (개정)
   ① 활동지역 인증 상태 + 모의위치 + 촬영지 행정동==users.region_code  (0018 그대로)
   ② 대상 펫의 기준 프레임 조회(pet_identity_frames). 미등록 → fail(pet_not_enrolled)
   ③ Gemini 멀티이미지: [기준프레임 N장] + [게시사진 1장]
        → { same_individual, identity_score, is_real, dog_real,cat_real,dog_fake,cat_fake, reason }
   ④ 통과 판정(§7): identity_score >= 임계(중간) AND is_real
        실패 → fail(identity_mismatch | not_real_pet)
   ⑤ 통과: 사진 업로드 + photo_verifications(토큰, matched_pet_id, identity_score) 발급
   ▼ { pass:true, token, imageUrl, matchedPetId }
[posts INSERT 트리거] 토큰 유효 + image_url 일치 + matched_pet_id 가 글에 연결된 펫인지 확인 → 소진
```

> 핵심 보안 모델은 0018과 동일: **카메라 강제(1차)** + **서버 토큰(2차, 위변조 차단)**. 0019는 거기에 **"같은 펫"** 이라는 의미를 더한다.

## 3. 펫 등록 — 영상 인증 UX (pmdart)

`pet_edit_screen.dart` 의 등록 흐름에 신원 인증 단계를 둔다. 두 가지 배치 중 택1:

- **(권장) 등록 직후 유도**: `pets` INSERT 로 `pet_id` 확보 → "신원 인증(영상)" 단계로 이동 → 완료해야 `identity_verified=true`. 미완료 펫은 게시글 검증 대상에서 제외(작성 시 안내).
- **별도 액션**: 펫 상세(`pet_detail_screen`)에 "신원 인증하기" 버튼.

**동작 임무(challenge) 풀.** 앱이 아래에서 매번 **무작위 1~2개**를 골라 화면에 지시한다(예: "① 머리를 쓰다듬어 주세요  ② 앞발을 잡아 주세요"). 무작위라 미리 찍어둔 영상으로는 통과할 수 없다.

| 코드 | 지시 문구 | AI가 볼 것 |
| --- | --- | --- |
| `pat_head` | 머리를 쓰다듬기 | 사람 손이 펫 머리를 쓰다듬는 동작 |
| `hold_paw` | 앞발(손) 잡기 | 손이 펫의 앞발을 잡는 동작 |
| `scratch_chin` | 턱·목 쓰다듬기 | 손이 턱 아래를 만지는 동작 |
| `stroke_back` | 등 쓰다듬기 | 손이 등을 따라 쓰다듬는 동작 |
| `hand_in_frame` | 손을 펫 옆에 대기 | 같은 프레임에 사람 손과 펫이 함께 |

> 임무는 "사람 손 ↔ 실제 펫의 물리적 상호작용"을 요구하므로 사진/화면으론 재현 불가. 너무 어려운 묘기(악수 등 훈련 필요)는 피하고 누구나 가능한 접촉 동작 위주로 구성.

**★ 카메라 강제 원칙(영상에도 동일 적용).** 0018의 사진과 똑같이, "영상 촬영" 버튼을 누르면 **즉시 기기 카메라가 열려 그 자리에서 촬영**해야 하며 **갤러리에 미리 찍어둔 영상은 올릴 수 없다.** 구현상 `ImagePicker().pickVideo(source: ImageSource.camera)` 만 호출하고 `ImageSource.gallery` 경로는 신원 인증 화면에서 절대 제공하지 않는다. 갤러리 picker 를 띄우는 코드가 이 플로우에 존재하지 않는지 코드리뷰로 확인한다.
>
> 다만 클라이언트 강제는 "정상 사용자 우회 방지" 수준임을 정직하게 본다. 루팅/탈옥 기기의 가상 카메라(미리 찍은 영상을 카메라 입력으로 주입)까지는 클라만으론 못 막는다. 그 잔여 위험은 **무작위 임무(challenge)**가 메운다 — 임무가 매번 달라 사전 제작 영상은 통과 불가. 즉 *카메라 강제(편의·1차 차단) + 무작위 임무(능동형 라이브니스)* 2중으로 본다.

촬영/추출:

```dart
// 0) 무작위 임무 선택
final challenge = (List.of(_challengePool)..shuffle()).take(2).toList();
// → UI 에 지시 문구 표시 후 촬영 시작

// 1) 카메라로 임무 수행 영상 (갤러리 금지). 임무 2개 순차 수행 → 11초.
//    UI: 임무1을 전반부(~5초), 임무2를 후반부(~10초)에 안내해 시점이 갈리게 한다.
final video = await ImagePicker().pickVideo(
  source: ImageSource.camera,
  maxDuration: const Duration(seconds: 11),
  preferredCameraDevice: CameraDevice.rear,
);
if (video == null) return;

// 2) 클라에서 프레임 N장 추출 (video_thumbnail)
//    ★ 앞부분에 몰지 않고 영상 "전 구간"에 걸쳐 시점을 벌려 뽑는다.
//      임무1(전반)·임무2(후반)을 각각 커버 → 포즈/각도 다양성 확보.
//    (image_picker 는 영상 길이를 돌려주지 않으므로 고정 ms 사용. 사용자가 일찍
//     멈춰 영상이 더 짧으면 video_thumbnail 이 마지막 프레임으로 클램프한다.)
const timesMs = [1500, 4500, 7500, 10000];   // 4장, 11초 전 구간 분산
final frames = <Uint8List>[];
for (final t in timesMs) {
  final data = await VideoThumbnail.thumbnailData(
    video: video.path,
    imageFormat: ImageFormat.JPEG,
    timeMs: t,
    maxWidth: 1024,
    quality: 85,
  );
  if (data != null) frames.add(data);   // 영상이 더 짧으면 마지막 프레임으로 클램프됨
}
// 3) 영상 bytes(임무 검증용) + 프레임 N장(저장용) + petId + challenge[] 전송
final videoBytes = await File(video.path).readAsBytes();   // 인라인 100MB 내
//   body: { petId, challenge, videoBase64, videoMime:'video/mp4', frames:[…] }
//   → enroll-pet-identity. 응답 후 video.path 임시파일은 즉시 삭제(로컬 잔존 방지).

// 4) 응답 처리: enrolled=true 면 등록 완료. 단 warnings 가 있으면 확인 안내(차단 아님)
//    예) ["species_kind","breed"] → "등록한 종류/품종과 영상이 달라 보여요. 맞나요?"
//    사용자가 '맞아요' 누르면 그대로 진행, '수정' 누르면 펫 정보 편집으로 이동.
```

패키지 추가(`pubspec.yaml`):

```yaml
image_picker: ^1.1.2      # 이미 있음 (pickVideo 지원)
video_thumbnail: ^0.5.3   # 영상→프레임 추출 (신규)
```

플랫폼 권한: 카메라(0018에서 추가) + (필요 시) 마이크. 영상이지만 소리는 불필요하므로 `pickVideo` 기본값 사용. `video_thumbnail` 은 추가 권한 없음.

> 프레임 수는 4장(정면/측면/다른 포즈가 섞이도록 영상 가이드 문구 제공: "반려동물이 잘 보이게 천천히 둘러 찍어주세요"). 원본 영상은 업로드/저장하지 않는다.

## 4. Edge Function `enroll-pet-identity` (pmdb)

위치: `supabase/functions/enroll-pet-identity/index.ts`. 배포·JWT 패턴은 0017/0018과 동일(`--no-verify-jwt` + `JWT_SECRET` 수동검증).

요청:

```jsonc
POST /functions/v1/enroll-pet-identity
Authorization: Bearer <login JWT>
{
  "petId": "…",
  "challenge": ["pat_head","hold_paw"],   // 앱이 무작위로 고른 임무 코드
  "videoBase64": "<mp4 base64>",          // 임무 검증용(저장 안 함)
  "videoMime": "video/mp4",
  "frames": ["<jpeg base64>", "…"],       // 저장용 N장(클라 추출)
  "mimeType": "image/jpeg"
}
```

> 등록정보(동물종·품종)는 **서버가 `pets` 행에서 직접 읽어** 대조한다(클라가 보낸 값을 신뢰하지 않음). 클라는 `petId`만 주면 된다.

응답:

```jsonc
// 성공(불일치 없음 또는 색만 다름)
{ "enrolled": true, "species": "dog", "breed": "말티즈", "colors": ["white"],
  "frameCount": 4, "challengePassed": true, "frames": ["https://…/0.jpg","…"],
  "infoMatch": { "species_kind": true, "breed": true, "color": true },
  "warnings": [] }
// 성공이지만 등록정보와 달라 보임 → 등록은 통과, 앱에서 확인 유도
{ "enrolled": true, "species": "cat", "breed": "코리안숏헤어",
  "infoMatch": { "species_kind": false, "breed": false, "color": false },
  "warnings": ["species_kind","breed"],   // color 는 경고에 안 넣음(정보성)
  "frames": ["…"] }
{ "enrolled": false, "reason": "not_guardian" }
{ "enrolled": false, "reason": "not_real_pet" }
{ "enrolled": false, "reason": "not_consistent_pet" }   // 영상 내 다른 개체로 보임
{ "enrolled": false, "reason": "challenge_failed",      // 지시 임무 미수행
  "missing": ["hold_paw"] }
```

> **불일치 정책:** `species_kind`(강아지/고양이)·`breed`(품종) 불일치는 **소프트 경고**(`warnings[]`)일 뿐 **등록을 막지 않는다.** `color`(털색)는 대조해 `infoMatch.color` 로 기록만 하고 경고에도 넣지 않는다(불일치 무관). 명백한 사기(강아지로 등록 후 고양이 영상 등)는 앱이 `warnings` 를 받아 "등록정보와 달라 보여요. 확인해 주세요" 로 사용자에게 확인을 받는다.

로직 스켈레톤:

```ts
const uid = await getUidFromJwt(req);                  // 0017과 동일
const { petId, challenge = [], videoBase64, videoMime = "video/mp4",
        frames, mimeType = "image/jpeg" } = await req.json();
const admin = serviceClient(SUPABASE_URL, SERVICE_ROLE);

// 1) 보호자 확인 (app.is_pet_guardian 은 app.uid() 기반이라 service_role 컨텍스트에선
//    uid 를 직접 비교하는 SQL 로 확인)
const { data: g } = await admin.from("pet_guardians")
  .select("id").eq("pet_id", petId).eq("user_id", uid).maybeSingle();
if (!g) return json({ enrolled: false, reason: "not_guardian" }, 403);

// 1-1) 등록정보(대조 기준)를 pets 행에서 직접 읽는다(클라 입력 불신).
//      species_kind='dog'|'cat'(신규, CHECK), species=품종 자유텍스트(기존).
const { data: pet } = await admin.from("pets")
  .select("species_kind, species").eq("id", petId).maybeSingle();
const regType  = (pet?.species_kind ?? "").toLowerCase();  // 'dog' | 'cat'
const regBreed = (pet?.species ?? "").trim();              // 자유 입력 품종

if (!videoBase64) return json({ enrolled: false, reason: "no_video" }, 400);
if (!Array.isArray(frames) || frames.length < 3)
  return json({ enrolled: false, reason: "too_few_frames" }, 400);
if (!Array.isArray(challenge) || challenge.length < 1)
  return json({ enrolled: false, reason: "no_challenge" }, 400);

// 2) Gemini(영상 입력): 실사 + 동일개체 일관성 + 지시 임무 수행
//    영상은 인라인 전송(2026.1부터 100MB까지). 우리 Storage 엔 절대 쓰지 않는다.
const ai = await verifyEnrollmentVideo(videoBase64, videoMime, challenge);   // §6-1
const real = Math.max(ai.dog_real, ai.cat_real);
const fake = Math.max(ai.dog_fake, ai.cat_fake);
if (!(real >= ENROLL_REAL_THRESHOLD && real > fake))
  return json({ enrolled: false, reason: "not_real_pet", ai });
if (!ai.consistent)
  return json({ enrolled: false, reason: "not_consistent_pet", ai });
// 지시한 임무가 모두 수행됐는지 (AI 가 challenge 별 수행여부 반환)
const missing = challenge.filter((c: string) => !ai.challenges_done?.includes(c));
if (missing.length > 0)
  return json({ enrolled: false, reason: "challenge_failed", missing, ai });

const species = ai.dog_real >= ai.cat_real ? "dog" : "cat";

// 2-1) 등록정보 교차검증 (소프트 — 통과 여부에 영향 없음)
//   · species_kind: AI 판별 종 vs 등록 동물종 (강한 신호, 그래도 경고만)
//   · breed: 자유 텍스트라 부분 포함/유사 매칭으로 관대하게 비교
//   · color: 기록만, 경고 미포함(불일치 무관)
const aiBreed  = (ai.detected_breed ?? "").trim();
const aiColors = ai.coat_colors ?? [];
const typeOk  = !regType  || regType === species;                  // 등록값 없으면 통과 처리
const breedOk = !regBreed || looseBreedMatch(regBreed, aiBreed);   // 느슨 비교(§6-1 주석)
const colorOk = colorOverlap(regBreed, aiColors);                  // 참고용(등록 색 필드 없음)
const warnings: string[] = [];
if (!typeOk)  warnings.push("species_kind");
if (!breedOk) warnings.push("breed");
const infoMatch = { species_kind: typeOk, breed: breedOk, color: colorOk };
// ★ 이 시점 이후 videoBase64 는 더 이상 쓰지 않는다(저장 호출 없음 → 영상 소멸).

// 3) 통과: 프레임 업로드 + DB 반영 (RPC, definer). 경고가 있어도 등록은 진행.
const urls: string[] = [];
for (let i = 0; i < frames.length; i++) {
  const path = `${uid}/pet_identity/${petId}/${i}.jpg`;
  const bytes = Uint8Array.from(atob(frames[i]), c => c.charCodeAt(0));
  await admin.storage.from("media").uploadBinary(path, bytes,
        { contentType: mimeType, upsert: true });
  urls.push(admin.storage.from("media").getPublicUrl(path).data.publicUrl);
}
await admin.rpc("enroll_pet_identity", {
  p_pet: petId, p_species: species, p_breed: aiBreed, p_colors: aiColors,
  p_info_match: { ...infoMatch, warnings },
  p_paths: urls.map((_, i) => `${uid}/pet_identity/${petId}/${i}.jpg`),
  p_urls: urls,
});
return json({ enrolled: true, species, breed: aiBreed, colors: aiColors,
  frameCount: urls.length, frames: urls, infoMatch, warnings });
```

시크릿: `GEMINI_API_KEY`, `JWT_SECRET`, `SUPABASE_SERVICE_ROLE_KEY`, `SUPABASE_URL` (0018과 공유).

> **재등록**: 같은 `petId` 재인증 시 `upsert:true` 로 프레임을 덮어쓰고 RPC 가 기존 행을 교체한다(성장·외형 변화 대응).

## 5. DB 반영 RPC (security definer)

```sql
-- migration: 2026XXXX_enroll_pet_identity_rpc.sql
create or replace function app.enroll_pet_identity(
  p_pet uuid, p_species varchar, p_paths text[], p_urls text[],
  p_breed varchar default null, p_colors text[] default null,
  p_info_match jsonb default null
) returns void
language plpgsql security definer set search_path = ''
as $$
begin
  delete from public.pet_identity_frames where pet_id = p_pet;     -- 재등록 시 교체
  insert into public.pet_identity_frames (pet_id, frame_index, image_url, image_path)
    select p_pet, i-1, p_urls[i], p_paths[i]
      from generate_subscripts(p_urls, 1) as i;
  update public.pets
     set identity_verified = true, identity_verified_at = now(),
         ai_species = p_species, ai_breed = p_breed, ai_colors = p_colors,
         info_match = p_info_match, updated_at = now()
   where id = p_pet;
end;
$$;
revoke all on function app.enroll_pet_identity(uuid,varchar,text[],text[],varchar,text[],jsonb)
  from public, anon, authenticated;
```

## 6. Gemini 호출 상세

### 6-1. 등록 영상 판별 `verifyEnrollmentVideo(videoBase64, challenge[])`

**영상 1개**를 인라인으로 넣고 "실사 + 동일개체 + 지시 임무 수행"을 한 번에 묻는다. 동작(쓰다듬기/잡기)은 시간축 움직임이라 **영상이어야** 판별 가능하다.

```ts
const ENROLL_SCHEMA = {
  type: "object",
  properties: {
    species:    { type: "string", enum: ["dog","cat","other","none"] },
    dog_real:{type:"number"}, cat_real:{type:"number"},
    dog_fake:{type:"number"}, cat_fake:{type:"number"},
    consistent:     { type: "boolean" },   // 영상 내내 동일 개체로 보이는가
    challenges_done:{ type: "array", items: { type: "string" } },  // 수행 확인된 임무 코드
    detected_breed: { type: "string" },     // 추정 품종(한국어, 예: "말티즈"·"믹스")
    coat_colors:    { type: "array", items: { type: "string" } },  // 주요 털색(예: ["white","tan"])
    reason:     { type: "string" },
  },
  required: ["species","dog_real","cat_real","dog_fake","cat_fake",
             "consistent","challenges_done","detected_breed","coat_colors","reason"],
};

// challenge 코드 → AI 설명 매핑(프롬프트에 주입)
const CHALLENGE_DESC: Record<string,string> = {
  pat_head:    "사람 손이 반려동물의 머리를 쓰다듬는다",
  hold_paw:    "사람 손이 반려동물의 앞발을 잡는다",
  scratch_chin:"사람 손이 반려동물의 턱·목을 만진다",
  stroke_back: "사람 손이 반려동물의 등을 쓰다듬는다",
  hand_in_frame:"사람 손과 반려동물이 같은 화면에 함께 있다",
};

// parts = [ {text: ENROLL_PROMPT}, {inline_data:{mime_type:videoMime, data:videoBase64}} ]
```

`ENROLL_PROMPT`(요지): "이 영상은 반려동물 등록 인증용이다. 판단하라:
(1) 실제 살아있는 개/고양이인가(화면 재촬영·인쇄물·일러스트·인형·AI 생성은 fake),
(2) 영상 내내 같은 개체인가,
(3) 다음 지시 동작이 영상에서 **실제로 수행**됐는가 — 수행된 코드만 `challenges_done` 에 담아라:
`{challenge.map(c => `${c}=${CHALLENGE_DESC[c]}`).join('; ')}`,
(4) (참고용) 추정 품종 `detected_breed` 와 주요 털색 `coat_colors` 를 적어라 — 확실치 않으면 '믹스'/근사값으로.
신뢰도 0~1, reason 한국어 80자."

> **(4)는 등록정보 대조용 서술값일 뿐 통과/탈락 기준이 아니다.** 서버가 이 값을 `pets`의 등록 동물종·품종과 비교해 소프트 경고를 만들 뿐, 등록을 막지 않는다(품종·색 오판이 흔하므로). 비교 로직(서버):
> - `looseBreedMatch(reg, ai)`: 양쪽 소문자화 후 부분 포함(`includes`)·대표 동의어(시고르자브종=믹스 등) 일부만 봐주는 **관대한** 비교. 애매하면 일치로 처리(오경고 최소화).
> - `colorOverlap(reg, aiColors)`: 등록 털색 필드가 없으므로 현재는 참고값(항상 true 처리 가능). 향후 등록 색 필드가 생기면 교집합 비교.
>
> Gemini 2.5 Pro 는 영상을 기본 1fps 로 샘플링한다. 쓰다듬기 같은 느린 동작은 충분히 포착되나, 매우 빠른 동작은 놓칠 수 있으니 임무 풀은 "천천히 해도 되는 접촉 동작"으로 둔다. 영상은 검증 직후 폐기(저장 없음).

### 6-2. 게시 매칭 판별 `matchIdentity(refFrames[], shot)`

기준 프레임 N장 + 게시 사진 1장을 함께 넣고 동일 개체 여부를 묻는다.

```ts
const MATCH_SCHEMA = {
  type: "object",
  properties: {
    same_individual: { type: "boolean" },
    identity_score:  { type: "number" },   // 같은 개체일 신뢰도 0~1
    is_real:         { type: "boolean" },   // 게시 사진이 실사(라이브)인가
    dog_real:{type:"number"}, cat_real:{type:"number"},
    dog_fake:{type:"number"}, cat_fake:{type:"number"},
    reason:          { type: "string" },
  },
  required: ["same_individual","identity_score","is_real",
             "dog_real","cat_real","dog_fake","cat_fake","reason"],
};
```

`MATCH_PROMPT`(요지): "앞의 N장은 등록된 반려동물 A의 기준 사진이다. 마지막 1장이 **같은 개체 A**인지 판정하라. 각도·조명·성장에 따른 차이는 허용하되, 품종·무늬·색·체형으로 종합 판단해 identity_score(0~1)를 매겨라. 동시에 마지막 사진이 화면 재촬영/인쇄/AI 생성이 아닌 실사인지(is_real) 판단하라. reason 한국어 80자."

- parts 순서: `[{text:MATCH_PROMPT}, ...refFrames(inline_data), shot(inline_data)]`
- `generationConfig`: `responseMimeType:"application/json"`, `responseSchema:MATCH_SCHEMA`, `temperature:0`.
- 호출 실패/타임아웃 → `ai_unavailable`(통과 아님), 재시도 안내.

> 인라인 한도(요청 20MB)는 1024px·85% 프레임 5장+사진 1장이면 충분. 초과 시 해상도/장수 축소.

## 7. `verify-post-photo` 개정 (0018 → 동일개체 매칭)

0018 §5 함수에 **대상 펫 프레임 조회 + 매칭** 단계를 더한다. 요청에 `petId` 추가.

```jsonc
POST /functions/v1/verify-post-photo
{ "imageBase64":"…","mimeType":"image/jpeg",
  "lat":37.2,"lng":127.07,"accuracy":12.4,"isMocked":false,
  "petId":"…" }                       // ← 검증 대상(작성자가 선택한 펫)
```

추가 로직(0018 ①②③ 통과 후):

```ts
// ④ 대상 펫 기준 프레임
const { data: frames } = await admin
  .from("pet_identity_frames")
  .select("image_path")            // service_role: 비공개 경로로 직접 다운로드
  .eq("pet_id", petId).order("frame_index");
if (!frames?.length) return json({ pass:false, reason:"pet_not_enrolled" });
// (선택) 작성자가 이 펫의 보호자인지 확인
const refB64 = await downloadAsBase64(admin, frames.map(f => f.image_path));

// ⑤ Gemini 매칭
const m = await matchIdentity(refB64, imageBase64, mimeType);   // §6-2
const real = Math.max(m.dog_real, m.cat_real);
const fake = Math.max(m.dog_fake, m.cat_fake);
const livenessOk = m.is_real && real > fake;
const idOk = m.identity_score >= IDENTITY_PASS_THRESHOLD;       // 중간신뢰
if (!livenessOk) return failPhoto(admin, uid, petId, m, "not_real_pet");
if (!idOk)       return failPhoto(admin, uid, petId, m, "identity_mismatch");

// ⑥ 통과: 업로드 + 토큰(matched_pet_id, identity_score)
//   record_photo_verification 에 p_matched_pet, p_identity_score, p_same_individual 인자 추가
```

판정 정책(중간 신뢰):

```
PASS ⇔ identity_score >= IDENTITY_PASS_THRESHOLD (기본 0.63)
       AND is_real = true AND max(real) > max(fake)
```

> `IDENTITY_PASS_THRESHOLD` 는 **오거부↔위조** 트레이드오프의 핵심 튜닝값. "중간 신뢰" 기준 **초기 0.63**(다소 보수적). 운영 로그(`identity_score` 분포)로 ±0.05 조정.

## 8. 게시글 토큰 검사 보강 (트리거)

0018의 `tg_posts_check_write` 토큰 검사에 **매칭 펫이 글에 연결된 펫인지**를 더한다.

```sql
-- 0018 트리거의 사진 토큰 검사 블록에 추가:
--   v_pv.matched_pet_id 가 이 글의 post_pets 중 하나여야 한다.
if new.image_url is not null then
  -- … (0018) 토큰 유효성: 본인·pass·미만료·image_url 일치 …
  if v_pv.matched_pet_id is null
     or not exists (select 1 from public.post_pets pp
                    where pp.post_id = new.id and pp.pet_id = v_pv.matched_pet_id) then
    -- post_pets 는 posts 이후 insert 되므로, 단일 펫 검증 글은
    -- create_post_verified RPC 에서 펫 연결까지 한 트랜잭션으로 처리(아래 주의)
    raise exception 'posts: 검증된 펫과 게시글 펫이 일치하지 않습니다';
  end if;
  update public.photo_verifications set consumed_at = now() where id = v_pv.id;
  new.photo_verification_id := v_pv.id;
  new.ai_pet_species        := v_pv.ai_species;
  new.is_pet_verified       := true;
end if;
```

> **주의(트랜잭션 순서):** `post_pets` 는 `posts` INSERT 후에 들어간다. 따라서 BEFORE INSERT 트리거 시점에는 아직 `post_pets` 가 없다. 두 가지 해법 중 택1:
> 1. `create_post_verified` RPC(0018 §8)에 `p_target_pet` 인자를 받아 **토큰의 matched_pet_id == p_target_pet** 인지 RPC 본문에서 먼저 검증하고, posts→post_pets 를 같은 트랜잭션에서 insert. (권장 — 트리거에서 post_pets 의존 제거)
> 2. 트리거를 `AFTER INSERT` 보조 트리거로 분리해 post_pets 까지 본 뒤 검증. (복잡, 비권장)
>
> 권장안(1)에서는 위 트리거 블록 대신 **RPC가 `matched_pet_id` 일치를 검사**하고, 트리거는 0018 그대로(토큰 유효성+image_url 일치)만 둔다.

`create_post_verified` 확장(권장안):

```sql
-- p_target_pet uuid 인자 추가. 본문에서:
if p_photo_token is not null then
  select * into v_pv from public.photo_verifications
   where id = p_photo_token and user_id = v_uid and result='pass'
     and consumed_at is null and expires_at > now()
     and image_url = p_image_url;
  if not found then raise exception 'posts: 유효하지 않은 사진 검증'; end if;
  if v_pv.matched_pet_id is null or v_pv.matched_pet_id <> p_target_pet then
    raise exception 'posts: 검증된 펫과 선택한 펫이 다릅니다';
  end if;
end if;
-- posts insert → post_pets insert(p_target_pet 포함) → 토큰 consumed 처리
```

## 9. 임계값·정책 정리

| 값 | 기본 | 의미 |
| --- | --- | --- |
| `ENROLL_REAL_THRESHOLD` | 0.70 | 등록 프레임이 실제 개/고양이일 신뢰도 하한 |
| 등록 프레임 수 N | **4** | 다양한 포즈 확보 |
| 임무(challenge) 개수 | **2개** | 등록 영상에서 무작위 지시(추후 보강) |
| 임무 통과 기준 | **지시한 임무 전부 수행** | `missing[]` 비어야 통과 |
| 영상 maxDuration | **11초** | 임무 2개 순차 + 전 구간 분산 샘플링(인라인 한도 여유) |
| 프레임 샘플 시점 | **[1.5,4.5,7.5,10]s** | 4장, 전 구간 분산(앞 몰림 방지) |
| 기준 프레임 버킷 | **공개 `media`** | 도용 동기 낮음 판단(별도 private 버킷 미사용) |
| 등록정보 교차검증 | **종·품종=소프트 경고 / 털색=기록만** | 불일치해도 등록 차단 안 함 |
| `IDENTITY_PASS_THRESHOLD` | **0.63** | 게시 사진=등록 펫 동일개체 통과선(중간신뢰, 약간 보수적) |
| 라이브니스 | `is_real && real>fake` | 화면 재촬영·AI 생성 차단 |
| 토큰 TTL | 15분 (0018) | 검증→게시 유효시간 |
| 지역 만료 | 0017 정책과 일치 | 활동지역 재인증 주기 |

### 9-1. 정책값 선택지별 트레이드오프 (확정 전 검토)

각 값은 **보안(악용 차단)** 과 **사용성(정상 사용자 오거부·번거로움)** 사이의 저울이다. 한쪽으로 치우칠 때 생기는 문제를 정리한다.

| 값 | 낮게/적게 잡으면 문제 | 높게/많게 잡으면 문제 | 권장 |
| --- | --- | --- | --- |
| `IDENTITY_PASS_THRESHOLD` (동일개체 통과선) | **너무 관대** → 외형 비슷한 남의 펫·다른 펫도 통과(재식별 본래 목적 약화). 0.4 이하면 사실상 종(種)만 맞으면 통과 | **너무 엄격** → 같은 펫인데 각도·조명·미용 차이로 거부(정상 사용자 오거부↑, 문의↑). 0.75 이상이면 재촬영 반복 | **0.63 확정**(중간신뢰 중 약간 보수적). 운영 로그로 ±0.05 조정 |
| 등록 프레임 수 N | **적으면(1~2장)** 한 각도만 저장 → 게시 사진 각도 다르면 매칭 실패↑ | **많으면(8장+)** Gemini 입력·비용·지연↑, 저장 용량↑, 비교 노이즈↑ | **4장 확정**(전 구간 분산) |
| 영상 maxDuration | **짧으면(2~3초)** 임무 동작이 안 잡혀 `challenge_failed`↑, 프레임 다양성↓ | **길면(15초+)** 파일 100MB 인라인 한도 위험, 업로드·AI 지연↑, 사용자 인내심↓ | **11초 확정**(임무 2개 순차 + 전 구간 분산) |
| 임무(challenge) 개수 | **0~1개** → 라이브니스 약함. 운 좋게 사전 영상이 우연히 임무와 일치할 확률↑ | **3개+** → 짧은 영상에 다 담기 어려움, 노약자·활동량 적은 펫에 부담, 실패율↑ | **2개 확정**(추후 풀 보강) |
| 프레임 추출 방식 | **앞부분 몰림** → 한 포즈만 저장돼 게시 사진 각도 다르면 매칭 실패↑ | **미션별 영상 2회 촬영** → 다양성은 좋으나 업로드·AI 호출 2배, 마찰↑ | **단일 영상 전 구간 분산 확정**(호출 1회로 다양성 확보) |
| `ENROLL_REAL_THRESHOLD` (등록 실제펫 하한) | **낮으면** 인형·그림·화면 속 펫도 기준으로 등록될 위험 | **높으면** 조명 나쁜 정상 영상도 거부 | **0.70** |
| 토큰 TTL | **짧으면(1분)** 촬영→작성 중 만료되어 재검증 강제(불편) | **길면(수시간)** 탈취 토큰 재사용 창 넓어짐 | **15분**(0018과 동일) |
| 기준 프레임 저장 버킷 | **공개 `media` 재사용** → 구현 간단하나 URL 알면 펫 외형 프레임 열람 가능 | **private `pet_identity`** → 안전하나 서명URL/권한 처리 추가 구현 | **공개 `media` 확정**(도용 동기·민감도 낮다고 판단) |
| 펫 없는 글(자유게시판 등) 사진 정책 | **검증 면제** → 무관 사진 업로드 가능(설계상 의도) | **펫 선택 강제** → 펫 없는 정상 글 작성 막힘(사용성↓) | **고의 면제 확정**(자유게시판이 있어 무관 사진 허용. 펫 연결 글에만 매칭 적용) |

> 핵심 원칙: **거짓 통과(보안 사고)는 신고·관리자 검토로 사후 보완 가능하지만, 거짓 거부(정상 사용자 차단)는 곧바로 이탈로 이어진다.** 그래서 매칭·임무는 "중간 신뢰·관대" 쪽으로, 대신 카메라 강제+지역 일치+1회용 토큰+무작위 임무의 **다중 방어 결합**으로 전체 보안을 확보하는 설계다.

## 10. 보안 / 위변조 방어

| 위협 | 방어 |
| --- | --- |
| 남의 펫 사진으로 게시 | 등록된 **기준 프레임과 동일 개체** 매칭 필요 |
| 등록 영상에 가짜(화면/AI) 펫 | 등록 시 라이브니스 + 프레임 간 일관성 검사 |
| 미리 찍어둔 영상·화면 재생으로 등록 | **무작위 동작 임무(challenge) — 능동형 라이브니스.** 임무가 매번 무작위라 사전 제작 영상으론 통과 불가 |
| 사진 1장을 영상인 척 제출 | 임무는 "사람 손↔실제 펫 상호작용" 동작이라 정지 이미지로 수행 불가 |
| 등록정보와 다른 동물로 등록 | AI 가 종/품종/털색 추출 → 등록값과 대조, 불일치 시 **소프트 경고**(사기 의심 신호로 신고·관리자 검토 연계 가능) |
| 게시 사진을 등록 펫 사진의 화면 재촬영 | 게시 라이브니스(`is_real`) 차단 |
| 갤러리 옛 사진 | 카메라 강제(0018) + 서버가 본 바이트만 업로드 |
| 검증 우회 직접 INSERT | 서버 토큰 + 트리거/RPC matched_pet 검사 |
| 토큰 재사용 | `consumed_at` 1회 소진 + TTL + user_id |
| 다른 동네 촬영 | 촬영지 행정동==users.region_code (0017/0018) |
| 기준 프레임 컬럼 위조 | `pet_identity_frames`·`pets.identity_verified` 는 definer/RPC 만 기록 |
| Gemini 키 노출 | 키는 Edge Function 시크릿, 클라 직접 호출 없음 |
| 원본 영상 유출 | **영상 미저장.** AI 인라인 전송 후 요청 메모리에서 소멸, Storage·DB 어디에도 기록 안 함(프레임만 저장) |

> 한계(정직하게): Gemini는 **전용 재식별(re-ID) 모델이 아니다.** 외형이 매우 비슷한 동종(검은 시고르자브종 등)은 오인 가능. 그래서 "중간 신뢰 + 지역 일치 + 카메라 강제 + 1회용 토큰"의 다중 결합으로 억제하며, 완전 차단이 아니라 **비용 대비 충분한 억제**가 목표다.

## 11. 엣지 케이스

- **미등록 펫으로 게시**: `pet_not_enrolled` → 먼저 신원 인증 유도.
- **성장/미용으로 외형 변화**: `identity_score` 하락 → 재등록(§4 upsert)로 기준 갱신.
- **매우 유사한 동종**: 중간신뢰라 통과될 수 있음(설계상 허용). 악용 의심 시 신고/관리자 검토 경로로 보완.
- **품종/털색 오판(믹스견·조명)**: AI 가 등록값과 다르게 추정 → `warnings` 로 소프트 경고만, 등록은 통과. 사용자 확인으로 마무리(하드 차단 안 함).
- **동물종(강아지/고양이) 불일치**: 가장 강한 신호지만 정책상 소프트 경고. 앱이 눈에 띄게 확인을 받고, 반복/명백 시 관리자 검토 대상으로 표시 권장.
- **다견/다묘 가정**: 펫별로 각각 등록·매칭. 게시 시 **대상 펫 1마리** 선택 필수.
- **임무 미수행/일부만 수행**: `challenge_failed`(+`missing[]` 어떤 임무를 못했는지) → 동일/새 임무로 재촬영. 무한 시도 방지용 횟수 제한 권장.
- **영상이 너무 짧음(임무 동작 안 잡힘)**: 최소 길이(예: 3초) 미만 → 재촬영 안내.
- **영상이 너무 큼(인라인 100MB 초과)**: maxDuration·해상도 제한으로 사전 방지. 그래도 초과 시 `video_too_large` → 재촬영(또는 Files API 경로는 향후 과제).
- **프레임 추출 실패(짧은 영상/코덱)**: 최소 3장 미만 → 재촬영 안내.
- **Gemini 지연/오류**: `ai_unavailable` → 재시도(통과로 처리 금지).
- **펫 없는 카테고리(자유게시판 등) 사진**: **고의 면제(확정).** 자유게시판 카테고리가 있어 무관 사진 업로드를 허용한다. 사진 검증·동일개체 매칭은 **펫을 연결한 글에만** 적용한다.
