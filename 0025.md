# 업체(사업자) 계정 — 가입 플로우 개편 · 국세청 검증 · facilities 대조 자동승인 · 관리자 승인

> 상태: 설계(미구현). 작성 2026-07-14. 3차 개정(같은 날): 외부 리뷰 2건 반영 — 점수 규칙 테이블화(무마이그레이션 튜닝), 정규화 알고리즘 명세, 물리 업소 단위 dedup(겸업 복수 행 모순 해소), 자동승인 필수 조건 AND화(업종 우회 봉쇄), 관리자 override, 데이터 갱신 정책, 구버전 앱 호환, 서류 파기 정책, 예외 처리 케이스표.
> 범위: 가입 시 유형 선택 화면 재배치, 업체 등록(사업자번호 국세청 자동확인 + 등록증 업로드 + facilities 공공데이터 대조 점수제 자동승인 + 관리자 승인), 하나의 계정에서 일반↔업체 프로필 전환.
> 관련: pmdart `signup_phone_screen.dart`(가입 5단계), `profile_edit_screen.dart`(내정보 수정), pmdb `signup` Edge Function · admin RPC 패턴, **0021(시설 지도 — `facilities` 테이블·LOCALDATA 적재)**, pmlegal(수집항목·보존기간 추가 시 약관·처리방침 동시 수정).

## 1. 검토 결과 — 현행 구조와의 충돌점

### 1.1 `user_type='business'`는 계정 전환 정책과 충돌한다 (핵심)

현행 `users.user_type`은 단일값(`pet_owner`/`no_pet`/`business`/`admin`, CHECK `users_user_type_check`)이다. "업체이면서 펫 보유자"인 계정을 표현할 수 없으므로, **업체 여부를 `user_type`에서 분리**해야 한다.

- **결정**: 업체 정보는 신규 테이블 **`business_profiles`**(users와 1:1)로 분리. `user_type`은 펫 보유 축(`no_pet`/`pet_owner`)과 `admin`만 담당.
- 신규 가입의 `user_type`은 항상 `no_pet`으로 시작(업체등록 선택자도 동일). CHECK의 `'business'` 값은 기존 계정 호환을 위해 당분간 유지.
- **기존 `user_type='business'` 계정 처리**: 적용 전 건수 확인(`select count(*) from users where user_type='business'`). 이들은 `no_pet`으로 백필하고, 앱에서 "업체 인증을 완료해 주세요" 안내 후 새 등록 플로우로 유도(자동 승인 아님 — 등록증이 없으므로).

### 1.2 사업자등록증 업로드는 가입 "완료 후"에만 가능하다

Storage RLS(`media owner insert`)가 `storage.foldername(name)[1] = app.uid()::text`를 요구하므로 **로그인 상태여야 업로드 가능**. 따라서 가입 단계 안에서 등록증을 받을 수 없고, 다음 순서가 강제된다:

```
가입 완료 → 자동 로그인 → 업체등록 화면(사업자번호·국세청확인·업종·상호·주소·전화·이메일·등록증)
```

이는 기존 펫 등록 패턴(가입 → 자동 로그인 → `PetEditScreen(draft:)`)과 동일한 구조라 자연스럽다. 업체등록 화면을 **하나 만들어 가입 직후와 내정보 수정 버튼 양쪽에서 재사용**한다. 가입 직후 이탈해도 계정은 정상 생성되며, 내정보 수정에서 이어서 신청하면 된다.

### 1.3 유형 선택을 "가장 앞"에 두되, 약관 동의는 그보다 앞이어야 한다

법적으로 개인정보 수집 전 동의가 선행돼야 하므로 0단계(약관 동의)는 유지. 유형 선택은 **그 직후 1단계**로 이동한다.

**새 가입 플로우:**

| 단계 | 화면 | 비고 |
| --- | --- | --- |
| 0 | 약관 동의 | 현행 유지 |
| 1 | **유형 선택: 일반사용자 / 업체등록** | 현행 4단계(5/5)의 라디오 3개 → 2개로. 펫 보유 라디오 삭제 |
| 2 | 전화번호 | 현행 1단계 |
| 3 | 인증코드 | 현행 2단계 |
| 4 | 아이디·비밀번호·닉네임 | 현행 3단계 |
| 5 | (일반) 펫 정보 입력 — **선택** | 현행 4단계의 펫 필드만 남김. 건너뛰기 가능 |
| 후속 | (업체) 자동 로그인 후 **업체등록 화면** | §1.2 |

- `signup` Edge Function으로 보내는 `user_type`은 항상 `'no_pet'`(펫 드래프트가 있어도 — 정식 등록은 AI 신원인증 통과 후이며, 통과 시 기존 트리거가 승격).
- 주의(현행 확인 사항): 승격 트리거는 **승격만** 있고 펫 전부 삭제 시 강등은 없다. 현행 유지(강등 안 함)로 두되 별도 논의 대상으로 남김.

### 1.4 공개 `media` 버킷은 사업자등록증에 부적합하다

`media`는 public 버킷이라 URL만 알면 누구나 열람 가능. 사업자등록증은 대표자명·주소 등 개인정보 포함 문서이므로 **비공개 버킷 신설**이 필요하다(§3.3).

### 1.5 대조 데이터(`facilities`)의 성격과 한계 — 자동승인 설계의 전제

0021에서 적재한 `public.facilities`(약 24,550건)가 대조 원본이다. 컬럼: `category`(enum: `animal_hospital`/`grooming`/`pet_hotel`/`pet_cafe`/`pet_sales`), `name`(사업장명), `address`, `phone`, `biz_status`/`is_open`, `license_date`, `region_code`, `geom`. 설계 전 반드시 인지할 한계:

1. **사업자등록번호가 없다.** LOCALDATA 인허가 데이터에는 사업자번호가 포함되지 않으므로 **정확 조인 불가, 상호·주소·전화·업종의 퍼지 매칭만 가능**하다. 대조는 "이 신청이 실존 인허가 업소와 부합하는가"를 보는 것이지 "이 사업자번호가 그 업소의 것"임을 증명하지 못한다.
2. **전화번호 결측이 많다**(LOCALDATA `소재지전화`는 nullable). 전화 없는 업소는 자동승인 필수 조건(§4.4)에 도달할 수 없어 관리자 검토로 흐른다(의도된 동작).
3. **`address`는 지번 주소다**(LOCALDATA `소재지전체주소`). 신청자가 등록증의 도로명 주소를 적으면 문자열 유사도가 거의 안 나온다 → 주소 입력은 **주소검색(juso) 위젯**으로 도로명·지번·행정코드를 함께 확보하고(§4.2), 재적재 시 `road_address`(도로명전체주소) 컬럼을 추가한다(§8). 그 전까지 주소 유사도 규칙은 `enabled=false`로 꺼둘 수 있다(§2.5).
4. **갱신 지연**: 스냅샷 데이터라 신규 개업·상호 변경·폐업이 반영 안 됨 → 이전 상호명 입력란(§4.2)·신규개업 트랙(§4.5)·월 1회 재적재(§8)로 흡수.
5. **겸업은 복수 행**: 같은 업체(이름+주소 동일)가 카테고리별 여러 행(병원+미용+위탁). 행 단위로 다루면 "겸업 업소는 항상 동점 복수 → 영구 관리자 검토"가 되고, 시설 유니크도 카테고리 행을 바꿔 우회된다. → **후보·유니크·업종 판정 전부 물리 업소 단위(`biz_key` = 정규화 상호+주소, §4.3)로 처리**한다. `facility_all_categories()`와 동일한 관점.
6. **`pet_cafe`는 대조 대상 제외**(사용자 정책 확정): 카페는 인허가 데이터가 아니라 네이버 검색 기반이라 신뢰도가 다르다. 카페·기타 업종 신청은 신규개업 트랙(§4.5)과 동일하게 관리자 검토로.
7. **상호명 입력 기준 불일치**: 등록증의 "상호"(예: ㈜OO컴퍼니)와 인허가 데이터의 "사업장명"(간판명, 예: OO동물병원)이 다를 수 있다 → **사업장명(간판명)을 별도 입력란**으로 받아 현 상호·사업장명·이전 상호 중 최고 유사도로 대조한다(§4.2~4.3).

### 1.6 ⚠️ 자동승인의 잔여 위험 — 대조 원본이 "공개" 데이터라는 점

`facilities`는 공개 데이터이고 **우리 앱 지도에서도 업체명·주소·전화가 그대로 노출**된다. 즉 악의적 신청자가 남의 실존 업소 정보를 베껴 적고, 자신의(무관한) 유효 사업자번호로 국세청 01을 통과하면 **점수제만으로는 원리적으로 뚫릴 수 있다**. 이전 상호명 입력란 역시 검증 불가 자유 텍스트라 점수 부풀리기 벡터가 될 수 있다(같은 범주의 잔여 위험). 완화책(§4.6): 자동승인 필수 조건(전화·업종) + 등록증 원본 보관·표본 감사 + 자동승인 감사로그·사후 모니터링 + 신고 시 재검토·즉시 강등. 그래도 잔여 위험은 0이 아니므로, **초기 물량이 적을 때는 전건 관리자 검토로 운영하다가 물량 증가 시 자동승인을 켜는 운영 스위치**(`business_match_rules`의 `auto_approve_enabled`, §2.5)를 둔다.

### 1.7 구버전 앱 호환 — `signup`에서 `'business'`를 즉시 거부하면 기존 배포 앱이 깨진다

현행 배포 앱의 가입 화면은 라디오 3개로 `user_type='business'`를 보낼 수 있고, 서버 검증(`signup/index.ts:71-73`)에서 제거해 버리면 미업데이트 앱의 업체 가입이 마지막 단계에서 400으로 실패한다. **전환기 정책: 서버는 `'business'`를 받으면 `'no_pet'`으로 매핑해 저장**(§1.1 백필 방침과 일관). 신규 앱 배포가 충분히 확산된 뒤(강제 업데이트 이후) 검증에서 제거한다.

## 2. DB 설계

### 2.1 `business_profiles` (신규, public 스키마)

```sql
create table public.business_profiles (
  user_id             uuid primary key references public.users(id) on delete cascade,
  business_reg_no     varchar(10) not null
                        check (business_reg_no ~ '^\d{10}$'),   -- 숫자 10자리로 정규화 저장
  declared_category   varchar(20) not null                      -- 신청 업종(§4.2)
                        check (declared_category in
                          ('pet_sales','pet_hotel','animal_hospital','grooming','other')),
  business_name       text not null,                            -- 상호명(등록증 기준)
  storefront_name     text,                                     -- 사업장명(간판명) — 인허가 데이터 대조용(§1.5-7)
  prev_business_name  text,                                     -- 상호 변경 시 이전 상호명(§4.2)
  business_address    text not null,                            -- 도로명 주소(juso 검색 결과)
  business_address_jibun text,                                  -- 지번 주소(juso 검색 결과 — facilities 대조용)
  business_region_code varchar(20),                             -- 행정코드(juso 검색 결과)
  business_phone      varchar(40),                              -- 사업장 전화(대조용)
  representative_name text,                                     -- 대표자명(관리자 대조용, 선택)
  contact_email       text not null
                        check (contact_email ~* '^[^@\s]+@[^@\s]+\.[^@\s]+$'),  -- 인증 없음, 형식만
  license_image_path  text not null,        -- business-docs 버킷 내 사업자등록증 경로
  extra_doc_path      text,                 -- 신규개업 트랙 추가 서류(§4.5) 경로
  nts_status_code     varchar(2),           -- 신청 시점 국세청 b_stt_cd (01 계속/02 휴업/03 폐업)
  nts_checked_at      timestamptz,
  -- ▼ facilities 대조 결과(§4) — 전부 서버가 기록, 클라이언트 위조 불가
  matched_facility_id uuid references public.facilities(id) on delete set null,  -- 대표 행(참고용)
  matched_biz_key     text,                 -- 물리 업소 키(정규화 상호|주소, §4.3) — 유니크 단위
  match_score         integer,
  match_detail        jsonb,                -- 규칙별 점수·유사도·후보 수·적용 가중치 스냅샷(감사용)
  review_track        varchar not null default 'review'
                        check (review_track in ('auto','review','new_business')),
  auto_approved       boolean not null default false,
  review_note         text,                 -- 관리자 메모·override 사유(§6)
  status              varchar not null default 'pending'
                        check (status in ('pending','approved','rejected')),
  rejected_reason     text,
  reviewed_by         uuid references public.users(id),
  reviewed_at         timestamptz,
  created_at          timestamptz not null default now(),
  updated_at          timestamptz not null default now()
);
```

- `user_id`가 PK → **계정당 업체 1개**(다점포/복수 사업자는 범위 밖, 필요 시 추후 별도 논의).
- 재신청은 반려된 **같은 행을 pending으로 되돌려 갱신**(INSERT 아님) → 이력은 `admin_logs`로 추적.
- `matched_facility_id`는 표시용 대표 행(신청 업종과 같은 카테고리 행 우선)이고, **중복 판정 기준은 `matched_biz_key`**(§2.2). FK는 `on delete set null` — 0021 동기화는 upsert-only라 당장 삭제는 없지만 수동 정리 대비.

### 2.2 유니크 — 사업자번호 + 물리 업소 키 (부분 유니크 인덱스)

단순 `UNIQUE(business_reg_no)`는 "A계정이 반려된 번호를 B계정이 정당하게 신청"하는 경우까지 막는다. **pending/approved 상태만 유일**하도록 부분 인덱스로:

```sql
create unique index business_profiles_regno_active_uq
  on public.business_profiles (business_reg_no)
  where status in ('pending','approved');

-- 실존 업소 1곳 = 활성 계정 1개. 행(facility_id)이 아니라 물리 업소 키 기준 —
-- 겸업 업소의 병원 행/미용 행을 서로 다른 계정이 각각 잡는 우회를 차단(§1.5-5).
create unique index business_profiles_bizkey_active_uq
  on public.business_profiles (matched_biz_key)
  where matched_biz_key is not null and status in ('pending','approved');
```

- 탈퇴(soft delete) 시 번호·업소 키가 잠기지 않도록 `withdraw_account` RPC에 `business_profiles` 행 **status='rejected' 처리(또는 삭제) + 서류 파기 흐름(§8)** 추가 — 잊으면 폐업체 번호가 영구 점유됨.

### 2.3 계정 전환 — `users.active_mode`

```sql
alter table public.users
  add column active_mode varchar not null default 'personal'
    check (active_mode in ('personal','business'));
```

- 전환은 definer RPC **`switch_account_mode(p_mode)`** 로만: `business` 전환은 `business_profiles.status='approved'`일 때만 허용(아니면 P0001 `business_not_approved`). 컬럼 직접 UPDATE 권한은 부여하지 않음(users 컬럼 권한 정책과 일관).
- 승인 취소/반려로 전환 자격이 사라지면 `admin_set_business_status`에서 `active_mode='personal'`로 강제 복귀.
- **공개 노출 정책(확정)**: `public_profiles` 뷰에 `is_business boolean` = **approved 여부(모드 무관 — 신뢰 정보라 personal 모드에서도 배지 유지)**, `business_name`은 `active_mode='business'`일 때만 노출. 뷰는 SELECT만 GRANT(definer 뷰 쓰기 GRANT 금지 원칙).
- 업체 모드가 게시글·채팅·시설 연동에서 구체적으로 무엇을 바꾸는지(업체 명의 게시 등)는 **후속 문서에서 확정** — 본 문서는 전환 인프라까지만.

### 2.4 RLS·권한

- `business_profiles` RLS: 본인 행 SELECT만(`user_id = app.uid()`). INSERT/UPDATE는 정책 없이 **definer RPC로만**(§5·§6) — `contact_email`·등록번호를 타인이 못 보게 하고, `status`/`nts_*`/`match_*` 필드 위조를 원천 차단.
- 신청/재신청 RPC와 admin RPC 모두 **public 스키마**에 생성(`.rpc()` 노출 규칙), 시그니처 변경 시 구버전 **DROP 필수**(오버로드 섀도잉).

### 2.5 `business_match_rules` — 점수·임계값·스위치의 테이블 분리 (하드코딩 금지)

가중치를 RPC 상수로 박으면 40→50점 조정마다 마이그레이션이 필요하다. 규칙 테이블로 분리해 **운영 중 무마이그레이션 튜닝**:

```sql
create table public.business_match_rules (
  rule_key   varchar primary key,   -- 아래 시드 참조
  weight     integer not null,      -- 배점(임계값·스위치 행은 값으로 사용)
  enabled    boolean not null default true,
  params     jsonb,                 -- 유사도 임계 등 (예: {"sim": 0.85})
  note       text,
  updated_at timestamptz not null default now()
);

-- 시드(초깃값 — §4.4 점수표와 동일)
insert into public.business_match_rules (rule_key, weight, enabled, params) values
  ('phone_exact',          35, true,  null),                 -- 전화 정규화 완전일치
  ('name_high',            30, true,  '{"sim":0.85}'),       -- 상호 유사도 상위
  ('name_mid',             20, true,  '{"sim":0.60}'),       -- 상호 유사도 중위
  ('addr_region',          10, true,  null),                 -- 행정코드(시군구) 일치
  ('addr_sim',             10, false, '{"sim":0.70}'),       -- 주소 유사도 — road_address 적재 전까지 OFF(§1.5-3)
  ('category_match',       15, true,  null),                 -- 업종 일치
  ('threshold_auto',       80, true,  null),                 -- 자동승인 임계값
  ('threshold_review',     50, true,  null),                 -- 관리자검토/신규개업 경계
  ('auto_approve_enabled',  0, false, null);                 -- 자동승인 운영 스위치(초기 OFF, §1.6)
```

- 쓰기는 service_role만(GRANT 없음, RLS enable) — 관리자 조정은 admin RPC `admin_set_match_rule(p_key, p_weight, p_enabled, p_params)`(is_admin 게이트 + `admin_logs` 기록)로.
- `apply_business_profile` RPC가 매 신청 시 이 테이블을 읽어 점수를 계산하고, **적용된 가중치 스냅샷을 `match_detail`에 함께 기록** — 나중에 규칙이 바뀌어도 당시 판정 근거가 감사 가능.
- ⚠️ 필수 조건(§4.4의 전화·업종 AND)은 가중치가 아니라 **로직**이므로 이 테이블로 끌 수 없다 — 끄고 싶으면 마이그레이션으로 의도를 명시(안전장치의 우발적 해제 방지).

## 3. 국세청 검증 · 등록증 업로드

### 3.1 국세청 API

공공데이터포털 「국세청_사업자등록정보 진위확인 및 상태조회 서비스」:

- **상태조회** `POST https://api.odcloud.kr/api/nts-businessman/v1/status?serviceKey=<KEY>` — body `{"b_no":["1234567890"]}`. 응답 `data[0]`: `b_stt`(계속/휴업/폐업), `b_stt_cd`(`01`/`02`/`03`), `tax_type`. **미등록 번호는 `tax_type`에 "등록되지 않은 사업자등록번호" 문구**로 옴 → 미등록도 거절 처리.
- 진위확인(`/validate`, 대표자명·개업일자 대조)은 입력 마찰이 커서 **MVP 제외**. 단 §1.6 위험 완화를 강화하고 싶으면 **자동승인 트랙에 한해** /validate(등록증 기재 대표자명·개업일자 일치)를 추가 요구하는 확장이 가능 — 도용된 남의 사업자번호로 등록증을 위조하는 경로를 어렵게 한다.
- **허용 정책**: `01`(계속사업자)만 신청 진행 허용. `02`(휴업)·`03`(폐업)·미등록은 클라이언트에 사유 반환 후 차단(§10 Case 1·2).
- 시크릿: `NTS_API_KEY`(공공데이터포털 serviceKey, 일 100만건 무료) — Edge Function Secrets 추가, `.env.example` 갱신.
- 클라이언트 선검증: 10자리 + 체크섬(가중치 1,3,7,1,3,7,1,3,5) 검사로 오타를 API 호출 전에 걸러 호출 낭비 방지.

### 3.2 Edge Function `check-business-no` (신규)

- `POST { b_no }` → JWT 검증(로그인 사용자만) → 국세청 상태조회 → `{ ok, status_code, status_label, tax_type }` 반환. **조회만** 하고 DB 기록 없음(기록은 §5 `apply-business`가 서버측 재조회 값으로).
- 구현 템플릿: `search-petcafe/index.ts`(외부 API 호출·에러 폴백 패턴), JWT 검증 헬퍼는 `resolve-region/index.ts:24-45` 재사용.
- `config.toml`에 **`verify_jwt = false` 명시**(커스텀 JWT라 게이트웨이 검증이 무의미 — 전 함수 공통 패턴이며, 항목을 빠뜨리면 배포 시 true로 리셋돼 함수가 깨진다). `apply-business`도 동일.

### 3.3 Storage — 비공개 버킷 `business-docs` (신규)

```sql
insert into storage.buckets (id, name, public, file_size_limit, allowed_mime_types)
values ('business-docs','business-docs', false, 10485760,
        array['image/jpeg','image/png','image/webp','application/pdf']);

-- 본인 폴더 업로드/열람 (경로: <uid>/license/<timestamp>.<ext>)
create policy "business docs owner insert" on storage.objects for insert to authenticated
  with check (bucket_id='business-docs' and (storage.foldername(name))[1] = app.uid()::text);
create policy "business docs owner select" on storage.objects for select to authenticated
  using (bucket_id='business-docs' and (storage.foldername(name))[1] = app.uid()::text);
-- 관리자 열람(승인 심사)
create policy "business docs admin select" on storage.objects for select to authenticated
  using (bucket_id='business-docs' and app.is_admin());
```

- public=false → 열람은 **signed URL**(`createSignedUrl`)로만. 앱 `StorageService`에 비공개 버킷용 헬퍼 추가 필요(현행은 `media` 공개 URL 전제).
- `media` 버킷과 달리 크기·MIME 제한을 버킷에 명시(등록증은 이미지/PDF 10MB면 충분).
- **파기 정책(§8 체크리스트 연동)**: 반려 확정 후 6개월(재신청 유예) 및 탈퇴 시 30일 격리 후 파일 삭제 — 기존 보존·파기 배치(`retention_purge_batch` 계열)에 편입. `withdraw_account`가 행만 처리하고 파일을 남기면 개인정보 잔존이므로 **파일 경로를 파기 큐에 넣는 단계 필수**. 승인 건은 이용계약 지속 중 보관(분쟁·감사 대비), 보존기간은 pmlegal 처리방침에 명시.

## 4. 무관 사업자 사칭 방지 — `facilities` 대조 · 점수제 자동승인 (핵심)

"앱과 무관한 업종의 사업자가 업체 계정을 만드는" 것을 막기 위해, 신청 정보를 인허가 공공데이터(`facilities`, 카페 제외 4종: 분양 `pet_sales`·위탁 `pet_hotel`·동물병원 `animal_hospital`·미용 `grooming`)와 대조해 점수를 매긴다.

### 4.1 트랙 결정 흐름

```
국세청 01 통과
   │
   ├─ 신청 업종 ∈ {분양, 위탁, 병원, 미용} ──→ facilities 후보 검색(§4.3, 물리 업소 단위)
   │      ├─ 후보 있음 → 점수 산출(§4.4)
   │      │      ├─ 필수 조건(전화·업종·업소 유일·영업 중) AND score ≥ threshold_auto
   │      │      │        → [auto]  자동승인 (auto_approve_enabled ON일 때, OFF면 pending 유지)
   │      │      ├─ threshold_review ≤ score < threshold_auto (또는 필수 조건 미충족)
   │      │      │        → [review] 관리자 검토 (매칭 결과 첨부)
   │      │      └─ score < threshold_review
   │      │               → [new_business] 신규개업 트랙(§4.5)
   │      └─ 후보 없음 → [new_business] 신규개업 트랙(§4.5)
   │
   └─ 신청 업종 = 기타(카페 포함) ──→ [new_business] 관리자 검토(대조 데이터 없음)
```

### 4.2 신청 입력 항목 (업체등록 화면)

| 항목 | 비고 |
| --- | --- |
| 업종 | 라디오 5개: 분양 / 위탁(호텔) / 동물병원 / 미용 / 기타. **고지 문구**: "사업자등록증의 업태·종목을 확인하고 실제 등록된 업종을 선택해 주세요" (등록증의 업태·종목 분류와 인허가 업종 분류가 달라 혼동 방지) |
| 상호명 | 등록증 기재 상호 기준(예: ㈜행복컴퍼니) |
| 사업장명(간판명) | 상호와 다르면 입력(예: 행복동물병원). 인허가 데이터의 "사업장명"과 대조되는 핵심 필드(§1.5-7) |
| 상호 변경 여부 | 토글. **공공데이터가 최신이 아닐 수 있음을 고지**하고, 변경했다면 **이전 상호명** 입력 → 대조는 현 상호·사업장명·이전 상호 중 최고 유사도 사용 |
| 사업장 주소 | **주소검색(행안부 juso API) 위젯**으로 선택 — 도로명·지번·행정코드를 표준화된 형태로 동시 확보(자유 텍스트 입력 금지). 지번은 facilities.address(지번, §1.5-3)와, 행정코드는 region_code와 대조 |
| 사업장 전화 | 인허가 데이터에 등록된 업장 전화. **유선(02/031 등)이 기본 가정** — facilities.phone 자체가 대부분 유선이며, 대조는 계정 OTP 번호(010)가 아니라 이 기입값 vs facilities.phone의 숫자 정규화 완전일치. 자동승인의 필수 신호(§4.4) |
| 연락 이메일 | 인증 없음, 형식만 |
| 사업자등록증 | business-docs 업로드 |

### 4.3 정규화 · 후보 검색 (물리 업소 단위)

**정규화 알고리즘(명세)** — 상호·사업장명·주소 공통, DB 함수 `app.norm_biz_text(t)`로 구현:

1. 법인 표기 제거: `주식회사`, `유한회사`, `(주)`, `㈜`, `(유)` 등 사전 목록 치환
2. 괄호쌍과 그 내용 제거: `\([^)]*\)` (예: `(강남점)` — 지점 구분이 사라지는 부작용은 감수, 지점 구분은 주소가 담당)
3. 영문 소문자화
4. **한글·영문·숫자 외 전부 제거**(공백·특수문자·중점 등): `[^0-9a-z가-힣]` → `''`
5. 전화는 별도: 숫자만 남김(`\D` → `''`)

예시(리뷰 지적 케이스):

| 입력 | 정규화 결과 | 판정 |
| --- | --- | --- |
| `행복동물병원` / `행복 동물병원` | 둘 다 `행복동물병원` | **완전일치** → name_high |
| `행복24시동물병원` | `행복24시동물병원` | trgm 유사도 ≈ 0.6대 → name_mid(다른 업체일 수 있으므로 고점 아님이 의도) |
| `㈜행복컴퍼니` (등록증 상호) | `행복컴퍼니` | 사업장명 입력란(`행복동물병원`)이 대조를 담당 |

**후보 검색**:

- 유사도는 `pg_trgm`(이미 설치됨 — **extensions 스키마 v1.6**, 게시글 검색용). ⚠️ 이 프로젝트 RPC 패턴이 `set search_path = ''`이므로 **`extensions.similarity()` / `operator(extensions.%)` 로 스키마 한정** 필수 — 안 하면 함수를 못 찾는다. `facilities` 정규화 상호 표현식에 GIN trgm 인덱스 추가.
- 후보 쿼리: `(정규화 상호 유사도 ≥ name_mid.params.sim — 현 상호·사업장명·이전 상호 중 max) OR (전화 정규화 완전일치)`. **`is_open` 필터 없음** — 폐업 표시가 데이터 지연일 수 있으므로 후보에 포함하되 자동승인만 막는다(§4.4, §10 Case 7).
- **물리 업소 단위 dedup**: 후보 행들을 `biz_key = norm(name) || '|' || norm(address)`로 그룹핑. 업종 집합 = 그룹 내 행들의 category union(= `facility_all_categories` 관점). 점수는 그룹 단위로 1회 산출 — 겸업 업소가 카테고리 행 수만큼 동점 후보로 잡혀 "영구 동점 복수 → 자동승인 불가"가 되는 문제를 없앤다(§1.5-5).
- 그룹이 여러 개면 점수 최고 그룹 선택. **서로 다른 biz_key 그룹이 동점**이면 자동승인 불가(관리자 검토). 선택 그룹의 대표 행(신청 업종과 같은 카테고리 행 우선)을 `matched_facility_id`에, `biz_key`를 `matched_biz_key`에 기록.

### 4.4 점수표 · 자동승인 필수 조건

배점·임계값은 `business_match_rules`(§2.5)에서 읽는다. 아래는 시드 초깃값:

| rule_key | 배점 | 기준 |
| --- | --- | --- |
| `phone_exact` | 35 | 정규화 완전일치. `facilities.phone`이 NULL이면 0점 |
| `name_high` / `name_mid` | 30 / 20 | trgm 유사도 ≥ 0.85 / ≥ 0.60 (현 상호·사업장명·이전 상호 중 max) |
| `addr_region` | 10 | 행정코드(시군구 단위 prefix) 일치 — juso 검색 결과 vs facilities.region_code |
| `addr_sim` | 10 | 정규화 지번 주소 유사도 ≥ 0.70 (**road_address 적재 전까지 enabled=false**, §1.5-3) |
| `category_match` | 15 | 신청 업종 ∈ 매칭 업소(biz_key 그룹)의 카테고리 집합 |

**자동승인(track=auto) 조건 — 합계 점수와 별개의 명시적 AND**:

```
phone_exact 충족  AND  category_match 충족  AND  biz_key 그룹 유일(동점 복수 아님)
AND 그룹 내 is_open=true 행 존재  AND  score ≥ threshold_auto(80)
AND auto_approve_enabled = ON
```

> ⚠️ 필수 신호를 합계로만 다루면 우회 조합이 생긴다 — 예: 전화 35 + 상호 30 + 주소 20 = **85점으로 업종 불일치인데 임계값 80을 넘는다**. "무관 업종 차단"이 이 설계의 목적이므로 **전화·업종은 점수 기여와 무관하게 AND 조건**이다. (이 AND는 `business_match_rules`로 끌 수 없음 — §2.5.)

- **관리자 검토(track=review)**: `threshold_review(50) ≤ score < 80`, 또는 80 이상이어도 필수 조건 미충족(전화 미일치·업종 불일치·동점 복수·그룹 전체 폐업 표시).
- **신규개업(track=new_business)**: `score < 50` 또는 후보 없음(§4.5).
- 임계값·배점 조정은 `admin_set_match_rule`로 무마이그레이션 반영(§2.5). 초기 운영은 `auto_approve_enabled=OFF`로 전건 검토하며 auto 판정 정확도를 검증한 뒤 ON(§1.6).

### 4.5 신규개업 트랙 (facilities 미조회)

LOCALDATA 갱신 지연으로 신규 업체가 조회 안 되는 것은 정상이므로 거절이 아니라 **강화된 수동 검토**로:

- 자동승인 불가, 관리자 검토 필수.
- **추가 서류 1종 업로드 요구**(`extra_doc_path`): 동물판매업·동물위탁관리업·동물미용업은 동물보호법상 **영업 등록증**, 동물병원은 **개설신고확인증**. 인허가 데이터에 없더라도 이 서류가 "동물 관련 실사업"임을 직접 증명한다 — 무관 사업자 차단의 최후 방어선.
- 기타 업종(카페 등)도 이 트랙: 어떤 서류를 요구할지 화면에서 안내(카페는 영업신고증 등), 관리자 재량 승인.
- **추가 서류 검증은 `business_profiles` INSERT 전에 수행**(§5) — pending 행이 먼저 생기면 사용자가 스스로 보완할 경로가 없다.

### 4.6 자동승인 사후 안전장치 (§1.6 위험 완화)

1. 자동승인 시에도 `admin_logs`에 `action_type='business_auto_approved'` + `match_detail`(적용 가중치 스냅샷 포함) 기록 — 관리자 콘솔에서 `admin_list_business_applications(p_status:'approved', auto_only)`로 사후 감사.
2. 사업자등록증은 자동승인 건도 보관(표본 감사·분쟁 대비, 보존기간 §3.3). 진짜 업주의 사칭 신고 접수 시 관리자가 등록증 대조 → `admin_set_business_status('rejected')`로 즉시 강등(active_mode 복귀 포함).
3. `matched_biz_key` 부분 유니크(§2.2) — 실존 업소 1곳 = 활성 계정 1개(카테고리 행 우회 불가). 사칭자가 선점했더라도 진짜 업주 신고로 반려되면 자리가 풀린다.
4. (선택 확장) 자동승인 트랙 한정 국세청 `/validate` 추가(§3.1).
5. (선택 확장, 최강) **ARS 업장전화 점유 확인**: 서버가 `facilities.phone`(유선 02/031 등 — SMS 불가)으로 전화를 걸어 인증코드를 음성 안내, 신청자가 앱에 입력(`phone_verifications` 검증 로직 재사용). 그 업장 전화기를 받을 수 있는 사람만 통과하므로 "공개정보 베끼기" 스푸핑을 원천 차단한다(발신 대상이 신청자 기입값이 아니라 **인허가 데이터의 번호**라는 점이 핵심). **Solapi 지원 확인(2026-07)**: TTS 음성 발송 + 키패드 응답(DTMF)까지 지원, 기본 200원 + 초당 4원(코드 안내 ~10초 ≈ 240원/건), 발신번호는 기존 `SOLAPI_SENDER` 재사용. 부재중 시 1분 간격 최대 3회 자동 재발송이 기본이므로 신청 화면에 "업장으로 전화가 갑니다" 안내 필요. 비용·마찰이 있어 MVP 제외하되, **자동승인 스위치 ON의 전제조건 후보 1순위**.

## 5. 신청 플로우 (클라이언트 → 서버)

```
[업체등록 화면]
  사업자번호 ─ 체크섬 ─→ check-business-no (국세청 01 확인)
  업종·상호명·사업장명·(이전 상호명)·주소(juso 검색)·업장 전화·연락 이메일 입력
  등록증 업로드 → business-docs/<uid>/license/…   (신규개업 트랙 예상 시 추가 서류도)
  제출 → apply-business (Edge Function)
            ├─ 국세청 재조회(서버가 직접 — 클라이언트 값 신뢰 안 함)
            ├─ RPC apply_business_profile(...):
            │    ├─ business_match_rules 로드 → 후보 검색·dedup·점수 산출(§4.3~4.4) — 전부 서버 계산
            │    ├─ 트랙 판정. new_business인데 extra_doc 없음 → INSERT 전에
            │    │    'extra_doc_required' 반환(행 생성 안 함 — 사용자가 서류 추가 후 재제출)
            │    ├─ business_profiles upsert (track·score·match_detail·biz_key 기록)
            │    │    └─ 부분 유니크 위반 → 409 'business_no_taken' / 'facility_taken'
            │    ├─ track=auto & 스위치 ON → status='approved' + 알림 + admin_logs
            │    └─ 그 외 → status='pending'
            └─ 응답: { track, status } → 화면에서 "승인 완료 / 심사중 / 추가서류 필요" 분기
```

- 신청·재신청을 Edge Function `apply-business` 하나로: JWT 검증 → 국세청 확인 → service_role로 definer RPC 호출. RPC는 service_role 전용 GRANT(`signup_user`와 동일 패턴).
- 재신청은 `status='rejected'`인 본인 행만 pending으로 갱신 허용. **`pending` 중 수정·재제출 차단은 의도된 정책**(심사 대상 고정) — 오타·서류 보완은 관리자 반려(사유 명시) 후 재신청으로만. 이 때문에 §4.5의 "추가 서류 검증은 INSERT 전" 규칙이 중요하다.

## 6. 관리자 승인 · Override (기존 admin RPC 패턴)

- **`admin_list_business_applications(p_status default 'pending', p_track default null, ...)`** — is_admin 게이트. 신청 목록 + `match_score`/`match_detail`/`matched_facility_id`(콘솔에서 facilities 행과 나란히 표시) + 서류 경로(signed URL 발급 열람).
- **`admin_set_business_status(p_user uuid, p_status varchar, p_reason text default null)`** — is_admin 게이트 definer RPC:
  1. `approved`/`rejected`만 허용, `reviewed_by/at` 기록, 반려 시 `p_reason` 필수(`rejected_reason` 저장)
  2. **Override 승인**: 자동승인 조건 미달(낮은 점수·전화 미일치·신규개업 트랙 포함)이라도 관리자가 "이 업체 맞음" 판단 시 **강제 승인 가능 — 단 `p_reason` 필수**(`review_note`에 저장). 트랙·점수는 판정 기록으로 남고 승인 권한은 항상 관리자가 최종.
  3. 반려·승인취소 시 `users.active_mode='personal'` 복귀(§2.3)
  4. `admin_logs`에 before/after + override 여부·사유 기록(`admin_set_report_status` 패턴 — `action_type='business_override_approved'` 별도 구분)
  5. 알림 발송: `notifications` INSERT — **신규 `notification_type` 2종 `business_approved` / `business_rejected`** (자동승인·override 승인 모두 `business_approved` 재사용)
- **`admin_set_match_rule(p_key, p_weight, p_enabled, p_params)`** — §2.5 규칙 튜닝용, is_admin 게이트 + `admin_logs` 기록.
- ⚠️ **`notifications_notification_type_check` CHECK 제약을 같은 마이그레이션에서 DROP/ADD로 확장**할 것 — 빼먹으면 알림 트리거가 조용히 실패한다(기존 13종 + 2종). `admin_logs.action_type`은 CHECK 없음 — 확장 불필요(확인됨).
- 승인 상태 변경은 결과 행이 관리자에게 SELECT 비가시일 수 있으므로 반드시 definer RPC 경유(42501 예방) — 위 설계로 충족.

3. **앱**: 가입 플로우 재배열 → 업체등록 화면(juso 위젯 포함) → 내정보 수정 버튼 2종 → 라벨/배지 정리
4. **운영 준비**: facilities 재적재(road_address·is_open 동기화) 및 월 1회 룰 시작, 점수 규칙 튜닝(실데이터 표본으로 auto 판정 검증), 자동승인 스위치 ON 판단
5. **정리**: 기존 business 계정 백필, pmlegal 반영, 파기 배치 편입, 문서(supabase-db/api.md) 갱신
